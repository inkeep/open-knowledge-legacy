---
title: Agent Write Path
description: How external agents write into the CRDT via markdown, and how the server-authoritative bridge keeps WYSIWYG and source mode views in sync.
---
External agents (Claude Code, Cowork, Cursor, etc.) write into the Y.Doc via HTTP APIs. The system provides two write endpoints; the server-authoritative cross-CRDT bridge then propagates the write to both `Y.XmlFragment` (TipTap) and `Y.Text` (CodeMirror) so the result is visible in either editor mode without any client-side merge.

## Write endpoints

### Raw write (`POST /api/agent-write`)

Appends a paragraph to the Y.Doc using raw Yjs operations:

```
Y.XmlFragment('default')
  └─ Y.XmlElement('paragraph')
      └─ Y.XmlText() with applyDelta([{ insert: "text" }])
```

This is the original V3 validation endpoint. It works but bypasses the markdown pipeline, so the content must be structured as y-prosemirror-compatible nodes.

### Markdown write (`POST /api/agent-write-md`)

Accepts a JSON body:

```json
{
  "markdown": "## New section\n\nAgent-written paragraph.",
  "position": "append"
}
```

The endpoint routes through `applyAgentMarkdownWrite` per the **Y.Text-is-truth contract** (AGENTS.md precedent #38). The agent's payload bytes are the source-of-truth — they land in `Y.Text` verbatim (modulo the equivalence classes documented in `normalizeBridge`); the structural view derives from those bytes via `parse`.

1. Opens a `DirectConnection` to the Y.Doc
2. Reads the current `Y.Text('source')` — `currentYText = ytext.toString()` — to compose against. This is the user-typed source bytes (CRDT-synced from any concurrent CM6 source-mode editor)
3. Composes the new full body bytes by `position`:
   - `'replace'` — `payload`
   - `'append'` — `currentYText + '\n\n' + payload`
   - `'prepend'` — `payload + '\n\n' + currentYText`
   No `.trim()`, no normalization. Agent payload is preserved verbatim — leading newlines, trailing whitespace, intentional blank lines all survive (modulo NG1 architectural floor: 3+ blank lines collapse to 2 on the next save)
4. Routes through the shared `composeAndWriteRawBody(document, newBody, embedResolver)` primitive in `packages/server/src/bridge-intake.ts`, inside one `session.dc.document.transact(..., session.origin)` block under `AGENT_WRITE_ORIGIN` (a paired-write origin per AGENTS.md precedent #1):
   - **Y.Text side:** `applyFastDiff(ytext, currentYText, newBody)` — character-level DMP write; preserves CRDT Items for unchanged byte ranges
   - **Fragment side:** `parse(newBody) → updateYFragment(document, fragment, pmNode)` — structural diff preserves user-content Items at matching positions

Both halves of the cross-CRDT write are atomic from any other observer's perspective — the paired-write origin marker tells Observer A and Observer B to short-circuit and refresh their baselines without dispatching the bridge sync (the writer already made both CRDTs consistent).

**What the contract preserves on round-trip:** the agent's `__foo__` (vs `**foo**`), `1)` ordered lists (vs `1.`), single backtick on `` `code` `` with no internal backticks, indented vs fenced code blocks, ATX trailing hashes (`## H ##`), setext underline length, blockquote marker spacing (`>foo` vs `> foo`), single-quoted vs double-quoted titles, doc-start `***` vs `---` (modulo `normalizeBridge` tolerance), CRLF line endings (modulo tolerance), UTF-8 BOM (modulo tolerance), and any source-form attr captured by FR-1..FR-25.

**What it does NOT preserve** (architectural floor): blank-line counts beyond 2, raw HTML entity decoding (`&amp;` → `&`), MDX edge cases not represented in the PM tree, and within-conflict-region char-level DMP rewrites in collaborative concurrent editing. See `specs/2026-05-01-bridge-ytext-is-truth-contract/SPEC.md` §4d for the full enumeration with future-work pathways.

## Source mode live injection

Source mode (CodeMirror) binds `Y.Text('source')` directly via `y-codemirror.next`. Agent writes are visible in real time without any application-level observer:

1. `applyAgentMarkdownWrite` writes the agent's composed body bytes into `Y.Text` (and derives the fragment) inside one server-side `doc.transact(..., AGENT_WRITE_ORIGIN)` block.
2. The Y.Text mutation propagates to every connected client through the standard CRDT sync.
3. `y-codemirror.next` applies the incoming delta to the CodeMirror buffer with character-level precision — cursor and selection on the local keystroke side are preserved by the binding.

The server's cross-CRDT sync runs exclusively on the server under the `afterAllTransactions` settlement dispatch (precedent #13(b)). One outermost `doc.transact()` call produces exactly one settlement fire. **Observer A (XmlFragment → Y.Text) runs before Observer B (Y.Text → XmlFragment)** within each drain so Observer B Phase 1 reads the post-A converged Y.Text. Under the contract, **Observer B is watchdog-only**: Phase 1 derives the fragment from `parse(ytext) → updateYFragment`; Phase 2 asserts the bridge invariant

```
normalizeBridge(ytext.toString())
  === normalizeBridge(prependFrontmatter(fm, mdManager.serialize(fragment)))
```

and emits / throws on violation — never writes back to Y.Text. The pre-contract canonicalize-write-back step at the Phase 2 site was deleted by spec FR-31; re-adding it is a STOP_IF (see `packages/server/src/server-observers.ts` and the AGENTS.md STOP rules). There is no debounce window, no injected `Scheduler`, and no wall-clock timing in either bridge observer file (enforced by the grep gate at `packages/server/src/bridge-no-wallclock.test.ts`).

Paired-write origins (`AGENT_WRITE_ORIGIN`, `FILE_WATCHER_ORIGIN`, `ROLLBACK_ORIGIN`, `MANAGED_RENAME_ORIGIN`) declare `context.paired: true` at their definition site. Observer A AND Observer B both detect that marker structurally via `isPairedWriteOrigin(transaction.origin)` and refresh their shared baseline synchronously inside the observer callback — the settlement handler then has no work to dispatch for the drain (the paired writer already made both CRDTs consistent through `composeAndWriteRawBody`).

Client-side keystroke protection is not needed on the bridge layer because the client observer does not write the derived CRDT under precedent #14.

## Find-and-replace (`POST /api/agent-patch`)

`edit_document` (the MCP tool) hits `POST /api/agent-patch`. The handler builds the find-target from `ytext.toString()` — the user-typed source bytes — not from `mdManager.serialize(fragment)`. So the agent's `find` matches against what it would see if it `read_document`'d the file in source mode. Agents that compute offsets against canonical bytes (e.g., by re-serializing fragment) will see `agentPatchFindMismatches` rise; the contract is "match what the user typed."

Frontmatter-intersecting patches return HTTP 400 with `error: "Frontmatter edits are not supported via edit_document; edit frontmatter directly via the property panel binding"` — agents must use `write_document` (full-doc replace, position `'replace'`) for FM rewrites.

## Undo (`POST /api/agent-undo`)

`Y.UndoManager.undo()` mutates `Y.Text` directly — the `trackedOrigins` machinery operates on Y.Text. The handler then runs `parse(ytext) → updateYFragment` for the fragment side inside one `doc.transact(..., session.undoOrigin)` block. Post-undo ytext IS the restored user bytes — there is no canonicalize-write-back step. The pre-contract re-serialize step (which would have re-canonicalized the user's restored bytes) was deleted by spec FR-40.

## Frontmatter write (CRDT-direct via `bindFrontmatterDoc`)

Frontmatter lives in the YAML region of `Y.Text('source')` — the same CRDT text the body editor and source-mode editor bind to. There is no separate `Y.Map('metadata')` for FM (a per-key map was the predecessor's design and was removed in the realtime-frontmatter-entries spec). Edits to *different* keys land at non-overlapping byte ranges and merge cleanly through Y.Text's character-level CRDT; same-key concurrent value edits collapse to last-writer-wins on the overlapping bytes.

The browser property panel writes through `bindFrontmatterDoc(provider)` (in `@inkeep/open-knowledge-core/bridge`) — no HTTP round-trip. The binding exposes `patch(patch)` (JSON Merge Patch — `key: value` sets/creates, `key: null` deletes), `rename(oldKey, newKey)` (preserves source position so `title` → `titles` doesn't reorder), and `reorder(orderedKeys)` (drag-to-reorder commit). Each write parses the YAML region into a yaml@2 `Document`, edits at the `Pair` level, re-stringifies, and replaces the Y.Text byte range in one `doc.transact(fn, FORM_WRITE_ORIGIN)` block. `FORM_WRITE_ORIGIN` is non-paired (single-root writer — touches only `Y.Text`); Observer B fires normally and exits via the already-in-sync gate when the body half is unchanged.

L1 validation runs at the binding boundary against `FrontmatterPatchSchema` and `FrontmatterValueSchema` (Zod) before any Y.Text mutation; on failure the binding returns `Result.err({ code: 'SCHEMA_INVALID', issues })` and leaves the doc untouched. The binding additionally enforces a 64 KB cap on the FM region and rejects the reserved key `'frontmatter'`.

There is no L3 server-side validation hook. The Y.Text region IS the source of truth — including malformed bytes — so defense moves to (a) the binding's L1 commit gate against UI-driven malformed input and (b) the panel's last-valid render against transient invalid bytes from source-mode typing or disk drift. Disk persistence writes `ytext.toString()` verbatim; malformed YAML round-trips through disk unchanged.

There is no MCP surface for frontmatter editing today. `edit_document` (`POST /api/agent-patch`) rejects FM-intersecting patches with HTTP 400; the redirect message tells callers to use `write_document` with `position: "replace"` and a payload that includes the new YAML block.

**Telemetry.** Counter `ok.frontmatter.edit_surface_total` labels writes by source (`mcp-write`, `file-watcher`, `source-mode`) so adoption is queryable. Form writes do not emit a server-side counter — the originating client is responsible for any client-side span.

## Mode toggle (no merge)

The WYSIWYG/source mode toggle is a UI visibility flip, not a serialize-and-merge operation. `EditorActivityPool` mounts both `TiptapEditor` and `SourceEditor` concurrently per active document and uses `display:none` to swap between them. Each editor stays bound to its Y type (`Y.XmlFragment('default')` for TipTap via `Collaboration`; `Y.Text('source')` for CodeMirror via `y-codemirror.next`) for the lifetime of the mount.

The server's bidirectional bridge keeps the two Y types continuously in sync, so toggling back from source mode reveals a TipTap view that already reflects every Y.Text edit the user made — no client-side three-way merge runs on toggle. Concurrent agent writes that arrived during a source-mode session are visible to TipTap on toggle-back for the same reason.

Three-way merge for content preservation has moved to the bridge layer itself: when an inbound CRDT update arrives during local Y.XmlFragment editing, Observer A's Path B uses the hybrid diff3+DMP `mergeThreeWay` algorithm (`@inkeep/open-knowledge-core/bridge`) with a content-preservation post-condition (`assertContentPreservation`). On post-condition violation, the production policy is to (a) emit a structured `bridge-merge-content-loss` log, (b) write a silent named version-history checkpoint via `saveInMemoryCheckpoint` (rendered in `TimelinePanel` so users can recover via the existing Restore UI), and (c) apply the merge as-computed so the editor keeps responding. Dev/test re-throws so integration suites fail loudly; conversion-class regressions are caught at PR tier by the deterministic fidelity PBT (`packages/app/tests/fidelity/bridge-observer-conversion.test.ts`). The bridge-convergence fuzz is preserved for ad-hoc sampling via `bun run measure:fuzz` (see `specs/2026-04-19-ci-signal-quality/` for the CI signal quality rationale). See `specs/2026-04-16-bridge-correctness/SPEC.md` §6 R1/R7 for the full contract and `packages/server/src/server-observers.ts` for the wiring.

## Attribution journal (summaries)

Each agent-write API call records a contributor entry in the in-memory accumulator `pendingContributors: Map<agentId, ContributorEntry>` (`packages/server/src/contributor-tracker.ts`). The optional fifth argument to `recordContributor(docName, agentId, displayName, colorSeed, summary?)` appends a summary to the contributor's flat `summaries: string[]` array, preserving insertion order across the L2 debounce window.

On the 15-30s L2 debounce, `swapContributors()` drains the accumulator and `formatContributorsFrom(snapshot)` emits one `ok-contributors:` JSON line per contributor into the shadow-repo commit body:

```
ok-contributors: {"v":1,"id":"agent-abc","name":"Claude","colorSeed":"...","docs":["foo.md","bar.md"],"summaries":["Fixed token-refresh race","Added example block"]}
```

The `summaries` field is flat (per-contributor, not per-doc) and omitted entirely when no summaries were recorded — legacy commits and summary-less writes are byte-identical to pre-feature behavior. The JSON line stays at `v: 1`; the field is purely additive (precedent #9).

**Truncation single-source-of-truth.** `normalizeSummary` in `packages/server/src/agent-write-summary.ts` is the only place summary text is validated and truncated. The mutating API handlers (`/api/agent-write`, `/api/agent-write-md`, `/api/agent-patch`, `/api/rename-path`, `/api/rollback`) each call it at the request boundary before threading the normalized value into `recordContributor`. The cap is 80 characters (79 visible + U+2026 `…`); the Zod schemas on MCP tool inputs cap at 200 as a separate transport-safety bound.

**Identity boundary for agent writes vs rename/rollback.** Agent-write handlers (`handleAgentWrite`, `handleAgentWriteMd`, `handleAgentPatch`, `handleAgentUndo`) call `extractAgentIdentity(body)` at entry per precedent #24 — body `agentId` resolves to an `agent-<id>` writer. Rename and rollback handlers (`handleRenamePath`, `handleRollback`) instead route through `extractActorIdentity(body, getPrincipal)` in `packages/server/src/extract-actor-identity.ts`, which adds a fallback for human-driven UI calls:

1. Body has `agentId` → `agent-<id>` contributor (same as `extractAgentIdentity`); `actor.principalId` is also populated from `getPrincipal()` when a principal is loaded so the agent-on-behalf-of-principal audit trail mirrors `buildAgentActor` for non-rename writes.
2. No `agentId`, server has a loaded principal → `principal-<uuid>` contributor with the principal's display name. UI drag-rename in FileTree and the in-editor Restore button hit this branch — both post with no identity, both attribute to the loaded principal.
3. No `agentId`, no principal → no contributor entry (anonymous). When principal load fails at server start, rename and rollback rows stay anonymous rather than inventing identity.

**Trust boundary.** `extractActorIdentity` does NOT read `body.principalId` — the HTTP body is unauthenticated and a body-supplied principal could be spoofed. The server's `getPrincipal()` (loaded from `<contentDir>/.ok/principal.json` at boot) is the only source of principal identity for HTTP handlers.

**Side-effect carve-out.** When a rename rewrites inbound wiki-links and markdown links across N linking docs, those linking docs do NOT receive an actor contributor entry — they flow through the existing `defaultWriter` (anonymous) path. The renamed doc itself records the actor via `recordContributor`; the cascading rewrites stay mechanical. Applies symmetrically to agent-driven and principal-driven renames.

**Default summaries.** When no summary is provided, MCP-driven `rename_document`, `rename_folder`, and `rollback_to_version` substitute a server-generated default (`"Renamed <from> → <to>"` or `"Restored to <sha-short>"`) through the same `normalizeSummary` path. The folder rename's default applies to every affected-doc contributor entry — users who want per-doc nuance should supply their own summary. `write_document` and `edit_document` have no default — an absent summary means an absent bullet.

**Parser tolerance.** `parseContributors` in `packages/core/src/shadow-repo-layout.ts` accepts both legacy (no `summaries` field) and new commits. A malformed `summaries` value drops just that field while preserving the contributor entry — a deliberate divergence from the whole-entry-skip policy applied to other optional fields, because decorative loss beats attribution loss.

**Consumers.**

- `TimelinePanel` renders summaries as a collapsible bulleted list under the author line (first bullet inline, remainder behind a "Show N more" expander). The full doc list stays visible as ground truth.
- `exec` / `read_document` enrichment output carries the field through automatically via `history.contributors[*].summaries`, so agents reading prior timeline state see intent alongside identity.
- Metrics counters in `packages/server/src/metrics.ts`: `agentWriteCalls` (denominator), `summariesProvided` (numerator for M1 adoption rate), and `summariesTruncated` (M2 cap-efficacy) — all surfaced via `GET /api/metrics/reconciliation`.

## Agent simulator

The CLI tool supports both write modes:

```bash
bun run src/server/agent-sim.ts                      # Single raw write
bun run src/server/agent-sim.ts --rapid 5            # 5 raw writes, 100ms apart
bun run src/server/agent-sim.ts --markdown           # Single markdown write
bun run src/server/agent-sim.ts --markdown --rapid 5 # 5 markdown writes, 100ms apart
```

The `--markdown` flag uses `POST /api/agent-write-md` (the unified path). Without it, the simulator uses `POST /api/agent-write` (raw Y.XmlElement).
