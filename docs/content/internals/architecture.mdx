## Editor layer

TipTap v3 with ProseMirror provides the WYSIWYG editing surface. Key extensions:

- **Collaboration** (`@tiptap/extension-collaboration`) -- binds Y.Doc from Hocuspocus provider to the editor via y-prosemirror
- **Frontmatter** -- regex strip before parse, re-prepend after serialize (~25 LOC)
- **Image** (`@tiptap/extension-image`) -- built-in markdown support in TipTap v3
- **Task lists** (`TaskList` + `TaskItem`) -- native markdown round-trip in v3
- **JsxComponent** -- block-level MDX JSX node (`content: 'block*'`, `isolating: true`). Descriptor-dispatched at render time via a component registry (18 built-in fumadocs-ui components + wildcard fallback). Supports live React preview, auto-generated PropPanel, and editable children in WYSIWYG.
- **JsxInline** -- inline MDX JSX node (`content: 'text*'`, zero attrs). Renders as visible source text in WYSIWYG -- no live React render, no PropPanel. Characters editable like any prose.

Markdown round-trip fidelity: zero semantic loss after ~80 LOC of fixes. Convergence confirmed (cycle 2 byte-identical to cycle 1).

## CRDT layer

Yjs v13 with y-prosemirror provides conflict-free concurrent editing. The Yjs v14 unified delta protocol was tested (V7) but is not yet viable -- the ecosystem pins to v13.

Key constraint: source toggle uses `updateYFragment()` (diff-based), never `prosemirrorJSONToYDoc()` which would destroy collaboration state.

## Collab server

Hocuspocus embeds in Vite via `configureServer()` plugin hook with a standalone `ws.WebSocketServer({ noServer: true })`. No `listen()` call -- the embedding pattern intercepts WebSocket upgrades on `/collab`.

Two agent write endpoints use `hocuspocus.openDirectConnection()`:

- **`POST /api/agent-write`** -- raw Y.XmlElement write (appends a paragraph with `applyDelta()`)
- **`POST /api/agent-write-md`** -- markdown write (unified path). Accepts `{ markdown, position? }`. Routes through `applyAgentMarkdownWrite` (XmlFragment-authoritative composition per AGENTS.md precedent #10): reads the current `Y.XmlFragment` (reflects all CRDT-synced content including concurrent client WYSIWYG typing), composes the agent's delta at the markdown level per `position` (`'append'` / `'prepend'` / `'replace'`), applies to XmlFragment via `updateYFragment()` (structural diff preserves user-content Items), then mirrors `Y.Text` via `applyByPrefixSuffix` (minimal mutation, preserves non-agent Y.Text Items and their origins). Replaces the deleted `syncTextToFragment` which used `Y.Text` as the authoritative input and destroyed concurrent user XmlFragment content — Bug-A in the `2026-04-14-bridge-convergence-under-concurrent-writes` spec.

## Source toggle

Two-mode toggle between WYSIWYG (TipTap) and source (CodeMirror 6):

- **To source:** `editor.getJSON()` -> `MarkdownManager.serialize()` -> CodeMirror. A snapshot of the markdown is stored for the three-way merge.
- **From source:** Three-way merge (`three-way-merge.ts`) compares the snapshot, the user's edits, and the current Y.Doc. Agent-added paragraphs (those beyond the snapshot's block count) are appended to the user's edits. The merged result applies via `updateYFragment()`.

**Live agent writes in source mode:** When source mode is active, a Y.XmlFragment observer detects changes to the Y.Doc (from agent writes via `/api/agent-write-md`) and serializes the updated content into CodeMirror via React state. The observer is unsubscribed before toggle-back to prevent interference with the merge.

**Typing coordination:** Both editors register DOM listeners (keydown, paste, drop, cut) that signal the observer system to defer background sync while the user is actively typing. This applies symmetrically -- source-mode typing receives the same protection from observer-driven tree replacement as WYSIWYG typing.

**Conflict handling:** If both the user and an agent modified the same paragraph, the user's version wins. Conflicts are logged to the console with the paragraph index.

## Persistence pipeline

Three-tier auto-persistence with no "save" button:

1. **Crash recovery** (CRDT to disk) -- Hocuspocus `onStoreDocument` hook, 2s quiet / 10s max debounce
2. **Auto-commits** -- `simple-git` plumbing: `git add` -> `write-tree` -> `commit-tree` -> `update-ref refs/wip/main`, 30s debounce
3. **Named checkpoints** -- user-initiated (future)

Server-side serialization uses `yXmlFragmentToProsemirrorJSON()` (pure Yjs/JSON, no DOM) then `MarkdownManager.serialize()` to markdown string.

## Component blocks (MDX JSX editing)

MDX JSX elements are first-class editable blocks in the editor. Block-level JSX (`<Callout>`, `<Tabs>`, `<Steps>`, etc.) is parsed from `mdxJsxFlowElement` mdast nodes into `jsxComponent` PM nodes with structured attrs and editable children.

The system uses a **descriptor registry** with 18 built-in fumadocs-ui component descriptors and a wildcard `'*'` fallback. Registration is configuration -- adding a descriptor to the built-ins manifest upgrades every existing instance of that component name at render time with zero schema migration.

**Rendering branches** (descriptor-dispatched at render time):
- **Registered + hasChildren** -- live React component preview + floating PropPanel + editable children via `<NodeViewContent>`
- **Registered + no children** -- live React render + PropPanel; content wrapper has zero visual footprint when empty
- **Wildcard (unregistered)** -- name badge + editable children, no PropPanel
- **Error state** -- `ComponentErrorBoundary` catches render failure; NodeView swaps to a nested CodeMirror editor showing the block's source for in-place fixing

**Serialization (γ pattern):** Pristine nodes (`sourceDirty: false`) serialize via `sourceRaw` for byte-identical round-trip. Edited nodes reconstruct the `mdxJsxFlowElement` mdast from structured attrs + recursively-serialized children. A source-dirty observer plugin tracks user-intent edits (deny-lists sync/agent/rollback origins).

Inline JSX (`<Icon />`, `<Badge>text</Badge>`) renders as visible source text via the thin `jsxInline` PM node -- no live React render, no PropPanel, editable like any prose text.
