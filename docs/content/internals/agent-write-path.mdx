---
title: Agent Write Path
description: How external agents write into the CRDT via markdown, and how the source toggle preserves those writes through three-way merge.
---

External agents (Claude Code, Cowork, Cursor, etc.) write into the Y.Doc via HTTP APIs. The system provides two write endpoints and a three-way merge to ensure agent writes survive source mode editing.

## Write endpoints

### Raw write (`POST /api/agent-write`)

Appends a paragraph to the Y.Doc using raw Yjs operations:

```
Y.XmlFragment('default')
  └─ Y.XmlElement('paragraph')
      └─ Y.XmlText() with applyDelta([{ insert: "text" }])
```

This is the original V3 validation endpoint. It works but bypasses the markdown pipeline, so the content must be structured as y-prosemirror-compatible nodes.

### Markdown write (`POST /api/agent-write-md`)

Accepts a JSON body:

```json
{
  "markdown": "## New section\n\nAgent-written paragraph.",
  "position": "append"
}
```

The endpoint routes through `applyAgentMarkdownWrite` (XmlFragment-authoritative composition per AGENTS.md precedent #10):

1. Opens a `DirectConnection` to the Y.Doc
2. Reads the current `Y.XmlFragment` — reflects all CRDT-synced content including any concurrent client WYSIWYG typing that has propagated via the tree-sync message. Server-side cross-CRDT sync (precedent #14) runs under a single-writer `afterAllTransactions` settlement dispatch (one fire per outermost `doc.transact()` drain, SPEC 2026-04-16 bridge-correctness §6 R4); the client observer is a baseline-tracking shell that does not write Y.Text.
3. Serializes XmlFragment to markdown via `yXmlFragmentToProsemirrorJSON` + `MarkdownManager.serialize`
4. Composes the agent's delta at the markdown level per `position` (`'append'` / `'prepend'` / `'replace'`)
5. Parses the composed markdown and applies to XmlFragment via `updateYFragment()` — structural diff preserves user-content Items at matching positions
6. Mirrors the canonical post-fragment markdown to `Y.Text` via `applyByPrefixSuffix` (shared `@inkeep/open-knowledge-core` utility) — minimal mutation, preserves non-agent Y.Text Items and their origins

The XmlFragment-authoritative shape is required to preserve concurrent user WYSIWYG content during agent writes. The prior implementation (`syncTextToFragment`, deleted by the `2026-04-14-bridge-convergence-under-concurrent-writes` spec — FR-9) read `Y.Text` as the authoritative input, which lagged the XmlFragment for client-originated WYSIWYG typing and caused `updateYFragment` to stomp user content (Bug-A, deterministic).

V0-14's future `applyAgentUndo` handler must follow the same pattern — see `evidence/bug-d-mechanism.md` in the bridge-convergence spec for the template.

## Source mode live injection

When the user is in source mode (CodeMirror), agent writes are still visible in real-time:

1. `App.tsx` subscribes to Y.XmlFragment changes via `onContentChange()` when entering source mode
2. When an agent write modifies the Y.Doc, the observer fires and serializes the updated content to markdown
3. The markdown is pushed to CodeMirror via React state (`setSourceContent`)
4. The observer is unsubscribed before toggle-back to prevent the merge from triggering the observer

The bidirectional observer sync runs exclusively on the server under the `afterAllTransactions` settlement dispatch (precedent #13(b)). The server's Observer B (Y.Text → XmlFragment) is ordered after Observer A within each drain and canonicalizes `Y.Text` via `applyFastDiff` after `updateYFragment` so the bridge invariant `ytext === serialize(fragment)` holds at every settlement point — no debounce window or wall-clock timing is involved. Client-side keystroke protection is not needed on the bridge layer because the client shell does not write the derived CRDT.

Current limitation: the full CodeMirror content is replaced on each change, so the cursor may jump. This is acceptable for P0; cursor preservation is a future refinement.

## Three-way merge on toggle-back

When the user toggles back from source to WYSIWYG, a three-way merge (`three-way-merge.ts`) reconciles the user's edits with any concurrent agent writes.

### Three versions compared

- **Base (snapshot):** The markdown captured when entering source mode
- **Theirs (current Y.Doc):** The Y.Doc's current state, which may include agent writes
- **Ours (user edits):** The markdown the user edited in CodeMirror

### Merge strategy

1. **No user changes:** If the user didn't edit anything, the Y.Doc is left as-is (agent writes fully preserved)
2. **No agent changes:** If the Y.Doc hasn't changed since the snapshot, the user's edits apply directly
3. **Both changed (selective merge):**
   - Agent-added blocks (those beyond the snapshot's paragraph count) are appended to the user's edits
   - Blocks within the snapshot range that both user and agent modified are conflicts -- user wins
   - The merged result is applied via `updateYFragment()`

### Conflict resolution

When both the user and an agent modify the same paragraph:

- The user's version wins
- A warning is logged to the console: `[three-way-merge] Conflict at paragraph N: both user and agent modified. User version wins.`
- Agent additions to *other* paragraphs are still preserved

## Agent simulator

The CLI tool supports both write modes:

```bash
bun run src/server/agent-sim.ts                      # Single raw write
bun run src/server/agent-sim.ts --rapid 5            # 5 raw writes, 100ms apart
bun run src/server/agent-sim.ts --markdown           # Single markdown write
bun run src/server/agent-sim.ts --markdown --rapid 5 # 5 markdown writes, 100ms apart
```

The `--markdown` flag uses `POST /api/agent-write-md` (the unified path). Without it, the simulator uses `POST /api/agent-write` (raw Y.XmlElement).
