# Story: Open in agent desktop (Claude Cowork / Claude Code / Codex / Cursor)

**Last verified:** 2026-04-21
**Upstream research:** [reports/deep-linking-ai-desktop-apps-2026/](../../reports/deep-linking-ai-desktop-apps-2026/REPORT.md) — 5 rounds of primary-source bundle inspection + 2026-04-21 live-testing round against installed Claude.app 1.2581.0 / Codex.app 26.415 / Cursor.app 3.1.15.
**Upstream platform work:** [specs/2026-04-11-electron-desktop-app/SPEC.md](../../specs/2026-04-11-electron-desktop-app/SPEC.md) — Electron M1 landed on `main` in PR ending at commit `c29a5a14` (2026-04-21). This story builds on the M1 shell; it doesn't require M2–M7 signing/DMG/auto-update to ship internally.

---

## Problem (SCR-lite)

**Situation.** OK is positioning itself as the knowledge layer for agent-driven work — wiki pages on disk, a CRDT editor for collaboration, an MCP surface that agents already call to read and write. Users today land on an OK page (a spec, a research report, a project doc) and routinely want to hand it off to a native AI desktop app — "have Claude read this and explain it to me," "have Codex open the repo and implement what this spec describes," "have Cursor apply this plan." The mechanical path today is six steps: copy the file path, switch apps, open a new chat, paste the path, type a prompt, attach the file. Every OK user does this manually, multiple times a day. Zero of OK's surfaces (sidebar, doc panel, command palette) expose any handoff affordance. (`grep -r 'claude://|codex://|cursor://' packages/ → no matches`, 2026-04-21.)

**Complication.** The 2026-04-21 deep-linking research established that atomic single-URL handoff IS available for two of the three target apps today (Claude Desktop via `claude://cowork/new?q=&folder=&file=` and `claude://code/new?q=&folder=&file=`, Codex via `codex://new?prompt=&path=&originUrl=`), is feasible but two-step for Cursor (`cursor /path` + `cursor://anysphere.cursor-deeplink/prompt?text=&mode=&workspace=`, single CursorJack-hardened confirmation modal), and requires a small matrix of per-app encoding / install-detection / fallback rules that isn't intuitive. The work has three simultaneous pressure points: (a) **customer friction** that's invisible in product analytics because it happens outside OK, (b) **GTM positioning** — "OK is the context layer for agents" is abstract today; a concrete "Open this page in Claude Cowork with the repo as workspace and the page as an attached file" button is the tangible demo, and (c) **platform precedent** — whatever handoff architecture ships here sets the pattern for Zed, Windsurf, VS Code, and future MCP-install handoffs. Separately: OK's Electron M1 that just merged (2026-04-21) introduced a hardcoded `shell.openExternal` scheme allowlist (`packages/desktop/src/main/shell-allowlist.ts` — D47 defense-in-depth against Shabarkin-class RCE) that allows only `https:`, `http:`, `mailto:`, `openknowledge:`. Extending that list to `claude:` / `codex:` / `cursor:` is a deliberate security-boundary decision, not an implementation detail — and OK still ships a web distribution (Vite + Hocuspocus via `open-knowledge start`) where `shell.openExternal` doesn't exist at all and URL-scheme invocation is a browser-side fire-and-forget. One coherent feature; two host shapes; one 1-way-door security-allowlist decision.

**Resolution.** Ship "Open in…" as a dropdown action surfaced on (at minimum) the per-doc header, the command palette, and — if coherent with the existing UX — the file sidebar context menu. For each target, use the strongest primitive verified by the research report: atomic single URL where it exists, two-step pattern where it's required. Gate per-target button state on install-detection. Route outbound URL-scheme invocations through the Electron main process (extending the shell allowlist with specific rationale + unit tests) in desktop builds, and through the browser `window.location.href = 'claude://...'` + timed web-fallback pattern in web builds. Fail gracefully and loudly when detection or invocation fails.

### Problem stress-test (5 probes)

- **Demand reality.** The user (Nick, CPO/CTO and primary dogfooder) confirmed live, 2026-04-21: fired ~14 candidate URLs during the research round, reported back on composer prefill / workspace selection / modal UX for each — and the closing conversation was "worth updating the evidence file (`evidence/claude-desktop-deep-links.md`) with the live-verified findings since it originally missed the `cowork/` and `code/` host routes entirely." That's real demand surfaced on a real session, not a hypothetical persona.
- **Status quo.** Without this, every OK-to-agent handoff is a six-step manual loop. OK's "knowledge layer for agents" positioning stays abstract because nothing in the product demonstrates the claim. Every demo walk-through requires the demoer to context-switch.
- **Narrowest wedge.** Ship "Open in Claude Cowork" FIRST, Claude-only. It's the single highest-capability target (atomic prompt + repeatable folder + repeatable file + no confirmation modal + clean web fallback to `claude.ai/new?q=`) and demonstrates the full pattern. Codex + Cursor are additive targets that reuse the same frontend affordance + IPC surface. (See PQ3.)
- **Observation.** Live-testing session on 2026-04-21 observed the end-to-end user flow from shell invocation through composer prefill. Four distinct Claude URL variations (`cowork/new?q=`, `cowork/new?q=&folder=`, `code/new?q=&folder=`, `cowork/new?q=&folder=&file=`) plus two Cursor variations (single and double encoding of prompts with literal `%` characters) each produced user-described outcomes matching the binary-probe prediction. That's behavioral verification at the highest confidence class available.
- **Future-fit.** URL-scheme handoff to agent desktop apps is a stable-and-growing pattern across 2026 — Linear's production registry has 19 tools (5 shell-exec + 14 URL-based); Mintlify's `contextual.options` has 14 identifiers; Zed joined Codex and Cursor in URL-based agent prompt seeding in January 2026. This is not a fad. The specific URL schemes CAN deprecate or consolidate (Codex's superapp consolidation announced 2026-03-19), but the need for a per-OK handoff-registry layer that shields OK from vendor URL-shape changes is the forward-fit design — this story lays the first plank.

## Value and goals

This story is the **concrete demonstration of OK's "knowledge layer for agents" positioning** — the place where abstract platform claims turn into a button a user clicks. The customer dimension is direct friction removal: one click replaces six steps of path-copying and app-switching, and the replaced path is something every dogfooder does multiple times a day already. The GTM dimension is positioning-to-evidence conversion: OK's pitch to agent-tool builders ("use OK as the shared knowledge layer") becomes more plausible when OK itself surfaces "Open this in Claude / Codex / Cursor" on every page — it's a marketplace-friendly affordance, not a vendor lock-in, AND it mirrors what Linear and Mintlify already ship (so it's consensus-validated). The platform-internal dimension is precedent: this is OK's first outbound-URL-scheme surface, AND it's the first time OK's shell-allowlist is extended from the M1 defensive baseline — the discipline established here (deliberate per-scheme rationale, unit tests, D47 cross-reference) becomes the pattern for future outbound schemes (Slack, Linear, Notion). The internal/efficiency dimension is load-bearing the research: the deep-linking report is 14 evidence files + ~3000 lines of prose; this story is the work that makes that research a shipped capability rather than a museum piece.

Intersection reasoning: **the customer-friction fix AND the GTM-positioning demo AND the platform-precedent for outbound scheme handoff are the same work** — you cannot ship the button without extending the allowlist (platform), and extending the allowlist without concrete user value wastes a defensive boundary (customer). The dimensions are genuinely mutually reinforcing, not three parallel rationalizations of the same feature. The load-bearing dimension is the **platform precedent**: the allowlist extension is a 1-way door (once `claude:` is allowed, it's hard to walk back without breaking users who built workflows on it), so the first extension must set the template — D47-aligned rationale, per-scheme tests, documentation of why each scheme is allowed. Getting that template right matters more than ship-date optimization on any individual target app.

**Observable success:**
- A user on the dogfood build (Electron M1 + this story) selects **"Open in Claude Cowork"** from any wiki page's doc-panel header. Claude.app foregrounds in the Cowork tab with the repo as workspace, the current page as an attached file, and an OK-composed prompt pre-filled in the composer (user did NOT type it — OK generated it from page context). Zero confirmation modals beyond Claude's own (which is none — `prefillOnly: true, source: "external"`).
- Same user selects **"Open in Claude Code"** → same behavior but lands in the Code (Epitaxy) tab via webview nav to `/epitaxy?q=&folder=&file=&src=external`. Four dropdown rows total (Cowork + Code + Codex + Cursor), per PQ2 LOCKED.
- Same user selects **"Open in Codex"** on the same page. Codex.app foregrounds on a new thread with the repo resolved via `path=` or `originUrl=<git>` and the OK-composed prompt pre-filled.
- Same user selects **"Open in Cursor"** on the same page. Cursor's open-knowledge window (or the named workspace window) is focused and shows exactly one CursorJack prompt-approval modal containing the prompt rendered as clean text — no literal `%20` / `%25` / mojibake visible — with `mode=agent` applied.
- All four targets ship in parallel (PQ3 LOCKED) — no Claude-first-then-Codex-then-Cursor sequencing.
- User on the web build (`open-knowledge start` served via browser) attempts the same four actions. When the corresponding target is installed: target app foregrounds. When not installed: the disabled-row + tooltip UX (see below) applies — identical to Electron per PQ6 LOCKED.
- User without any target app installed sees all four rows in a disabled state with a per-option tooltip explaining the prerequisite and linking to the install page. Claude rows' tooltips additionally expose a secondary "Open in claude.ai instead →" action that routes to `https://claude.ai/new?q=<OK-composed-prompt>` in a new browser tab — explicit user opt-in, no auto-fallback. Codex + Cursor have no web fallback available. No click produces an error modal; no click produces silent nothing.
- Encoding correctness: a page path containing `%` characters, em-dashes, spaces, and multi-byte UTF-8 (`/Users/who/My %Project — docs/café-notes.md`) + an OK-composed prompt mentioning percentages render as the exact strings in the target app's composer across all three apps — no literal escape sequences visible, no silent character substitution.
- URL length: no user-visible truncation UX in v0 (PQ7 LOCKED) — OK's composer bounds its own generated prompt to stay under each target app's documented cap by construction.

## Invariants

- **I1 — Atomic single-URL handoff for Claude and Codex.** "Open in Claude Cowork," "Open in Claude Code," and "Open in Codex" are each ONE URL-scheme invocation carrying prompt + workspace (and, for Claude, optional file attachment) in a single click. No pre-step. No follow-up click. Observable: the frontend dispatcher calls exactly one `openExternal` invocation per user click; integration test asserts single invocation and correct URL shape.
- **I2 — Two-step-with-single-modal handoff for Cursor.** "Open in Cursor" performs (step 1) `cursor /path` to focus/create the workspace window, (step 2) after a small settle delay, the prompt URL with `workspace=<basename>` and `mode=<agent|ask|debug|plan>`. Exactly ONE CursorJack confirmation modal fires per user click. No additional trust dialog, no workspace-switch dialog, no second modal. Observable: end-to-end test in Electron (or manual live-test script against a dogfood machine) counts modal-opens and asserts == 1.
- **I3 — Encoding correctness for OK-composed payloads (PQ5-aware).** Per-app encoding rule is applied correctly: single-encode for Claude + Codex; double-encode for Cursor (matches Linear production; mechanism verified as two-pass-decode-with-error-recovery in `reports/.../evidence/cursor-encoding-empirics.md`). Since OK composes the prompt (PQ5), the encoding class that matters is (a) file paths — which can contain `%`, em-dashes, unicode, spaces, mixed-case — and (b) OK's structured prompt content (titles, frontmatter values, possibly short excerpts). Observable: unit test corpus covering file paths like `/Users/who/My %Project — docs/café-notes.md` plus representative structured prompt fragments, asserting the target app's rendered composer text shows the exact strings with no `%20` / `%25` / mojibake visible and no silent character substitution.
- **I4 — Graceful degradation on "target app not installed."** If the target scheme is not registered on the user's OS, the UI shows the handoff option in a disabled state with a per-option tooltip, OR (Claude-only) offers a web fallback to `https://claude.ai/new?q=<prompt>` in a new browser tab. The click NEVER produces: (a) a silent no-op with no feedback, (b) a macOS `kLSApplicationNotFoundErr` OS-level error dialog from the raw shell invocation, (c) a thrown exception reaching the React error boundary. Observable: test matrix covers all 2³ = 8 combinations of (Claude installed, Codex installed, Cursor installed); every option state is either enabled-and-launches, disabled-with-tooltip, or (for Claude) enabled-but-web-fallback.
- **I5 — No auto-execute.** No handoff path causes the target app to START the prompt automatically. The target app's composer is pre-filled; the user presses Enter to send. This holds by native behavior today (Claude's `prefillOnly: true, source: "external"` dispatch, Codex's prefill-composer default, Cursor's CursorJack modal gate) but is an INVARIANT OK maintains going forward — if a future vendor adds a `send=true` URL param, OK does NOT include it. Observable: URL-builder unit tests assert no `send`/`autorun`/`execute` params are emitted.
- **I6 — Shell-allowlist extension carries per-scheme rationale and is reviewed against D47.** The extension to `packages/desktop/src/main/shell-allowlist.ts` adds `claude:`, `codex:`, `cursor:` ONLY, each with a docstring referencing the specific URL shape OK emits for that scheme (e.g., "`claude://cowork/new?q=&folder=&file=` — dispatched only from the outbound-handoff IPC channel; payload is validated at the renderer before `openExternal` is invoked"). Bulk-allowing `*:` is rejected; adding schemes without a rationale docstring is rejected at PR review. Observable: `shell-allowlist.ts` before-and-after diff on the PR shows exactly 3 new entries with 3 docstring additions; unit test asserts the allowlist's exact set is `{https:, http:, mailto:, openknowledge:, claude:, codex:, cursor:}` — no more, no less.
- **I7 — Dual-host feature parity at the customer-observable level.** The Electron desktop build and the `open-knowledge start` web build both render the "Open in…" affordance on the same surfaces. Feature parity is "user sees the button and clicking it has a sensible outcome," NOT "internal dispatch mechanism is identical" — the two hosts use different primitives (IPC channel vs `window.location.href`), different install-detection strategies (Electron main-process API vs server-MCP-tool subprocess), and different fallback UX (disabled button with tooltip in Electron vs best-effort web-fallback in browser). The spec must nail both; the seed says both must exist. Observable: Playwright E2E test in each host configuration renders the dropdown on at least one surface (doc panel header) and clicks produce the documented per-host outcomes.
- **I8 — OK's prompt-composer stays under per-app URL-length caps by construction.** Since OK composes the prompt (PQ5), the cap-handling story is construction-time, not dispatch-time: the composer bounds its own generated prompt to stay under each target app's documented cap minus a safety margin. Approximate per-app budgets (spec will pick exact values): Claude `q` ≤ 4K, Codex `prompt` ≤ 4K, Cursor `text` ≤ 8K. File paths and workspace paths are not in the cap — they're short by nature. Observable: unit test feeds the composer pages with progressively larger context (title + frontmatter + excerpt) and asserts the emitted URL stays under the per-app cap; no truncation-with-visible-footer UX required at v0 because the composer simply doesn't produce over-budget content.

## Constraints

- **C1 — Shell allowlist is the gate, D47 is the rationale.** Any outbound URL-scheme invocation in the Electron build MUST pass `checkOutboundUrl()` in `shell-allowlist.ts`. The spec must extend the allowlist, cross-reference D47 (`specs/2026-04-11-electron-desktop-app/SPEC.md`), and add per-scheme unit tests. Bypassing the allowlist (calling `shell.openExternal` without the check, or adding `*:`) is rejected at PR review. The allowlist is the ONLY place where new outbound schemes become possible.
- **C2 — Dual host support is non-optional.** OK has two actively-supported distribution shapes: Electron desktop (via `packages/desktop/`) and web-browser-served-from-CLI (via `open-knowledge start`). The "Open in…" feature must ship to both. Detecting `typeof window.okDesktop === 'undefined'` (the Electron-bridge sentinel per `electron-desktop-m1.md` changeset) is the host switch.
- **C3 — Install-detection strategy differs per host and is constrained by platform APIs.** Electron main process: `app.getApplicationInfoForProtocol('claude')` (Electron 25+, OK ships 41.2.1). Web browser served by OK's server: the server-side MCP tool runs `osascript -e 'id of app "Claude"'` on macOS / `reg query` on Windows / `xdg-mime query default x-scheme-handler/claude` on Linux and returns the result via an API endpoint the frontend polls. Cannot cleanly detect from pure-browser JavaScript; the server-side MCP path is the mechanism.
- **C4 — Electron 41.2.1 (current) supports the required APIs.** `app.getApplicationInfoForProtocol` was added in Electron 25; `shell.openExternal` is stable since the early days. No Electron-version bump required for the APIs; if M2–M7 bump Electron forward, verify no regression.
- **C5 — Respect each target app's idiomatic URL shape.** Do not invent a unified "OK handoff schema" that maps to each target. Use each target's documented shape (Claude: `q` / `folder` / `file`; Codex: `prompt` / `path` / `originUrl`; Cursor: `text` / `mode` / `workspace`). The URL-builder lives per-target; a shared "handoff payload" type is internal.
- **C6 — Cursor's router requires double-encoding; Claude and Codex do not.** Per-target encoding rule is a hard constraint, not a spec-time detail — mis-implementation causes silent user-prompt corruption. Source: `reports/deep-linking-ai-desktop-apps-2026/evidence/cursor-encoding-empirics.md` + `evidence/linear-ai-deeplinks-extraction.md` (Linear production applies `encodeURIComponent(encodeURIComponent(x))` to Cursor / Copilot / Windsurf and single for the others).
- **C7 — Appetite: single-iteration feature, not a multi-quarter platform project.** Scope the first-ship to the three researched targets + both hosts + install-detection + encoding + allowlist extension. A generic "handoff registry" that supports user-added URL templates (Linear ships this; Mintlify ships this) is out of scope — it's a natural follow-up but is its own story.

## Non-goals

- **[NEVER] Handoff that auto-executes the prompt without user-in-the-loop confirmation.** This is the trust-boundary invariant (I5). Native behavior today already precludes auto-execute on all three apps, and OK does NOT add a URL param or post-dispatch automation that would change that. Reason: prompt-injection / bidi / Trojan-Source attack class (CVE-2021-42574); Cursor's CursorJack hardening and Zed's `ExternalSourcePrompt` newtype-at-boundary pattern are the industry precedents for WHY this must hold.
- **[NEVER] `shell.openExternal` bypass.** All outbound URL dispatch in the Electron host goes through `ok:shell:open-external` IPC → `checkOutboundUrl()` → `shell.openExternal`. There is no direct-dispatch path, no "trusted renderer" carve-out, no testing-mode backdoor. Reason: D47 defense-in-depth against Shabarkin 2022 "1-click RCE" chain.
- **[NEVER] Silent truncation of user prompts.** See I8. Truncation may happen, but user always sees an inline indicator AND the truncated content is recoverable (either via a "copy full prompt to clipboard" affordance or by the OK page itself remaining the source of truth).
- **[NOT NOW] Zed / Windsurf / VS Code handoff.** The research characterized all three (Zed has `zed://agent?prompt=`, Windsurf has `windsurf://cascade?prompt=`, VS Code has no chat URL), but v0 scope is the Big 3 AI desktop apps that drove the research commission. Revisit if: (a) a user requests Zed specifically, (b) we decide to build a "handoff registry" with pluggable targets, OR (c) one of the Big 3 deprecates / consolidates (e.g. OpenAI's announced 2026-03-19 superapp consolidation of ChatGPT + Codex + Atlas) and we need to rebalance targets. Full precedent coverage in the research report's Addendum A.
- **[NOT NOW] MCP-install handoff via URL.** Cursor, VS Code, Mintlify, and Linear all support URL-based MCP server installation (`cursor://.../mcp/install?config=<b64>`, `vscode:mcp/install?<json>`, etc.). Including it in v0 would pull in Mintlify's 14-identifier schema complexity and muddy the "Open this page in X" customer story. Revisit when: OK's MCP surface is stable and a user asks "how do I hand off OK's MCP config to Cursor." Separate story — likely paired with publishing OK's own MCP server config as copy-paste artifacts.
- **[NOT NOW] Handoff-registry platform (user-added targets).** Linear ships a user-defined `customUrl` / `customTerminalScript` hook; Mintlify ships a 14-identifier schema. First-ship is hardcoded to Claude + Codex + Cursor (the researched set). Revisit when: ≥2 users request a specific fourth target we don't support, OR we start adding targets faster than once-per-story (> 4 targets total).
- **[NOT UNLESS] "Open in Claude" as a SINGLE option rather than separate Cowork + Code.** Cowork and Code are distinct Claude Desktop tabs reached via distinct URL routes (`claude://cowork/new` vs `claude://code/new`). v0 ships them as two separate dropdown entries. Revisit only if user-research signals that users are confused by having two "Claude" rows (at which point we'd collapse to one with a tab-default preference). Argument for keeping two: power users explicitly want Cowork-vs-Code routing control; the dropdown is small (3 entries becomes 4), not overwhelming.
- **[NOT UNLESS] Multi-page handoff — select N pages, hand off all at once with combined context.** Claude's `folder=` and `file=` params are both repeatable; Codex's `path=` is scalar. Technically Claude could do multi-file, but the UX for "select multiple pages first" is a different surface than "this page's Open-in button." Revisit only if: a user specifically asks for multi-page handoff AND we've shipped multi-select elsewhere in the UI.
- **[NOT UNLESS] Prompt templating / saved prompts per target.** E.g., "Ask Claude to review this page as an expert reviewer." Linear ships template variables (`{{issue.identifier}}`, `{{context}}`). First-ship lets the user type a free-form prompt. Revisit if: users copy-paste the same prompt prefix across handoffs repeatedly (we'd see this in dogfood feedback), OR we add a "prompt library" platform capability.
- **[NOT NOW] Embedding-aware UI — hide or reshape the "Open in…" affordance when OK's web viewer is itself embedded inside a third-party desktop shell** (Claude Desktop, Codex, Cursor, or a generic MCP / electron-based client rendering OK as an iframe or webview). The recursive case — "Open in Claude Cowork" rendered inside Claude Cowork's embedded OK viewer — is nonsensical UX. Other embedding-time adjustments likely needed: auto-collapse the left sidebar, adjust chrome density, maybe suppress header actions that duplicate the host's own. Detection candidates: `window.parent !== window` / `window.top !== window.self` (iframe sniff), host-injected globals, explicit `?embedded=1` / `?host=<app>` query param, `document.referrer`, User-Agent sniffing, or a `postMessage` handshake contract we offer to embedding hosts. v0 ships without this detection — explicit recognition here prevents the v1 discovery moment ("oh crap, Claude Cowork is showing 'Open in Claude Cowork'"). Revisit when: a partner indicates intent to embed OK in their shell, OR OK itself starts embedding — see XQ5 below.

## Acceptance criteria

- **AC1 — Claude Cowork handoff (Electron, happy path).** On a dogfood Electron build with Claude Desktop installed, the doc-panel header of any wiki page has an "Open in…" dropdown with four rows: Claude Cowork / Claude Code / Codex / Cursor. Selecting "Claude Cowork" fires `claude://cowork/new?q=<OK-composed-prompt>&folder=<repo-abs>&file=<page-abs>`. Claude.app foregrounds on the Cowork tab with all three pre-filled; no confirmation modal appears; the user sees the pre-filled composer and presses Enter to execute.
- **AC2 — Claude Code (Epitaxy) handoff.** Same scenario, "Claude Code" selected. Fires `claude://code/new?q=&folder=&file=`. Claude.app foregrounds on the Code tab (Epitaxy route `/epitaxy?q=&folder=&src=external`). Same single-click semantics as AC1.
- **AC3 — Codex handoff.** Same scenario, Codex Desktop installed. Selecting "Codex" fires `codex://new?prompt=<OK-composed-prompt>&path=<repo-abs>&originUrl=<git-remote>`. Codex.app foregrounds on a new thread, resolves workspace via `path=` (falls back to `originUrl=` match), prompt pre-filled. Single click, no modal.
- **AC4 — Cursor handoff with single modal.** Same scenario, Cursor.app installed. Selecting "Cursor" executes two-step dispatch: first `cursor /path/to/repo` (via the desktop host's outbound-command IPC OR, on web, via a server-side `/api/handoff/open-folder` endpoint per TQ4), settle delay, then `cursor://anysphere.cursor-deeplink/prompt?text=<double-enc>&mode=agent&workspace=<basename>`. Exactly one CursorJack prompt-approval modal appears. No workspace-switch modal. Prompt renders as clean text in the modal (no `%20` / `%25` / mojibake). `mode=agent` is applied end-to-end (confirmed in-product by the composer's mode indicator).
- **AC5 — Install-detection + disabled state.** When a target app is not installed (on either Electron or web build), its dropdown row renders in a disabled state. Tooltip on hover: "Requires [Claude Desktop | Codex Desktop | Cursor] — install from [link]". For Claude rows specifically, tooltip includes a secondary affordance: "Open in claude.ai instead →" (the PQ6 explicit-opt-in web-fallback). Clicking a disabled row's primary button does nothing; clicking the secondary tooltip action opens `https://claude.ai/new?q=<OK-composed-prompt>` in a new browser tab. No error modal, no silent failure.
- **AC6 — Four handoff surfaces shipped in parallel.** Per PQ3 LOCKED, all four rows (Claude Cowork + Claude Code + Codex + Cursor) ship in the same iteration. Per-target test coverage at unit + integration tier; AC1–AC4 are all in the v0 green-set, not sequenced.
- **AC7 — Encoding correctness for paths + OK-composed prompts.** Test paths containing `%`, em-dashes, spaces, multi-byte UTF-8 (`/Users/who/My %Project — docs/café-notes.md`) + OK-composed prompt fragments render as the exact strings in each target app's composer. No literal escape sequences visible. No silent character substitution (`%41`-style case). Double-encoding applied correctly for Cursor; single for Claude + Codex.
- **AC8 — Allowlist extension tests pass.** Unit tests in `packages/desktop/src/main/shell-allowlist.test.ts` (new or extended) assert: `checkOutboundUrl('claude://cowork/new?q=x&folder=y')` → `{ok:true}`; same for `codex://` and `cursor://`; `checkOutboundUrl('file:///etc/passwd')` → `{ok:false}`; `checkOutboundUrl('ms-msdt:/id/PCWDiagnostic')` → `{ok:false}`. The allowlist's exact contents are `{https:, http:, mailto:, openknowledge:, claude:, codex:, cursor:}` and the assertion is an exact-set match (not a subset-contains check).
- **AC9 — Zero dispatch bypasses.** Static code search (ripgrep in CI or PR-review checklist): zero usages of `shell.openExternal(` in `packages/desktop/src/` outside `main/index.ts`'s single IPC handler. Zero usages of `window.open('claude://...')` or `window.location.href = 'claude://...'` in `packages/app/src/` outside the single outbound-handoff dispatch module.
- **AC10 — Composer stays under per-app URL caps by construction.** Unit test feeds OK's prompt-composer a corpus of pages (small / medium / pathologically-long-frontmatter / pathologically-long-titles) and asserts the emitted `q=` / `prompt=` / `text=` value length stays under each target app's budget (spec-pinned per I8). No user-visible truncation UX in v0 (per PQ7 LOCKED).
- **AC11 — No phone-home.** Static code search asserts zero analytics-SDK imports (no Mixpanel, PostHog, Segment, Amplitude, Google Analytics, etc.) introduced by this feature. No new network endpoints other than the narrow `/api/handoff/open-folder` TQ4 shell-out endpoint (local-only) and the `/api/installed-agents` detection endpoint (local-only). Matches XQ3 LOCKED.
