# Story: Migrate desktop typed IPC to a library-backed contract

**Last verified:** 2026-04-28
**Status:** /spec-ready

## Problem (SCR-lite)

**Situation.** Open Knowledge's desktop package implements its rendererâ†”main typed IPC as a hand-rolled, channel-map TypeScript contract: a triplicated `OkDesktopBridge` interface (canonical at `packages/core/src/desktop-bridge.ts`, mirrored at `packages/desktop/src/shared/bridge-contract.ts` and `packages/app/src/lib/desktop-bridge-types.ts`), three custom wrappers (`createHandler` / `createInvoker` / `sendToRenderer`), and four shape-only drift catchers in `m1-smoke.test.ts` policed by a Bun integration test the README mis-describes as a Biome GritQL rule. Renderer code calls `window.okDesktop.<namespace>.<method>(...)`; preload exposes that surface via `contextBridge`; main registers handlers through the typed wrappers. The pattern works â€” and pattern-locked the team into a maintenance shape it never deliberately chose.

**Complication.** Three pressures intersect right now. **(1) The migration trigger is tripped.** The in-source comment at `packages/desktop/src/shared/ipc-channels.ts:13â€“15` (current authority, supersedes the older "6th channel" framing in PR #270's body) says: *"Scale-match trigger (FU-3): at >20 channels, migrate baseline to `@electron-toolkit/typed-ipc` or `@egoist/tipc`. Currently 21 â€” past the trigger; migrate before adding more."* **(2) The triplication tax is being paid in active work.** PR #345 (in-flight, by amikofalvy) adds three more channels for the `find` namespace through the current pattern â€” every new namespace duplicates the type into three files and relies on a shape-only walker that cannot detect signature-level drift. **(3) Adjacent debt has accumulated** alongside the typed-IPC layer: README inaccuracies (the "Biome GritQL rule" framing; the "only one main-process file" framing contradicted by `auto-updater.ts` + `mcp-wiring.ts`), three latent channels declared in the typed map but never produced or consumed (`ok:project:switching`, `ok:menu-action`, `ok:project:get-info`), and the drift catcher's one-level-nesting granularity gap that documents itself as future work but never gets revisited. The greenfield posture (no deferred tech debt) and the foundation framing (next desktop work â€” utility-process IPC, per-window subprocess concerns â€” should inherit a stronger substrate, not extend a pattern past its scale-match) close the case for doing this now rather than next channel.

**Resolution.** Replace the hand-rolled wrappers with a library-backed typed-IPC contract. Retire the triplication. Retire the shape-only drift catcher and the regex Bun-test. Do the full adjacent sweep in the same story: rewrite the desktop README's IPC section, delete the three latent unused channels, and ensure whatever drift-detection the new substrate provides catches signature-level drift the current walker misses. Library choice is DEFERRED to /spec â€” candidates listed below.

## Value and goals

**The DX dimension is the drift-safety dimension is the foundation dimension.** Eliminating triplicated contract maintenance (each new namespace currently writes the type into three files) is the same move as catching signature-level drift the current shape-only walker silently misses (a library's TS types fail at compile-time on return-type or arg-type drift, not just on member-set drift) â€” which is the same move as making the next desktop expansion (utility-process IPC, per-window subprocess management, deep-link channel growth) cheaper to add, because "add a typed IPC channel" stops being a four-file edit policed by a custom regex test. The contract-size dimension reinforces this: ~1000 lines of duplicated TypeScript collapses to a single source of truth, and the bespoke `no-loosely-typed-webcontents-ipc.test.ts` (mis-documented as a Biome rule, file-walk regex with a hand-maintained allowlist) collapses to whatever the chosen library provides natively.

**The ecosystem-alignment dimension is load-bearing for new contributors.** OK's hand-rolled pattern looks similar to `@electron-toolkit/typed-ipc` (channel-map, listener+handler split) but isn't it â€” every new contributor reads two surfaces (the wrapper helpers AND the contract triplication) before understanding the discipline. Adopting whichever library Electron tooling has converged on reduces "why doesn't OK look like other Electron apps" friction at the moment Andrew or any future contributor opens `packages/desktop/`.

**Greenfield-cut posture matters.** The full-sweep scope (migration + README + latent channels + drift-catcher granularity) is the deliberate stance: no deferred dust. A focused migration would leave the README inaccurate, three dead channels in the typed map, and the drift-catcher gap documented-but-unfixed â€” and history says those would compound until the next "let's do a sweep" round. Doing them together costs less than doing them separately and avoids the deferred-tech-debt failure mode the team has explicitly ruled out for greenfield work.

**Observable success.** When this story is done: (1) one canonical `OkDesktopBridge` declaration exists in the repo, not three; (2) the next desktop contributor adds a new IPC channel by editing one file; (3) signature drift (changing a return type from `Promise<void>` to `Promise<boolean>` in one place) breaks the build at the right place, not silently passes the shape-only walker; (4) the README's IPC section accurately describes the runtime in a way that survives `git grep` audits; (5) `bun run check` is the only quality gate.

## Invariants

- **Zero raw IPC primitives outside sanctioned wrappers.** No `ipcMain.handle`, `ipcMain.on`, `ipcRenderer.invoke`, `ipcRenderer.on`, `ipcRenderer.once`, or `webContents.send` appears in `packages/desktop/src/` outside the library-provided helpers (or whatever thin adapter wraps them, if any). Falsifiable: regex sweep equivalent to the current `no-loosely-typed-webcontents-ipc.test.ts`.
- **Signature-level drift detection.** Changing a return type or argument type in one place (and not the consumer site) fails `bunx tsc --noEmit` or `bun run check`. The current shape-only walker's gap (one-level-nested member-set only â€” see `m1-smoke.test.ts:158â€“164`) is closed. Falsifiable: a deliberate signature mutation test where the regression escapes only if the gap remains.
- **Renderer continues to receive a typed `window.okDesktop` global.** Exact shape DELEGATED to /spec; existence is invariant. Falsifiable: a renderer integration test asserts `window.okDesktop` is defined and a representative call (e.g. `okDesktop.shell.openExternal('https://example.com')`) routes through to the main-process handler with the correct typed payload.
- **Single-PR clean cut.** The migration lands in one PR â€” no half-migrated state on `main`. Greenfield posture; falsifiable by repo state at merge time (no parallel old-and-new contract on main between PR landings).
- **Three latent channels resolved, not deferred.** `ok:project:switching`, `ok:menu-action`, `ok:project:get-info` either ship a producer/consumer pair OR are deleted from the typed map. Falsifiable: each declared channel name has both an emit site and a subscriber after this story.
- **`bun run check` remains the canonical gate.** No new quality gate (separate `bun run check:ipc`, separate CI job) is introduced. Falsifiable: `bun run check` definition in `package.json` changes additively only â€” does not split into multiple sibling commands.
- **README accuracy.** `packages/desktop/README.md` IPC section after migration accurately describes: (a) what enforces the discipline, (b) which files are exempt and why, (c) where the canonical contract lives. Falsifiable: a paragraph-by-paragraph audit against the post-migration runtime returns zero contradicted claims.

## Constraints

- **The triplication root cause is `moduleResolution: bundler`.** Per `packages/desktop/src/shared/bridge-contract.ts:1â€“20`, the contract is duplicated rather than re-exported from core because a barrel re-export pulls in core's mdast/CRDT module-augmentation tree, which fails to resolve under desktop's `moduleResolution: bundler`. Bounds the solution: the library choice / contract structure must either sidestep this resolution failure OR solve it (e.g. by moving the canonical contract into a leaf module that doesn't transitively import the augmenting tree). Cannot just paper over it with the library types and re-introduce the resolution failure.
- **Three subsystems use raw `ipcMain.removeHandler` legitimately.** `auto-updater.ts:606`, `mcp-wiring.ts:885,914,921,928`, and `main/index.ts` rely on `removeHandler` for teardown semantics â€” that primitive is currently unbanned because the regex bans `handle`, not `removeHandler`. Bounds the library choice: it must support equivalent teardown OR provide a documented replacement pattern.
- **PR #345 is in-flight.** Adds 3 channels (`ok:find:start`, `ok:find:stop`, `ok:find:result`) through the current pattern. Bounds the sequencing: this story either (a) blocks #345 (Andrew rebases onto migration), (b) waits for #345 to land (then absorbs the larger surface), or (c) coordinates with Andrew to land #345 simultaneously through the new substrate. /spec resolves.
- **Renderer call sites are extensive.** `window.okDesktop.<namespace>.<method>(...)` is referenced across `packages/app/` (renderer) AND in `packages/app/tests/stress/fixtures/handoff-mocks.ts` which uses `satisfies OkDesktopBridge` as a compile-time fence. Bounds the migration plan: rewrites must either be mechanical (codemod-able) or scoped to fewer than ~20 distinct files.
- **Owner and curator.** Implementation lands in Andrew's (amikofalvy's) area â€” desktop. Nick (CPO/CTO) is curating the framing for this story. Coordination is required at /spec time, not deferred to implementation.

## Non-goals

- **[NOT NOW]** Migrate the mainâ†”utility-process `parentPort` MessagePort handshake to the same library. Different transport (parentPort, not ipcMain/ipcRenderer); the chosen library is unlikely to cover both. **Revisit if:** the chosen library adds parentPort support, OR utility-process IPC grows past ~5 distinct message types (currently 2: `initâ†’ready` handshake + `debug-request` correlation map).
- **[NOT NOW]** Adopt Zod / runtime validation at the IPC boundary. The current contract is compile-time TypeScript only. Adding runtime validation is a separate scope decision with its own bundle/perf calculus. **Revisit if:** a single sanitized-handler regression escapes type-checking AND the failure is observable in production telemetry, OR if /spec's library comparison surfaces that the candidate already ships runtime validation as a free side-effect.
- **[NOT NOW]** Migrate the rendererâ†”utility WebSocket (Hocuspocus `/collab`) or HTTP `/api/*` paths to the typed-IPC discipline. Those are deliberately not typed IPC â€” they ride a different transport and serve a different concern (CRDT, REST). **Revisit if:** the WebSocket/HTTP boundary develops its own type-drift problem that mirrors the typed-IPC one.
- **[NEVER]** Replace the typed-IPC discipline with raw `ipcMain.handle` + ad-hoc `contextBridge.exposeInMainWorld({})` exposure. Permanent exclusion: the discipline is a precondition for `sandbox: true` + `contextIsolation: true` â€” this is a security-boundary stance, not a stylistic preference.
- **[NEVER]** Move primary rendererâ†”main IPC over to MessagePorts as the transport. MessagePorts are the rendererâ†”renderer mechanism per Electron's architecture; the `invoke`/`handle` pair is canonical for rendererâ†”main. Migrating off it would be a 1-way door against ecosystem direction.
- **[NOT UNLESS]** Keep the current `createHandler` / `createInvoker` / `sendToRenderer` wrappers AND add the chosen library on top (hybridization). Default position is delete the bespoke wrappers. **Revisit only if:** the chosen library's API shape requires a thin adapter to preserve a load-bearing teardown or framing concern (e.g. the per-window `webContents` targeting in `sendToRenderer`'s broadcast helper), in which case the adapter must be â‰¤30 lines AND explicitly documented as a thin shim, not a parallel surface.

## Acceptance criteria

1. **Single canonical contract.** `grep -rn "interface OkDesktopBridge" packages/` returns exactly one match. The two former mirror files (`packages/desktop/src/shared/bridge-contract.ts`, `packages/app/src/lib/desktop-bridge-types.ts`) are either deleted or reduced to thin re-exports of the canonical.
2. **Drift catcher resolution.** The four catchers in `m1-smoke.test.ts` (`OkDesktopBridge` triplication, `EditorId` literal-union triplication, `KeyringSmokeResult` shape, `SWITCH_PROJECT_LABEL_WITH_ELLIPSIS` value drift) are refactored â€” bridge-triplication catcher is removed (no longer applicable); the others are evaluated for whether the same root cause exists, and either retained, refactored, or removed with documented rationale.
3. **Regex test resolution.** `packages/desktop/tests/integration/no-loosely-typed-webcontents-ipc.test.ts` is removed OR refactored. If retained, its allowlist reflects post-migration reality (not the current 7-file list with hardcoded `shared/ipc-*.ts` paths).
4. **Signature drift test.** A new test (or the refactored regex test) catches signature-level drift â€” e.g., changing `openExternal(url: string): Promise<void>` to `Promise<boolean>` in the contract without updating the renderer call site fails `bun run check`.
5. **README accurate.** `packages/desktop/README.md` IPC section is rewritten. No claim that the rule is a Biome GritQL plugin (it isn't; the current README at L114 is wrong). No claim that `main/index.ts` is "the only file allowed to touch raw electron IPC primitives" (auto-updater + mcp-wiring legitimately do too, via `removeHandler`).
6. **Latent channels resolved.** `ok:project:switching`, `ok:menu-action`, `ok:project:get-info` each have either (a) a producer call site in main + a renderer consumer, or (b) deletion from the typed map AND the contract AND any preload subscription wrappers. No "declared but never used" channels remain.
7. **Renderer integration smoke.** A test asserts `window.okDesktop` is exposed, and at least one representative call per namespace (`dialog`, `shell`, `clipboard`, `project`, `navigator`, `update`, `seed`, `skill`, `mcpWiring`) routes correctly through the new substrate.
8. **`bun run check` green.** Existing `packages/desktop` test suite passes (in-memory IPC bus tests, M1 smoke, mcp-wiring, auto-updater, etc.) â€” adapted to the new substrate as needed but not silently weakened.
9. **Bundle ceiling holds.** Renderer bundle size limits (per `packages/app`) do not increase by more than the threshold /spec sets. Threshold itself is a /spec decision; the AC is that some bound is enforced and met.
10. **Migration is one PR.** No "phase 1 / phase 2" split. Coordinates with PR #345 such that on `main` the typed-IPC surface is either fully old or fully new, never partially migrated.
