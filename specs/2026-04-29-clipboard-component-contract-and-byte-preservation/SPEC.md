## 1) Problem statement

**Situation.** Open Knowledge is a dual-mode markdown-canonical editor (TipTap WYSIWYG over `Y.XmlFragment` + CodeMirror Source over `Y.Text`, bridged via Y.Doc) with a shipped clipboard pipeline: 5-branch WYSIWYG paste dispatcher + 4-branch Source paste dispatcher, mdast canonical hub, 9 vendor cleanup rehype plugins, FR-20 escape contract, FR-21 chunked Source insert. PR #310 layered 5 canonical descriptors (Callout / img / video / audio / Accordion) + a canonical/compat descriptor split + a lowercase media pivot, and shipped **Option B**: name-keyed cross-app outbound emission of native `<img>`/`<video>`/`<audio>` HTML for the three media descriptors only.

**Complication.** Three concurrent pressures, none addressable by the existing pipeline:

1. **OK→OK silent identity loss for media descriptors.** PR #310's lowercase pivot makes TipTap's built-in `Image` extension parseDOM rule (`tag: 'img[src]:not([src^="data:"])'`, priority 50) win over `JsxComponent`'s `data-jsx-component` rule when the dispatcher's `data-pm-slice` Branch C runs PM-native parseFromClipboard. `<img>` JSX paste flips silently to markdown form (`![alt](src)`). The CTO hit this regression personally during normal authoring, satisfying the 2026-04-16 SPEC's NG1 revisit criterion verbatim.
2. **OK→OK conspicuous degradation for capitalized JSX.** `<Callout>`, `<Accordion>`, and any future custom component whose name isn't in `HTML_PRIMITIVE_TAGS` emit `<pre class="mdx-component"><code>...</code></pre>` to text/html. On paste back, `CodeBlockFidelity` (priority 60, `tag: 'pre'`) matches and the literal MDX source lands inside a fenced code block on disk. Bytes survive but as code, not component.
3. **No architectural pattern for custom-component clipboard semantics.** Option B is a name-keyed special case (`HTML_PRIMITIVE_TAGS = {img, video, audio}`), not a contract. Every future custom OK component (iframe per the planned report; Frame v2; AI-native primitives) inherits the same regression class on day one until someone adds it to the carve-out. There is no per-descriptor declaration of "what should this look like in cross-app HTML."
4. **Cross-machine paste failure for OK-canonical bytes.** The `is-markdown.ts` discrimination heuristic doesn't recognize JSX shape signals. A markdown file containing `<Callout>` shared via email/Slack/file pastes into OK as verbatim plaintext (Branch E text/plain fallthrough), even though the bytes are OK-canonical. The OK content one user sends is unrecognized by the OK another receives.

**Resolution.** Promote NG1 to in-scope; broaden it. Three independent fixes, all threaded through PM's documented hooks per precedent #19(b) (no DOM-level event handlers):

1. **Reorder the dispatcher to FR-13-first** (D5/D13). Move the markdown-first ambiguity tiebreak ahead of `data-pm-slice` Branch C so OK→OK paste goes through the canonical text/plain markdown path before PM-native parseFromClipboard can fight TipTap's parseDOM rules. Cross-PM-editor interop (Linear, Outline, BlockNote) is preserved — their text/plain is also markdown-canonical (verified 2026-04-30); FR-13 catches them too with equivalent results.
2. **Extend `is-markdown.ts` with six new signals** (D8/D18). Add blockquote (`> `), inline code (`` `code` ``), paired emphasis (`**bold**`/`__bold__`/`~~strike~~`) — three peer-survey gaps that AI-chat copy-button output trips. Plus capitalized JSX (`/<[A-Z]\w*[\s\/>]/`), lowercase JSX-with-attr (`/<[a-z]+\s+\w+="[^"]*"/`), and raw-HTML-inline (`/<[a-z]+>[^<\n]*<\/[a-z]+>/`) — JSX-shape and inline-HTML signals unique to OK's MDX-native authoring posture, needed for cross-machine D4 and the Underline/Highlight raw-HTML round-trip.
3. **Live-DOM walker for cross-app text/html outbound emission** (D19). At copy time, `clipboardSerializer.serializeFragment` walks the live editor's rendered DOM via `view.nodeDOM(pos)`, clones each node tree, copies allowlisted computed styles inline via `getComputedStyle`. Single source of truth = whatever the React component rendered + whatever CSS resolved. Generic across all descriptors. Pre-flight: chevron-as-real-DOM refactor for Callout collapsible + Accordion + HtmlDetailsAccordion (FR-13). Per-descriptor `toClipboardHast` becomes optional override (none in v1 5-pack). Replaces the per-descriptor `toClipboardHast` contract that earlier iterations of this spec had locked.

The full paste matrix collapses cleanly: OK→OK round-trips byte-for-byte via text/plain; cross-app outbound renders semantically; cross-machine markdown-text-only transport recovers OK-canonical descriptors via the heuristic. No new MIMEs, no DOM-level event handlers, no marker attributes — the existing dispatcher infrastructure does the work once correctly ordered and the heuristic correctly tuned.

## 2) Goals

- **G1.** OK→OK paste byte-for-byte preserves source identity for all canonical and compat descriptors (Callout, img, video, audio, Accordion + GFMCallout, CommonMarkImage, HtmlDetailsAccordion), modulo NG1-NG11 storage normalizations baked into the unified parse/serialize pipeline (D1).
- **G2.** OK→external paste of any custom component renders semantically in cross-app destinations (Slack, Notion, Gmail, GitHub) — real images, callout-shaped blockquotes, expandable details, etc. Foundational pattern (`toClipboardHast`) extends to every future descriptor without further pipeline work.
- **G3.** External→OK paste of OK-canonical markdown text recovers descriptor identity even when text/plain is the only surviving channel. Cross-machine D4 scenario (raw file via email/Slack/file copy) round-trips losslessly back into OK.
- **G4.** Both WYSIWYG and Source views ship the new behavior symmetrically (D2).
- **G5.** No existing precedent or LOCKED decision is violated. Implementation threads through PM's documented hooks (precedent #19(b)) and FR-20 escape contract.
- **G6.** Promotion notice on `2026-04-16-clipboard-mdast-canonical/SPEC.md` NG1 per the post-ship corrigendum-annotation pattern (CLAUDE.md "Post-ship corrigendum annotations").

## 3) Non-goals

- **[NEVER]** NG-S1: Custom MIME emission (sync-event `vnd.open-knowledge/slice` or Chromium-pickling `web text/x-ok-slice`). Verified contradiction with precedent #19(b); FR-13-first removes the need entirely. Reverting requires reopening the predecessor's D14 with new evidence.
- **[NEVER]** NG-S2: `data-ok-slice` attribute on text/html as a discriminator. Same goal achieved by FR-13-first heuristic; attribute injection adds copy-side overhead without functional benefit. (Considered and rejected after analysis at `evidence/structural-payload-mechanism.md` and the heuristic-coverage verification at `reports/tiptap-clipboard-round-trip-markdown/REPORT.md` lines 1242-1363.)
- **[NEVER]** NG-S3: `text/markdown` or `text/md` MIME emission. Verified 2026-04-30 against IANA registry, W3C Clipboard API §6.4, WebKit allowlist source, Chromium pickling docs, Firefox Bugzilla. Safari rejects, no destination reads, every peer markdown-canonical editor relies on `text/plain` + `text/html` only.
- **[NEVER]** NG-S4: DOM-level `handleDOMEvents.copy/cut/dragstart` on WYSIWYG. Per precedent #19(b); BlockNote's verbatim mechanism is prohibited. We adopt the contract shape (`toClipboardHast`), not the mechanism.
- **[NOT NOW]** NG-S5: Per-descriptor `fromClipboardHast` (inbound symmetric pattern). The OK→OK round-trip via FR-13-first uses text/plain markdown for inbound, so a parallel inbound contract isn't needed today. — Revisit if: a destination strips text/plain and we need to recover from text/html bytes alone (no current scenario).
- **[NOT NOW]** NG-S6: Round-trip identity preservation through destinations that strip OK metadata (Slack→OK, Gmail→OK, Notion→OK round-trips on JSX components). Destination-stripping is a third-party-quality issue; OK's text/html shape carries semantic content optimized for cross-app rendering, not for round-trip back to OK. The text/plain channel survives some of these (Slack/Gmail preserve text/plain on outbound); when it doesn't, we accept the descriptor → markdown-shape degradation. — Revisit if: a critical workflow demands lossless OK→external→OK and the text/plain path can't carry it.
- **[NOT NOW]** NG-S7: Iframe canonical descriptor for the paste pipeline. Separate report-driven spec planned (`reports/cb-v2-iframe-embed-pattern/`); paste pipeline gains the descriptor automatically once it lands.
- **[NOT NOW]** NG-S8: predecessor `2026-04-16-clipboard-mdast-canonical/SPEC.md` NG3 — CKEditor-grade Word list reconstruction. — Revisit if: Word becomes a priority paste source.
- **[NOT NOW]** NG-S9: predecessor `2026-04-16-clipboard-mdast-canonical/SPEC.md` NG4 — binary image paste, drag-drop image binary, RTF sibling-data extraction. PR #270 in flight on adjacent asset/embed surface.
- **[NOT NOW]** NG-S10: predecessor `2026-04-16-clipboard-mdast-canonical/SPEC.md` NG9 — complex table features (colspan/rowspan/Google Sheets data attrs). — Revisit if: table paste fidelity reported as priority gap.
- **[NEVER]** NG-S11: NG7 — paste-time DOMPurify / storage-layer sanitization. XSS is a render-layer concern (storage-fidelity contract).
- **[NOT UNLESS]** NG-S12: NG8 — CM6 `lastLinewiseCopy` smart-linewise-paste preservation. — Only if: Source-view users report material workflow loss.
- **[NEVER]** NG-S13: Forking the unified ecosystem to break NG1-NG11 storage normalizations. Permanent maintenance debt; out of scope for this spec; Future Work / Identified for the long-term ceiling.

## 4) Personas / consumers

### P1: OK author (human knowledge worker)

Single primary persona. Per `evidence/_user_outcomes.md`.

- **JTBD:** When I'm authoring across multiple OK docs, between OK and external apps, and across machines via raw markdown transport (email/Slack/file)... But OK→OK paste silently flips JSX descriptors to markdown form, OK→external paste of custom components yields escaped source blocks, and external→OK paste of OK-canonical markdown bytes doesn't recognize JSX shape... Help me preserve the source bytes of every paste interaction... So I can move OK content fluidly across the paste matrix without silent identity loss.
- **Current workflow + workarounds:** Today the user manually retypes JSX components after OK→OK paste, or copies as Source view text only (which does round-trip but loses the WYSIWYG editing flow). For cross-app paste of `<Callout>`, the user accepts the escaped-code-block degradation in destination apps.
- **Pain points:** Silent flipping of `<img/>` to `![](src)` is the worst case — visually identical render hides the descriptor identity loss until the source bytes are inspected. The cross-machine (email/Slack file transport) failure is an authoring blocker for distributed teams sharing OK content as raw markdown.
- **Trust/security sensitivities:** Storage-layer fidelity contract (CLAUDE.md): clipboard-side changes must not introduce new sanitization paths or compromise FR-20 escape contract. URL-scheme sanitization stays on outbound only.
- **Success in their terms:** "I can copy any block of OK content, paste it anywhere, and the bytes round-trip correctly when I paste back into OK. Cross-app destinations show something semantically right (a real image, a styled callout) without leaking OK metadata."

### Secondary destinations (data targets, not user personas)

External destination apps consume OK clipboard payload. From `reports/tiptap-clipboard-round-trip-markdown/REPORT.md` — 11+ surveyed destinations: Gmail web, GitHub textarea, Slack compose, VS Code `.md`, Notion, Google Docs, Discord, Apple Notes, TextEdit, Linear, Outline. Each has different MIME-priority + sanitization posture; the `text/plain` markdown + `text/html` rendered combo covers all surveyed cases.

### Secondary sources (data origins)

Same 11+ as paste sources. Plus AI chat surfaces (Claude / ChatGPT / Gemini) — emit markdown via copy-button (markdown-canonical text/plain), or rendered text via select-and-copy (degraded text/plain).

### AI agents — out of clipboard path

Agents write directly via MCP tools (`write_document`, `edit_document`); never via clipboard. Agent-produced content moves through clipboard only when humans copy from a chat surface and paste into OK. Agent-write attribution via `extractAgentIdentity` at handler entry — separate path from `principal-<UUID>` paste origin.

## 5) User journeys

### P1: OK author — 4 paste journeys

**J1 — OK→OK same-machine.** User copies a block in one OK doc, pastes into another OK doc (same tab or different tab). Most common journey. Currently fails on JSX descriptors per the regression. Spec target: byte-perfect round-trip via FR-13-first → text/plain canonical markdown path.

**J2 — OK→external.** User copies OK content, pastes into Slack/Notion/Gmail/GitHub/VS Code/AI chat. Cross-app rendering goal: real images, callout-shaped blocks, expandable details. Currently media descriptors render correctly via Option B; capitalized JSX shows as escaped MDX source. Spec target: every canonical descriptor renders semantically via `toClipboardHast`.

**J3 — external→OK where source produced OK-canonical bytes.** User copies markdown text from VS Code editing a `.md`, GitHub textarea, AI chat copy-button, another markdown-canonical editor (Linear, Outline, BlockNote, Milkdown), or a raw markdown file received via email/Slack. text/plain has canonical OK markdown. Spec target: FR-13-first + extended is-markdown.ts catches OK shapes; mdManager.parse reconstructs descriptors.

**J4 — external→OK where source produced rich HTML.** User copies from Notion, Google Docs, Gmail rich, Slack rich, Apple Notes. text/plain is degraded; text/html has structural content. Spec target (mostly existing): Branch D's `htmlToMdast` pipeline + 9 vendor cleanup plugins handles these. No changes needed to this journey.

## 9) Proposed solution (vertical slice)

### User experience / surfaces

- **Dashboard/admin UI:** No UI changes. All clipboard work is invisible to users except as restored byte-preservation behavior.
- **API endpoints:** None.
- **SDK:** None directly. Internal: `JsxComponentMetaBase.toClipboardHast?: (node, ctx) => HastNodes` added to descriptor registry types.
- **CLI:** None.
- **Docs/onboarding:** Update `packages/core/src/registry/README.md` (or equivalent) with `toClipboardHast` contract example. Update CLAUDE.md "Markdown pipeline" to reference the new contract once locked.
- **Error messages:** Existing structured telemetry covers new paths.
- **Billing/limits:** N/A.

### System design

_Detailed design comes in iterate phase. Skeleton below._

- **Architecture overview:** Three independent additive layers on the existing pipeline:
  1. Dispatcher reorder (WYSIWYG + Source): move FR-13 ahead of Branch C.
  2. Heuristic extension: 5 new signals in `is-markdown.ts`.
  3. `toClipboardHast` contract: per-descriptor sister method to `serialize`, dispatched at `mdast-to-hast-handlers.ts:mdxJsxFlowHandler/mdxJsxTextHandler` ahead of `tryNativeHtmlPrimitive`.
- **Data model:** No PM schema changes; no Y.Doc changes. Descriptor registry types extend by one optional method.
- **API/transport:** No changes.
- **Auth/permissions:** N/A (clipboard is user-origin).
- **Enforcement point(s):** Dispatcher in `handle-paste.ts` + `source-clipboard.ts`; heuristic in `is-markdown.ts`; descriptor dispatch in `mdast-to-hast-handlers.ts`.
- **Observability:** Existing telemetry + per-descriptor `toClipboardHast` invocation event.

### Alternatives considered

- **Option A: Sync-event custom MIME (`vnd.open-knowledge/slice`).** Rejected — contradicts precedent #19(b). Required DOM-level `handleDOMEvents.copy` to write; would break the drag-and-drop coupling preserved by D14. (`evidence/structural-payload-mechanism.md`.)
- **Option B' (`data-ok-slice` attribute on text/html).** Rejected — solves the same problem as FR-13-first with strictly more implementation surface (copy-side attribute injection + new dispatcher branch). Heuristic + dispatcher reorder achieves the same outcome with less code. (`evidence/structural-payload-mechanism.md`.)
- **Try-parse-and-validate** (Milkdown / Keystatic / BlockSuite pattern) instead of signal-count heuristic. Rejected for v1 — same false-positive shape as signal-count (ambiguous bytes resolve identically); 50-150x slower per paste; no functional gain. Listed as alternative architecture if signal-count drifts in future. (`reports/tiptap-clipboard-round-trip-markdown/REPORT.md` §2026-04-30 markdown-detection heuristic survey.)
- **Adopt BlockNote's `toClipboardHast` mechanism verbatim** (DOM-level `handleDOMEvents.copy`). Rejected — mechanism violates precedent #19(b). Adopt contract shape only; mechanism threads through existing `clipboardSerializer.serializeFragment`.
- **Per-descriptor `fromClipboardHast` inbound contract** (symmetric to `toClipboardHast`). Deferred — FR-13-first routes OK→OK through text/plain markdown which doesn't need an inbound HAST contract. NG-S5 [NOT NOW].
