# Install UX Docs Reconciliation — Design Challenge

**Spec under review:** [[specs/2026-04-20-install-ux-docs-reconciliation/SPEC]]
**Baseline commit:** `3f069185`
**Reviewer:** design-challenge subagent (cold reader)
**Date:** 2026-04-20

Ten challenges below follow a uniform shape: **decision → alternative → why the rejection may not hold → what would change my mind.** A summary at the end grades each challenge as STRONG (worth reconsidering), WEAK (rejection holds), or REVEALS A TRADE-OFF the spec didn't explicitly name.

Challenges are pressure-tests, not claims of "this is wrong." Several resolve to "rejection holds" — stated clearly when they do.

## Challenge 1 — D3: Is `bunx` primary the right first-impression for docs.inkeep.com?

| Field                              | Content                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Decision challenged**            | D3 LOCKED — "Primary runner stays `bunx` (first) with `npx`/`pnpm dlx` as co-equal alternates." Rationale: root README already ships `bunx` primary; making docs match preserves shipped precedent; the repo is itself a Bun monorepo.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **Alternative I'd argue for**      | `npx` primary on the docs site (and possibly flip the root README to match). Rationale: docs.inkeep.com is discovered later in the funnel than the repo (SEO, blog posts, third-party references); its audience skews toward the Node/npm ecosystem median developer who has not yet adopted Bun. The docs site's first-impression should align with the ecosystem-default toolchain.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| **Why the rejection may not hold** | Three gaps in the D3 rationale: (1) **"Shipped precedent" is a sunk cost**, not a forcing function. The parent spec §8's description of the README is a snapshot of current state, not a constraint on docs-site design. D3 cites parent §8 but §8 only describes what the README already says — it does not argue that the docs site must match. (2) **Audience skew is unaddressed.** The spec asserts P1 (GitHub-landing) and P2 (docs-landing) personas exist co-equally, but provides no evidence that their runtime distribution is identical. A reasonable prior is that docs-site visitors (arriving via Google, blog posts, or direct link from a Node-ecosystem project) are more Node-biased than repo visitors (arriving via word-of-mouth from Bun-adjacent users). (3) **The parent spec's own P1 persona is a "future contributor," not an end user** (verified — parent §4 P1 says "contributor touching CLI packaging"). This child spec introduced end-user personas; `bunx`-primary inherits from contributor-facing choices without re-validating against end-user evidence. |
| **What would change my mind**      | (a) Evidence that docs-site referrers skew Bun-heavy (e.g., analytics showing traffic from bun.sh, Bun Discord, Bun blog) — would confirm audience continuity. (b) A stated product hypothesis that Open Knowledge is intentionally targeting Bun-first developers as the V0 beachhead, making `bunx`-primary a deliberate positioning signal. (c) Demonstration that under D4's `<Tabs groupId="bin">` pattern, the docs site doesn't actually show a single "primary" — the user picks via tabs — in which case D3's "primary" choice only affects the first tab position, which is a weaker decision.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| **Verdict**                        | **REVEALS A TRADE-OFF.** The rejection reasoning ("preserves shipped precedent") is thin, but the decision is arguably reversible (one-line edit under D6's G6 mechanism) and the 90-day metric (§7 M2) provides a natural revisit trigger. Worth the spec surfacing the audience-skew assumption explicitly rather than resolving it by precedent.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |

## Challenge 2 — D4: Do `<Tabs>` actually propagate `ok` awareness, or hide it behind a click?

| Field                              | Content                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Decision challenged**            | D4 LOCKED — use Fumadocs `<Tabs groupId="bin">` in docs-site code blocks to present `open-knowledge` and `ok` as selectable alternatives. Rejects "double-show" (both forms in one block) as visually noisy across 40+ examples.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| **Alternative I'd argue for**      | Double-show via a comment annotation: render every code block as `bunx @inkeep/open-knowledge start   # or: ok start` (or an equivalent two-line pattern). The alias is visible to every reader, every block, without interaction.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **Why the rejection may not hold** | **Tabs hide half the product surface from readers who never click.** Fumadocs `<Tabs>` default to the first tab on initial render; `groupId` persists selection via localStorage across pages within a session but not across sessions. A reader visiting docs for the first time sees only `open-knowledge` (or only `ok`, depending on tab order) and never learns the other exists unless they (a) scroll, (b) notice the tab affordance, (c) click it. The D4 rationale frames this as "user-selectable alternatives" — but selection requires awareness of the alternative, which is the thing Tabs fail to propagate. **Contrast with the parent spec D1 implication verbatim** ("docs must show both forms in every install snippet"). Strictly read, Tabs show one form at a time per snippet — they satisfy D1 only under a charitable reading where "show" includes "offer via tab." The strict reading prefers double-show. **Additional cost of Tabs:** on mobile (where Fumadocs Tabs become a horizontal scroll), tab affordance discoverability is further reduced. Mobile is likely a non-trivial fraction of docs-site traffic post-launch. |
| **What would change my mind**      | (a) Evidence from Fumadocs-using projects (e.g., OpenAI Agents SDK docs, T3 docs) showing that `groupId` session-persistence + first-tab-label naming actually makes both forms visible enough in practice. (b) A UX study or even informal user test where docs readers are asked "what's the short form of `open-knowledge`?" after reading a Tabbed page. (c) A compromise position: keep Tabs as the primary, AND add a one-time banner/callout at the top of the getting-started page explicitly stating both bins exist — so tab-non-clickers at least learn about `ok` from the banner.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| **Verdict**                        | **STRONG.** The D4 rejection of double-show frames the trade-off as aesthetics ("visually noisy") vs. selection ergonomics, but the real trade-off is aesthetics vs. discoverability. Discoverability is load-bearing for the D1 implication that motivated D4 in the first place. Worth reopening — the compromise (Tabs + one top-of-page callout) is cheap and closes the gap.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |

## Challenge 3 — D5: Does a prereq warning actually close G3, or defer the failure?

| Field                              | Content                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Decision challenged**            | D5 LOCKED — keep `npx` hardcoded in generated MCP configs; close G3 via (a) Node-as-MCP-prereq note and (b) documented rationale. Option 2 (runner detection) deferred to ≥3 support tickets in 30 days.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| **Alternative I'd argue for**      | Option 2 (runner detection via `npm_config_user_agent`) ships with D5, not as a follow-up. D5's own evidence file ([[specs/2026-04-20-install-ux-docs-reconciliation/evidence/d5-runner-choice-analysis]]) empirically verified the detection works across all three runners and documented that husky/playwright-core/esbuild already ship this pattern in production for similar purposes. Shipping Option 2 now structurally eliminates the pure-Bun ENOENT failure instead of documenting it.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| **Why the rejection may not hold** | The spec's **G3 wording** is "silent-failure path closed." A prereq warning converts silent → documented, but does not close. The evidence file's own analysis admits this: *"Failure mode persists — users who skip the prereq section (a measurable fraction of developers) still hit `spawn npx ENOENT` at editor startup. The failure is now 'documented-but-hit' vs 'silent-and-hit,' which is an improvement but not a fix."* This is a direct concession that Option 1 does not satisfy G3's literal wording. The D5 rationale leans on three pillars, each challengeable: (1) **"Mastra does it this way"** — Mastra is cited as the install-UX north star, but the parent spec's own §D7 reference observes Mastra has "equivalent exposure to the silent-ENOENT failure." Inheriting a known failure mode from a reference app because it's the reference app is circular. (2) **"\~20 tests to update"** — the evidence file quantifies the Option 2 cost as \~150 lines of diff + \~40 test edits + 3-6 new tests. That is a moderate cost, not a blocker; the test count is dominated by parameterizing existing assertions. (3) **"Escalation triggers require post-launch evidence"** — but escalation triggers are only useful if the product has a feedback loop. Pre-launch, tickets are zero by definition; the trigger cannot fire before users hit the failure and report it. This structurally subordinates the silent-correctness goal (G3) to the ecosystem-parity goal. |
| **What would change my mind**      | (a) Evidence that the prereq warning is effective — e.g., user-study data showing readers-of-getting-started actually install Node before hitting MCP startup. (b) A product-owner statement that the V0 beachhead cohort is Node-majority (making ENOENT a small-residual failure mode rather than a common one) — in which case Option 1's cost-benefit improves. (c) A commitment to monitor for the ENOENT string in GitHub issues / Slack within the first 14 days (not 30) post-launch, so the escalation trigger actually fires fast enough to matter. (d) Evidence that Option 2 has a material "user switches runtime later" edge case in practice — the evidence file flags this but provides no empirical base rate.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| **Verdict**                        | **STRONG.** The spec explicitly names G3 as a co-equal driver (D2) but D5 subordinates G3 to D5's own ecosystem-parity goal. The evidence file is unusually candid about this (*"only partially satisfied — upgraded from silent to documented-but-still-hit"*) and the candor is load-bearing. If the team reads the spec's §7 Metric 2 as the fallback safety net, they should acknowledge that "0 tickets in 90 days" is measuring whether users are silent about their pain, not whether the pain exists. Worth a product-level re-examination of whether G3's "closed" really means "closed" or "mostly closed."                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |

## Challenge 4 — D6: Three surfaces vs. one linter

| Field                              | Content                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ---------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Decision challenged**            | D6 LOCKED — canonical source at `docs/content/_partials/install-path.mdx`; docs consume via Fumadocs `<include>`; README + plugin README sync via `scripts/sync-install-matrix.ts` with marker comments; AGENTS.md consumes a committed `install-path.generated.ts`; CI runs the script + `git diff --exit-code`. Rationale: "minimum viable G6 mechanism."                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| **Alternative I'd argue for**      | Ship ONE mechanism: a CI lint that greps each of the 6 surfaces for install-path blocks and asserts matrix-content equivalence. No canonical file, no sync script, no generated.ts module. When the matrix changes, the contributor edits all 6 surfaces by hand; CI catches any they forgot. "Canonical" lives in CI's equivalence check, not in a source file.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| **Why the rejection may not hold** | The D6 rationale for the three-mechanism design is implicit: Fumadocs `<include>` handles S2-S4 elegantly → marker-sync handles the plain-markdown surfaces → generated.ts handles the tsdown-bundled surface. Each mechanism is the right local answer. But **"the right local answer for each surface" is not the same as "the minimum viable overall solution."** Three mechanisms mean: (a) three places a future contributor can get it wrong; (b) three things that can break in a dependency bump (Fumadocs upgrade, tsdown upgrade, marker syntax collision); (c) contributor cognitive load — a new contributor editing install matrix phrasing must figure out which mechanism applies to their surface. A pure CI lint approach has a much smaller surface: one grep-and-compare script. Downside is contributor ergonomics — editing 6 files by hand is annoying — but with only \~5 expected matrix changes over the product lifetime (Homebrew Cask row, potential `pnpm dlx` removal, minor phrasing tweaks), hand-editing is \~5 × 6 = 30 edits total. The three-mechanism sync investment amortizes over 30 edits only if the engineering cost of the three mechanisms is under \~30 minutes per hand-edit, which is tight. |
| **What would change my mind**      | (a) A commitment signal that matrix changes will be frequent (e.g., we plan to experiment with runner ordering and rollout variants) — frequent change amortizes the sync investment. (b) Evidence that the 6 surfaces drift by more than phrasing — if they diverge in semantic content (e.g., one surface documents a prereq the other doesn't), a lint-only approach would fail to enforce consistency at the content level. (c) Demonstration that the lint-only approach can be written in under \~50 lines of grep + bash; if it's comparable in LOC to the sync script, the sync wins on ergonomics.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| **Verdict**                        | **REVEALS A TRADE-OFF.** "Minimum viable" is a misleading frame in D6 — three mechanisms is the minimum to make each surface feel natural to edit, but not the minimum to solve G6. Drift-check-only is genuinely simpler and might be sufficient. The spec should acknowledge that D6 optimizes for contributor ergonomics on the edit path and pays complexity in setup + dependency surface. Worth explicitly naming this choice rather than hiding it behind "minimum viable."                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |

## Challenge 5 — D7: Prose/code vocabulary split forces every reader to translate

| Field                              | Content                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ---------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Decision challenged**            | D7 LOCKED — prose uses `ok <cmd>` (short form); code blocks use the full matrix (Tabs-wrapped). Rationale: "prose is comprehension; code blocks are copy-paste."                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| **Alternative I'd argue for**      | Use the long form (`open-knowledge <cmd>`) consistently in both prose and code. Verbose but zero translation cost. Reader reading prose `open-knowledge start` can copy-paste directly into a terminal (post-global-install); reader copying a code block sees the same long form. The `ok` alias is documented once, visibly, in the install-path matrix, and mentioned in prose only when aliasing itself is the subject.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| **Why the rejection may not hold** | D7 inherits an implicit assumption: that prose readers and code-block users are different cohorts, or at least different modes of the same reader. In practice, the same reader alternates between prose ("Next, start the collab server with `ok start`") and copy-pasteable commands (fenced block with the full matrix) within one page. The reader's mental model must reconcile the two: "when I see `ok start` in prose, what do I actually type into my terminal?" If they haven't globally installed, `ok start` fails — the prose is misleading for the one-shot cohort. The D7 rationale acknowledges this indirectly ("copy-paste works without global-install assumption" for code blocks) but doesn't extend the logic to prose, which has the same assumption problem. **The stronger alternative:** pick the form that works universally (`bunx @inkeep/open-knowledge start` works for one-shot AND globally-installed users) and use it everywhere — but that's verbose, which D4 already rejected. The convention D7 picks is a compromise that works only if readers mentally maintain two vocabularies. |
| **What would change my mind**      | (a) Evidence that prose readers rarely attempt to copy-paste from prose (i.e., they only copy from fenced blocks) — would confirm the split is safe. (b) A docs-site interaction pattern like "click this word to see the full form" that makes the translation automatic. (c) A third convention I haven't considered: use `ok <cmd>` in prose but include a prominent one-time callout on every `guides/*.mdx` page — "Prose on this page uses `ok` (short alias). If you invoked via bunx/npx/pnpm dlx without global install, replace `ok` with `bunx @inkeep/open-knowledge` etc." This wouldn't satisfy reader ergonomics but would at least close the translation-ambiguity gap.                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| **Verdict**                        | **REVEALS A TRADE-OFF** tending STRONG. The prose/code split is defensible but not cost-free; the spec frames it as a correctness choice ("comprehension vs. copy-paste") when it's actually a translation-cost choice. Worth acknowledging explicitly that D7 asks readers to maintain two mental models, and validating that the trade is accepted (or picking a less-split alternative).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |

## Challenge 6 — D8: Two-level prereq or single clean prereq?

| Field                              | Content                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Decision challenged**            | D8 LOCKED — two-level prereq. Level 1: "Any JS runner — Bun ≥ 1.3.11, Node ≥ 22, or pnpm ≥ 10." Level 2 (MCP caveat): "If you use `ok init` to register MCP, you'll also need Node ≥ 22."                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| **Alternative I'd argue for**      | Single-level prereq: "Requires Node.js ≥ 22. Bun users can use `bunx` as a runner, but Node must be installed separately for MCP spawning." Collapse the two levels into one. This costs clarity around "do I need Node if I'm not doing MCP?" but gains simplicity for the P1/P2 first-touch audience who will do MCP (the product's primary value prop is MCP integration).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| **Why the rejection may not hold** | The two-level prereq assumes users segment into "CLI-only users" (Level 1 suffices) and "MCP-also users" (Level 2 applies). But **the entire install flow funnels through `ok init`, which writes MCP configs.** The spec's own §5 happy path says "User runs the command; `ok init` scaffolds; MCP config written." There is no well-documented non-MCP install path — the product's purpose IS MCP integration. A Level-1 user who doesn't install Node gets an `ok init` that writes configs they'll hit ENOENT on the moment they open their editor. Two-level prereq is technically precise but **psychologically misleading** — it implies "you might not need Node" when in practice you almost certainly will. D8's rationale cites D5 — but D5's Option 1 already chose to keep `npx` hardcoded, which universalizes the Node requirement. The two-level phrasing preserves an abstraction that matters for a minority of users (dev-ing on the CLI without MCP) at the cost of clarity for the majority. |
| **What would change my mind**      | (a) Evidence that a material fraction of `ok init` users skip MCP integration (e.g., they run `ok start` without editor integration). Would justify preserving Level 1 as a real use case. (b) A progressive-disclosure design where the getting-started page leads with Level 1, and the MCP prereq appears only at the MCP-configuration step — so the user hits Level 2 just-in-time. (c) A product pivot where MCP is truly optional (e.g., docs-author standalone use case) — would justify Level 1.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| **Verdict**                        | **WEAK tending REVEALS A TRADE-OFF.** The two-level prereq is pedagogically cleaner than it first appears — it lets Level 1 serve readers who aren't sure yet whether they'll want MCP, and escalates to Level 2 at the right moment. But the spec doesn't document the rationale for preserving the two-level abstraction beyond "avoids forcing Node AND Bun AND pnpm on every reader." The single-level alternative is more honest to the actual user journey. Worth the spec briefly articulating whether V0 serves a genuine non-MCP use case.                                                                                                                                                                                                                                                                                                                                                                                                                                                                |

## Challenge 7 — D2 scope: D5 subordinates G3 to ecosystem parity

| Field                                   | Content                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| --------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Decision challenged**                 | D2 LOCKED — "Treat V0-launch polish + D1 compliance + silent-correctness as co-equal drivers." Spec claims co-equality; I argue D5 empirically subordinates silent-correctness (G3) to ecosystem parity (aligned with V0-launch polish framing).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| **Alternative I'd argue for**           | Acknowledge that D5 makes G3 subordinate, and update D2's framing to: "V0-launch polish + D1 compliance are primary drivers; silent-correctness is a secondary driver satisfiable via documentation pending escalation." OR flip it: if G3 is truly co-equal, D5 must ship Option 2 now (see Challenge 3).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| **Why the co-equal claim doesn't hold** | The spec asserts three co-equal drivers at D2, then in D5 resolves the tension between them by picking the one path (Option 1) that technically satisfies G3 only via documentation — which the evidence file (in its own words) characterizes as "only partially satisfied" upgraded from "silent-and-hit" to "documented-but-still-hit." If silent-correctness is co-equal with V0 polish, then the G3-satisfying choice would be Option 2 (which structurally closes the failure). Picking Option 1 privileges ecosystem parity (V0 polish, Mastra-alignment) over silent-correctness. This is a legitimate product choice but inconsistent with the spec's stated co-equality. **Consequence for future readers of the spec:** if they try to derive implementation decisions from D2, they'll get contradictory guidance (D2 says all three are equal; D5 says one is more equal). |
| **What would change my mind**           | (a) Reframing D2 to name the ordering explicitly (e.g., "V0 polish = D1 compliance > silent-correctness with documented escalation"). Would keep the design intact while being honest about priorities. (b) A stronger argument that Option 1 DOES structurally close G3 — perhaps via a stricter prereq-check in `ok init` itself that probes for `which npx` and refuses to write MCP configs if absent. That would convert the failure from runtime-surface to install-time surface and genuinely close the silent path.                                                                                                                                                                                                                                                                                                                                                             |
| **Verdict**                             | **STRONG — REVEALS A TRADE-OFF.** The spec's D2 co-equality claim is aspirational, not empirical. The LOCKED design in D5 is fine on its own merits, but the co-equality framing obscures the actual priority ordering. Worth honest reframing.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |

## Challenge 8 — §15 Phasing: Why validate G6 on the lowest-traffic surface?

| Field                              | Content                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| ---------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Decision challenged**            | §15 Phase 1 — "Create canonical source + root README. This is a single PR that validates G6 with the smallest user-facing change." Phases 2-4 migrate docs site, secondary surfaces, and enforcement respectively.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| **Alternative I'd argue for**      | Phase 1 should validate G6 on the HIGHEST-volume consumer surface — the docs site — because that's where brittleness under real editing pressure surfaces fastest. Alternatively, collapse Phases 1-4 into one or two PRs since the validation cost dominates and incremental shipping doesn't reduce risk here.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| **Why the rejection may not hold** | The Phase 1 rationale ("smallest user-facing change") conflates user-impact minimization with validation-signal maximization. For a G6 mechanism whose risk is "does the sync script hold up under real editing pressure," the strongest validation is real editing pressure — which lives in the docs site (40+ code blocks across 15 files), not the root README. Phase 1 as designed tests the sync script against the simplest possible case (README has one marker block) and then commits to the mechanism; phases 2-4 discover problems after the commitment. **Alternative phasing:** Phase 1 = Fumadocs `<include>` on the docs site only (no sync script yet, no README touch) — validates the MDX partial works under real editing density. Phase 2 = sync script for the non-MDX surfaces. Phase 3 = enforcement. Shorter critical path, stronger early signal. **Or even stronger:** is there a reason Phases 1-4 aren't one PR? The total LOC change is \~200 lines of script + \~50 edits across surfaces. Split-by-PR adds review overhead and delays the G6 benefit (the matrix can't be updated until all phases land). |
| **What would change my mind**      | (a) A commitment criterion for Phase 1 → Phase 2 promotion that actually stress-tests the mechanism (e.g., "at least one matrix change lands in Phase 1 without hand-editing"). (b) A rollback plan if Phase 2 reveals the sync script doesn't scale — does Phase 1 revert cleanly? If yes, phasing is low-risk. (c) Evidence that collapsing into one PR would produce a review-unfriendly diff. If the diff is mostly mechanical (reformatting existing content to use Tabs + include), one PR is probably fine.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| **Verdict**                        | **REVEALS A TRADE-OFF.** The phasing as designed optimizes for small blast radius per PR (good for review) but under-optimizes for G6 validation signal. A user should be willing to either (a) commit to stronger Phase 1 criteria, or (b) collapse phases and accept a larger PR. The spec doesn't surface this trade.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |

## Challenge 9 — D5 escalation trigger: Pre-launch, the metric is structurally unobservable

| Field                              | Content                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| ---------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Decision challenged**            | D5 + §14 Future Work — "Activate Option 2 if ≥3 `spawn npx ENOENT` support tickets in rolling 30-day window post-launch."                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| **Alternative I'd argue for**      | Either (a) drop the support-ticket metric and use a different, observable signal (e.g., telemetry event — which the spec rules out via parent D14 LOCKED), or (b) ship Option 2 pre-emptively since the trigger cannot fire in a pre-launch window. Absent a working observability mechanism, D5's escalation clause is rhetorical, not operational.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| **Why the rejection may not hold** | The support-ticket metric has two structural problems: (1) **Pre-launch, tickets are zero by definition.** The metric measures a post-launch signal. During the V0 window, there's no basis for the escalation to fire. The spec treats this as a future-state trigger, but doesn't name the bridge from pre-launch (no data) to post-launch (first user hits ENOENT and must report it). (2) **Self-report bias.** Users who hit ENOENT and conclude "this tool doesn't work on my machine" are likely to abandon and NOT file a ticket. The ones who file tickets are the persistent, tool-invested users — a biased sample. A true rate of 10% silent ENOENT in the first 30 days could produce 0 tickets if the abandonment rate is high. The metric under-detects the failure it's designed to escalate on. (3) **Alternative observability:** the spec rules out telemetry (parent D14), which means there is no non-self-report signal for ENOENT. D5's escalation clause is therefore unobservable in practice — it requires users to persist past the failure and navigate to an issue tracker. |
| **What would change my mind**      | (a) A stated operational plan for the escalation trigger: who monitors, what tools, what cadence. If "Andrew greps GitHub issues once a week for 'npx ENOENT'," the trigger is plausibly operational for a small user base. (b) A lower threshold (e.g., ≥1 ticket) recognizing self-report bias. (c) A complementary signal like "announce this spec in release notes with an explicit call for bug reports on MCP startup failures" — raises the self-report rate. (d) Shipping Option 2 as a dark-launch (code lands, feature-flagged off) so it can activate on first ticket without a new PR.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| **Verdict**                        | **STRONG.** The escalation clause is load-bearing for D5's rationale — without it, D5 Option 1 would be unambiguously worse than Option 2 on the silent-correctness axis. But the trigger-observability gap makes the clause a rhetorical commitment, not an operational one. If the team accepts the status quo, they should acknowledge that G3 is "mostly closed via docs, with escalation deferred indefinitely." If they want the trigger to actually fire, they need an observability plan.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |

## Summary

### STRONG challenges (worth reconsidering before spec is final)

- **Challenge 2 (D4 — Tabs hide `ok` from non-clickers).** The D1 implication "docs must show both forms" reads strictly as "both visible," and Tabs don't deliver that on initial render. The compromise (Tabs + top-of-page callout) is cheap. Either strengthen D4 to include the callout, or revise D4 to double-show.
- **Challenge 3 (D5 — prereq warning ≠ closed G3).** The evidence file itself admits Option 1 does not close G3 in the literal sense. The spec's co-equality claim (D2) and G3's wording ("silent-failure path closed") are in tension with D5's chosen resolution. Either ship Option 2 now, or honestly re-frame G3 as "silent → documented."
- **Challenge 7 (D2 — co-equality is aspirational).** D5 empirically prioritizes ecosystem parity over silent-correctness. D2's co-equal framing obscures this. Honest reframing would serve future readers better.
- **Challenge 9 (D5 escalation trigger is unobservable).** Support-ticket threshold requires users to persist past the failure and self-report. Biased sample, zero pre-launch baseline. Either instrument observability differently, lower the threshold, or acknowledge the trigger is rhetorical.

### REVEALS A TRADE-OFF (spec should explicitly acknowledge)

- **Challenge 1 (D3 — `bunx`-primary audience-skew assumption).** The decision is arguably reversible, but the spec should explicitly say "we're betting V0 docs-site visitors are Bun-adjacent" rather than resolving by precedent.
- **Challenge 4 (D6 — three mechanisms is not minimum-viable).** "Minimum viable" misleadingly frames the choice. D6 optimizes for edit-path ergonomics, pays complexity in setup. Should be named as such.
- **Challenge 5 (D7 — prose/code split asks readers to translate).** The split is defensible but has a translation cost the spec doesn't surface. Should be acknowledged.
- **Challenge 8 (§15 phasing — G6 validated on lowest-pressure surface).** Phasing minimizes user-impact blast radius but under-optimizes validation signal. Alternative phasing (docs site first) or one-PR collapse both defensible; spec should explain why it picked small-first.

### WEAK (rejection holds)

- **Challenge 6 (D8 two-level prereq).** Two-level prereq is pedagogically reasonable; single-level is more honest but loses nuance. The spec could add one sentence explaining why Level 1 exists, but the LOCKED choice is fine.
- **Challenge 10 (G6 scaling to N surfaces).** The mechanism is adequate for the current count and one or two additions. Worth a runbook but not a blocker.

### Overall

The spec's **factual coverage and evidence depth** are strong (the D5 and D6 evidence files are unusually thorough). The weaknesses are at the **framing and priority-ordering layer** — D2's co-equality claim doesn't survive D5's resolution; G3's "closed" is weaker than its wording suggests; D4's Tabs satisfy D1 only under a charitable reading. These are framing debts, not design debts. A spec-review pass could tighten the framing without re-opening any LOCKED decision. If framing stays as-is, future readers will spend time re-deriving which priorities are actually load-bearing, which is the exact cost-of-drift the spec is designed to prevent on the surface level.

**Recommended action:** Address STRONG challenges 2, 3, 7, 9 before marking Status: Approved. Acknowledge REVEALS A TRADE-OFF challenges 1, 4, 5, 8 as explicit decisions in the prose (one or two sentences each). WEAK challenges 6, 10 do not require action.
