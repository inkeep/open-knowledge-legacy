- **2026-04-11 — /gtm:analyze pass on the 3 remaining escalated design questions.** User directive: "ultra on these. /explore as needed to gather full context." Ran full 5-phase analyze workflow with three parallel /explore subagents (failure-mode analysis of Phases 1-4, OK logging conventions, ServerOptions shape + test override patterns). Key outcomes:
  - **Q1 (per-phase try/catch) → D10 REVISED.** Original recommendation was "wrap Phases 1-4 uniformly." Explore found this was under-specified: Phase 2 (`sessionManager.closeAll()`) already has per-session try/catch at `agent-sessions.ts:168-177` — a destroy()-level wrap would be redundant dead code. The real risk is concentrated in Phase 4 (HIGH-frequency git subprocess failures — timeout on slow NFS, disk full, stale `.git` refs, missing binary) and Phase 3 (closeConnections + timeout + onStoreDocument errors). Phase 1 (~0.1% throw rate) is theoretical but worth wrapping as cheap insurance since a rare Phase 1 throw gates the valuable Phase 3-4 drain. **New D10 scope: wrap Phases 1, 3, 4 individually; leave Phase 2 alone; Phase 5 in outer finally.**
  - **Q2 (observability log) → D14 NEW + OQ-P2-07 PROMOTED.** Original "only observability we'll ever have per NG4" framing was **falsified** by /explore Finding 2: pino is already a dep and actively used across the server package (`getLogger()` at `logger.ts`, `standalone.ts:432`, `persistence.ts`, `api-extension.ts`). The **correct** argument is stronger: `destroy()` is currently the only unlogged lifecycle path in the server package — every other event (shadow repo init, watcher start, file events, document store) uses structured logging. Silent `destroy()` is an outlier, not a policy choice. Promoting to in-scope is **parity with the rest of the package, not scope creep**. Under D10's best-effort drain, the log is ALSO load-bearing — the only signal that distinguishes clean shutdown from partial failure. **New D14: structured shutdown log via `getLogger('server')` at the end of `destroy()`, info on clean exit, warn with phaseErrors on partial failure. Asserted by Test 1 as a behavioral contract.**
  - **Q3 (destroyTimeoutMs as ServerOption) → D11 REVISED.** Original rationale was "test wall-time." Analysis found a stronger rationale: **production safety valve**. A CLI user on slow NFS with legitimate >10s L1 writes gets silently truncated under the hardcoded ceiling. Existing test harness at `test-harness.ts:67-74` already overrides timing fields (`debounce: 200, maxDebounce: 1000`) — `destroyTimeoutMs` fits the same pattern. `ServerOptions` is internal-only (only CLI + test harness import it, confirmed via grep) — adding a 17th field is zero public-API risk. **New D11: default 10_000, configurable via `ServerOptions.destroyTimeoutMs`.**
  - **Cross-question cascade insight.** The three questions are NOT independent — they form a coherent "resilience + transparency" design stance. Q2's value is conditional on Q1 (under fail-fast, the log only fires on happy path — redundant; under best-effort drain, the log is the only partial-failure signal — load-bearing). Q2 and Q3 together enable behavioral-contract testing (test asserts the log was emitted with expected content under controlled timeout). Deciding them à la carte produces incoherent combinations.
