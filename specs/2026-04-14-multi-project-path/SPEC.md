## 1) Problem statement

**Situation.** Open Knowledge ships as a CLI (`@inkeep/open-knowledge`) + MCP server + Hocuspocus CRDT server + React editor. The documented quickstart is: `bunx @inkeep/open-knowledge init && bunx @inkeep/open-knowledge start`. After that, the user is supposed to open Claude Code (or Cursor, etc.) in the project and edit live alongside the agent. For a second project, the same two commands repeat. This is the "local-first, one binary, per-project process" story the project is built around.

**Complication.** An empirical walk of that quickstart against a fresh `/tmp/` project (audited 2026-04-14, see `reports/onboarding-walkthrough-audit/`) finds the happy path broken at multiple steps, and the multi-project story has no surface at all beyond "run two terminals and remember the port numbers." Specifically:

1. **P0: Browser UI silently broken on every npm install.** `tsdown` bundles `start.ts` into `dist/commands/start.mjs`. At runtime, `import.meta.dirname` resolves to `.../dist/commands/`, so the asset lookup checks `dist/commands/public/` instead of `dist/public/`. All three fallback paths miss. The server boots, logs `"No React app assets found — browser UI will not be available"`, and serves 404 for `/`. Users see nothing when they open the URL. (`onboarding-walkthrough-audit` F1.)
2. **P0: Package is not on npm.** README's `bunx @inkeep/open-knowledge` invocations return 404. `.mcp.json` generated by `init` uses `npx @inkeep/open-knowledge mcp` — also 404. Today, only monorepo dev-server + hand-patched `.mcp.json` works. (F2.)
3. **P1: Default `include: **/*.md` indexes node_modules.** A fresh Node project sees `init` report "Found 294 markdown files" — mostly dependency READMEs/CHANGELOGs, because the default exclude list is empty and a freshly `git init`-ed project has no `.gitignore`. (F3.)
4. **P1: Port 3000 collision is an uncaught crash.** `Error: listen EADDRINUSE: address already in use ::1:3000` with stack trace, no fallback, no guidance. Port 3000 is one of the most contested ports on developer machines. (F4.)
5. **Multi-project story has no surface.** For N projects, the user runs N terminals, each with `open-knowledge start`, each on a different port they have to remember. No list. No switcher. No registry. No way to know what's running. Per-project MCP registration requires editing `.mcp.json` per project. Claude in project A cannot see project B's content. (`multi-project-switching-landscape` §1.)
6. **Project identity = absolute path.** Moving, renaming, or cloning a project orphans any external state keyed by that path. The system has no concept of project identity separate from where it lives on disk. This blocks any registry-based multi-project UX before it starts. (`lifecycle-edge-cases` E3.)

The compound effect: even fixing the four P0/P1 bugs above only buys back the single-project story README already promises. The multi-project story the product needs — one place to see all your knowledge bases, switch between them, not think about ports or per-project MCP setup — requires a coherent slice of additive work on top.

**Resolution.** One vertical slice covering: (a) the four onboarding bug fixes, (b) project-id as identity primitive, (c) a user-scoped projects registry, (d) an `open-knowledge hub` command that runs as a persistent background service (macOS launchd LaunchAgent, following the pattern documented in `reports/hub-install-pattern/`), (e) user-scoped MCP registration for Claude Code so per-project `.mcp.json` is no longer required for that editor. Target outcome: after one-time machine setup, each new project needs only `open-knowledge init && open-knowledge start` to be fully set up, visible in the hub at `http://localhost:5100`, and editable by Claude without any per-project MCP config step.

## 2) Goals

- **G1:** A first-time user goes from nothing installed to watching Claude edit their first markdown file in under 5 minutes, following only the README.
- **G2:** Adding a second (or Nth) project adds zero new terminal-remembering burden: `open-knowledge init && open-knowledge start` in the new directory, done. The hub reflects the new project within seconds.
- **G3:** A user who has five projects can see, at a glance, which are running and which are dormant, at a single URL (`http://localhost:5100`) that they did not have to start themselves and that survives reboot.
- **G4:** A user can move or rename a project directory without losing its registry entry, history, or "which port was it on last time" memory.
- **G5:** Claude Code is configured for open-knowledge exactly once per machine. Subsequent projects do not edit `.claude.json` or any global Claude config.
- **G6:** Every component is opt-in and reversible: `open-knowledge hub uninstall` and `bun uninstall -g @inkeep/open-knowledge` remove all the state this spec adds except for per-project `.open-knowledge/` directories, which remain on disk for manual cleanup.

## 3) Non-goals

- **[NOT NOW]** NG1: Cross-project content access from one Claude session (federation, `list_projects`/`switch_project` MCP tools, cross-project wiki-link resolution). Deferred per `multi-project-switching-landscape` §5 "explicitly defer Option 5" — wait for the hub + per-project story to reveal whether users actually want this before building it. — Revisit when: multiple power users independently ask for it, or cross-project wiki-link demand is validated.
- **[NOT NOW]** NG2: Linux (systemd user unit) and Windows (Task Scheduler / Windows Service) parity for the hub background service. macOS first; other platforms document a manual `open-knowledge hub start` fallback. — Revisit when: one non-macOS user hits this as a blocker.
- **[NOT NOW]** NG3: Xbar/SwiftBar plugin and Raycast extension consuming the registry. These are additive surfaces over the `projects.json` + `open-knowledge projects --json` primitives that this spec does ship; they can be built later with zero changes to this spec's surface. — Revisit when: someone wants to spend half a day on polish.
- **[NOT NOW]** NG4: Auto-open-editor-from-MCP (`claude-auto-open-editor-ux` Proposal 3). Separate concern with its own design space; the hub URL + manual browser tab is sufficient for v1. — Revisit when: the hub is proven and the "Claude edited a file, I want to see it now" friction remains.
- **[NOT NOW]** NG5: Per-project MCP scope unification for Cursor / Windsurf / VS Code. User-scoped MCP is a Claude-Code-specific feature today; the other editors still require per-project config in this spec. — Revisit when: Cursor (or another editor) ships a comparable user-scope primitive.
- **[NOT NOW]** NG6: Lifecycle hardening fixes E2, E5, E6, E9, E10, E11, E12, E13 from `lifecycle-edge-cases`. They're real, but none are blocker-severity for the install/multi-project path, and each wants its own targeted fix. This spec does include **E3 (project-id primitive), E7 (default `.open-knowledge/**` exclude), and E8 (`.gitignore` append on init)** because those unblock scope delivery. — Revisit when: lifecycle hardening becomes its own project.
- **[NOT NOW]** NG7: "One UI, many collab servers" dynamic vault-switcher inside a single SPA session. The hub page in this spec is a project *picker* that navigates to each project's editor URL (shape A in `multi-project-topology-and-quickstart` §2.5). Shape B/C (simultaneous multi-provider, federated tree) is deferred. — Revisit when: users ask to have two projects' editors open side-by-side.
- **[NEVER]** NG8: Cloud-hosted hub / cross-machine sync. Local-first is a product invariant.
- **[NEVER]** NG9: Electron/Tauri desktop wrapper. The hub is a localhost web page; the OS is the window manager.
- **[NOT UNLESS]** NG10: Always-on per-project collab servers (launchd-managed per-project Hocuspocus, not just the hub). Keeps the "collab servers live with the user's terminal" contract that today's architecture depends on. — Revisit only if: there's a specific workflow where project servers need to survive terminal close independently of the hub, and it cannot be satisfied by "start it again when you come back."

## 4) Personas / consumers

- **P1 — First-time user.** Has never used Open Knowledge. Reaches the README, wants to try it on one existing project. Forms a trust judgment in the first 5 minutes. If the browser UI doesn't load, or if `init` indexes 294 node_modules READMEs, they bounce.
- **P2 — Multi-project returning user.** Already has one project set up and working. Wants to add a second (personal notes, research repo, client work). Should not have to re-learn anything or touch global config again.
- **P3 — Power user with 5+ projects.** Active on 2–3 at a time, dormant on the rest. Wants a "what do I have and what's running right now?" view without opening terminals. Does not want to remember ports.
- **P4 — Rename/refactor user.** Periodically moves project directories (`~/scratch/foo` → `~/work/foo`) or renames them. Expects the tool to follow rather than stranding history.
- **P5 — Uninstaller.** Decides Open Knowledge isn't for them. Expects `hub uninstall` + `bun uninstall -g` to leave no orphan processes, no orphan plists, no orphan registry entries.

## 5) User journeys

### P1 happy path — first-time install, single project
1. User runs `bun install -g @inkeep/open-knowledge` (or `npm install -g ...`).
2. User runs `open-knowledge hub install`. Interactive prompt: *"Register Open Knowledge as an MCP server for Claude Code globally? (Y/n)"*. On yes, runs `claude mcp add -s user open-knowledge -- open-knowledge mcp`. Writes `~/Library/LaunchAgents/com.inkeep.open-knowledge-hub.plist` and `launchctl bootstrap`. Prints: `"✓ Hub running at http://localhost:5100. Open it to see your projects."`
3. User `cd`s to their project, runs `open-knowledge init`. Interactive picks editors (for Cursor/Windsurf/VS Code; Claude Code is already user-scoped). Writes `.open-knowledge/` with project-id.
4. User runs `open-knowledge start`. Banner prints `http://localhost:5173` (or next free port if 5173 is taken). The hub at `http://localhost:5100` now shows this project as LIVE.
5. User opens Claude Code in the project directory. The (user-scoped) MCP subprocess inherits cwd, finds `.open-knowledge/project-id`, connects to the live server via `server.lock`. Claude can read/write and the user sees edits live in the browser editor tab.

Total commands: 4 (`bun install -g`, `hub install`, `init`, `start`). Total one-time-per-machine commands: 2. Total per-project commands: 2.

### P2 happy path — adding a second project
1. User `cd`s to second project, runs `open-knowledge init && open-knowledge start`. Picks a different port automatically (F4 fallback). Registers in the user-scoped registry.
2. Opens `http://localhost:5100` (already running from P1's hub install). Both projects visible; both LIVE.
3. Opens Claude Code in either directory. Scoped correctly to that project via cwd + project-id.

Per-project burden: 2 commands, zero global config edits.

### P3 happy path — 5+ projects, drive-by work
1. Opens `http://localhost:5100`. Sees all 5 projects, 2 LIVE, 3 STOPPED.
2. Clicks Open on a LIVE project → editor loads.
3. Clicks Start on a STOPPED project → hub shells `open-knowledge start` via a helper command that opens in the user's terminal (or fails with a clear "run `open-knowledge start` in the project directory" message if auto-shelling is too invasive). Once started, status flips to LIVE.
4. Picks another project via the hub, goes back and forth without terminal context-switching.

### P4 happy path — rename/move
1. User runs `mv ~/scratch/research ~/work/research` (server stopped).
2. `cd ~/work/research && open-knowledge start`. On boot, reads `.open-knowledge/project-id`, finds the matching registry entry (at the old path), updates `project.path` to the new path, preserves `firstSeen`, `lastSeen`, name.
3. Hub at `http://localhost:5100` reflects the new path on next page load. No orphan entry, no duplicate, no history loss.

### P5 happy path — uninstall
1. User runs `open-knowledge hub stop`. Launchd unloaded; port freed.
2. User runs `open-knowledge hub uninstall`. Plist deleted.
3. User runs `claude mcp remove -s user open-knowledge`.
4. User runs `bun uninstall -g @inkeep/open-knowledge`.
5. Remaining state on disk: `~/.open-knowledge/projects.json`, per-project `.open-knowledge/` directories. Both documented as safe to `rm` manually.

### Failure / recovery journeys
- **Port 5100 taken at hub install time.** Hub picks next free port; writes `~/.open-knowledge/hub.lock` with the real port; prints the real URL. launchd plist is written with a port field read from `hub.lock` on each launch, so at-boot starts pick a fresh free port if the configured one is taken.
- **Port 5173 taken on `open-knowledge start`.** Fallback to `port: 0` (kernel-assigned). Banner prints the real port with a one-line note: `"Port 5173 is in use — using 3847 instead."`
- **Project-id file missing or corrupt on `start`.** Regenerate it; warn once (`"No project-id found; generated a new one. History for this project starts now."`). Do NOT match by path as a fallback — path ambiguity is what project-id exists to solve.
- **User-scope MCP already registered under a different command.** `claude mcp add -s user open-knowledge ...` fails (Claude Code refuses to overwrite). Print a clear message: `"An 'open-knowledge' MCP server is already registered at user scope. Run 'claude mcp remove -s user open-knowledge' first if you want to re-register."` Do not silently overwrite.
- **Claude Code not installed.** `claude mcp add` command not available. Detect via `which claude`; skip the MCP registration step with a one-line note and continue with hub install.
- **Non-macOS host.** `hub install` detects `process.platform !== 'darwin'`, prints `"Auto-start is only supported on macOS. Run 'open-knowledge hub start' manually in a terminal to use the hub."` and exits 0.
- **Hub process dies mid-session.** launchd's `KeepAlive: true` restarts it within seconds. Users see a brief `:5100` hiccup; next page load works.
- **User upgrades CLI with `bun install -g @inkeep/open-knowledge@next`** — binary path in the plist may become stale. Mitigation: `hub install` is idempotent; re-run after upgrades. Document this. Non-goal to detect upgrade automatically.

### Non-functional
- **Performance:**
  - Hub `GET /api/projects` completes in < 200ms for up to 50 registry entries (synchronous FS stat + PID check per entry).
  - Hub at-login cold start: process running and responding to `/` within 2s of login.
  - Registry write contention: advisory lock held for < 50ms per update.
  - `open-knowledge projects` CLI list: < 300ms for 50 entries.
- **Reliability:**
  - Registry write is atomic (write-to-temp + rename). Corrupt JSON detected on read, logged once, replaced with empty registry, old file backed up to `~/.open-knowledge/projects.json.bak.{timestamp}`.
  - Stale lock detection for registry lock uses the same `isProcessAlive` helper as server.lock.
  - Hub crash → launchd restart within ~5s (standard `KeepAlive` behavior).
- **Security/privacy:**
  - Hub binds to `127.0.0.1` only. Not network-accessible.
  - Registry file mode: 0600 (user-only).
  - Plist is user-scope (`~/Library/LaunchAgents/`), not system-wide.
  - No telemetry.
  - Hub `/api/projects` response contains absolute paths — fine for localhost-only, should NOT be exposed to any broader origin.
- **Operability:**
  - `~/.open-knowledge/hub.log` grows until rotated. v1 does not rotate; document size cap warning in CLAUDE.md. Users can `truncate -s 0 ~/.open-knowledge/hub.log` safely.
  - `open-knowledge hub status` and `open-knowledge projects` are the two debug surfaces. Both work without the hub running.
- **Compatibility:**
  - macOS 13+ (launchctl bootstrap subcommand).
  - Node 22+ / Bun 1.3+ (unchanged from current engines).
  - Claude Code with `claude mcp add -s user` support (document minimum version in release notes).

## 7) Success metrics & instrumentation

- **Primary (qualitative):** Onboard a new user (internal or external) by asking them to follow only the README. Measure: do they reach "Claude is writing in my editor" in under 5 minutes, without asking for help, without reading source code, without editing any JSON manually?
- **Secondary:** After 30 days, internal team reports whether multi-project use feels natural. Specifically: are they using the hub, the `open-knowledge projects` CLI, neither? If neither, something is wrong.
- **Tertiary:** Count of bug reports tagged "onboarding" or "multi-project" in the month after ship, compared to pre-ship baseline.
- **Instrumentation:** None. Local CLI; no telemetry pipeline. Validation is qualitative + bug-report signal.

## 8) Current state

- **CLI entrypoint:** `packages/cli/src/cli.ts` — commander.js v14. Subcommands: `start`, `mcp`, `init`, `preview`.
- **Init:** `packages/cli/src/commands/init.ts` — scaffolds `.open-knowledge/`, registers MCP per editor from interactive prompt. Does NOT write `project-id`. Does NOT write to `~/.open-knowledge/` registry. Does NOT append to project `.gitignore`.
- **Start:** `packages/cli/src/commands/start.ts` — boots Hocuspocus, serves SPA via sirv on one HTTP listener. Broken asset path at line 99 (F1). No port auto-fallback (F4). Does NOT write to registry. No project-id read.
- **MCP:** `packages/cli/src/commands/mcp.ts` — stdio MCP server, reads `server.lock` for live port discovery. Already cwd-aware to some degree (reads config from cwd) but does not walk up directories looking for project root.
- **Config schema:** `packages/cli/src/config/schema.ts` — Zod schema. Defaults: `content.dir: '.', content.include: ['**/*.md'], content.exclude: []`. This is F3.
- **Server-side per-project:** `packages/server/src/` — Hocuspocus + persistence + file-watcher + shadow repo. `server.lock` primitive at `packages/server/src/server-lock.ts` already exists and is read by `mcp.ts`. It contains `{pid, hostname, port, startedAt, worktreeRoot}` but not project-id.
- **React SPA:** `packages/app/` — no hub route today. TiptapEditor + SourceEditor + presence UI. Asset output: `packages/app/dist/` (dev) or `packages/cli/dist/public/` (shipped). No project-picker route, no `/api/projects` consumer.
- **No registry:** `~/.open-knowledge/` does not exist today. No file at that path; nothing reads or writes it.
- **No hub command:** `open-knowledge hub` does not exist.
- **No launchd integration:** No plist writer, no launchctl invocations.
- **No user-scoped MCP registration:** `init` writes per-project `.mcp.json` only.

## 9) Proposed solution (vertical slice)

### User experience / surfaces

**Install path:**
```bash
bun install -g @inkeep/open-knowledge                        # one time
open-knowledge hub install                                   # one time
cd ~/my-project && open-knowledge init && open-knowledge start   # per project
```

**Hub UI at `http://localhost:5100`:**
```
Open Knowledge

LIVE (2)
  ● my-product    ~/work/my-product     :5173   [Open]
  ● research      ~/notes/research      :3847   [Open]

STOPPED (3)
  ○ proposal-q2   ~/work/proposal-q2            [Copy `cd ~/work/proposal-q2 && open-knowledge start`]
  ○ talk-notes    ~/talks/talk-notes            [...]
  ○ journal       ~/notes/journal               [...]

MISSING (0)
```

**CLI `projects` command:**
```
$ open-knowledge projects
  STATUS   NAME          PATH                     PORT   LAST SEEN
  live     my-product    ~/work/my-product        5173   now
  live     research      ~/notes/research         3847   2m ago
  stopped  proposal-q2   ~/work/proposal-q2       —      3d ago
```

### Technical approach

**New files:**
- `packages/cli/src/commands/hub.ts` — the `hub` subcommand with nested `install` / `uninstall` / `start` / `stop` / `status` / `logs` / (run-foreground entrypoint used by the plist).
- `packages/cli/src/commands/projects.ts` — the `projects` subcommand.
- `packages/cli/src/registry/index.ts` — read/write helpers for `~/.open-knowledge/projects.json` with advisory locking.
- `packages/cli/src/registry/schema.ts` — Zod schema for the registry file.
- `packages/cli/src/launchd/plist.ts` — plist template generator + launchctl invocation helpers, darwin-guarded.
- `packages/cli/src/launchd/install.ts` — `installHubLaunchAgent()` / `uninstallHubLaunchAgent()`.
- `packages/cli/src/mcp/user-scope-register.ts` — `claude mcp add -s user` invocation + detection helpers.
- `packages/app/src/hub/HubPage.tsx` — the project picker React component.
- `packages/app/src/hub/route.tsx` — route wiring (lightweight; a single page).

**Modified files:**
- `packages/cli/src/commands/start.ts` — (1) F1 asset-path fix: prepend `resolve(cliDir, '..', 'public')` to asset lookup array. (2) F4 port auto-fallback: wrap `httpServer.listen(...)` in a retry that catches EADDRINUSE and retries with `port: 0`. (3) On successful boot, read project-id + call `registry.upsert()`.
- `packages/cli/src/commands/init.ts` — (1) Generate and write `.open-knowledge/project-id`. (2) Append `.gitignore` patterns per R11. (3) Skip `.mcp.json` Claude Code stanza if `claude mcp list -s user` shows `open-knowledge` already registered. (4) Do NOT append to registry (R5 says start writes, not init — matches "never-started projects don't clutter the list" rule).
- `packages/cli/src/commands/mcp.ts` — Add upward directory walk to locate nearest project root. Fall back to cwd-disk-only on miss with clear log line.
- `packages/cli/src/config/schema.ts` — Default `content.exclude` expands to include `node_modules/**`, `.git/**`, `dist/**`, `build/**`, `.next/**`, `.open-knowledge/**` (R2).
- `packages/cli/src/cli.ts` — Register `hub` and `projects` subcommands.
- `packages/server/src/server-lock.ts` — Optional: add `projectId` field to lock schema (read by MCP subprocess for sanity-check on connect).

**Hub process lifecycle (R7):**
- Single process. HTTP server only (no Hocuspocus).
- Routes:
  - `GET /` → serves the SPA bundle with a hub-page-initial state
  - `GET /api/projects` → registry-backed list with live-probe
  - `GET /api/projects/:id` → individual lookup
  - `GET /assets/*` → static SPA assets
- No authentication (localhost-only bind).
- Startup: acquires `~/.open-knowledge/hub.lock` (own lockfile, different from per-project server.lock); on port collision, uses `port: 0`.
- Shutdown: `SIGTERM` → release hub.lock → exit 0. Idempotent on double-signal.

**Launchd plist (R8):** Exact template from `reports/hub-install-pattern/REPORT.md`, parameterized by absolute paths resolved at install time. Install writes the file, runs `launchctl bootout` (tolerant of "not loaded"), then `launchctl bootstrap`. Uninstall reverses.

**User-scoped MCP (R9):** Shell out to `claude mcp add -s user open-knowledge -- open-knowledge mcp` during `hub install`. Detection: `claude mcp list -s user` parsed. Cursor/Windsurf/VS Code continue using per-project config.

**Cwd-aware MCP subprocess (R10):** In `mcp.ts`, add a `findProjectRoot(cwd)` helper that walks up looking for `.open-knowledge/config.yml`. Cap the walk at filesystem root or home directory. On find, use that as the effective contentDir for this session. On miss, current behavior (cwd-scoped disk-only).

### What this explicitly does NOT touch

- `packages/app/src/editor/` — no changes to the editor itself.
- `packages/core/` — no markdown pipeline changes.
- `packages/server/src/persistence.ts`, `reconciliation.ts`, `file-watcher.ts` — unchanged. The hub doesn't run Hocuspocus.
- `.mcp.json` for Cursor/Windsurf/VS Code — still written per-project by `init`.

## 10) Decision log

| ID | Decision | Alternatives | Rationale |
|---|---|---|---|
| D1 | Identity primitive = UUIDv4 in `.open-knowledge/project-id` file | git-remote URL; content hash; inode | UUID is deterministic-for-this-project, stable across moves, doesn't require git, easy to regenerate on corruption. Committed-to-repo is fine — project-id is not secret. |
| D2 | Registry lives at `~/.open-knowledge/projects.json` (user-scope), not per-machine system-scope | `/Library/Application Support/...`; `$XDG_DATA_HOME/...` | Matches claude-max pattern. Per-user is right scope (one dev, one laptop, N projects). No sudo needed. Parallel to `~/.claude.json` conceptually. |
| D3 | Hub is its own command with its own process, not an option on `start` | `open-knowledge start --hub`; always-on hub inside one of the per-project servers | Hub has a different lifecycle (launchd-managed, long-lived, across projects) than per-project servers (terminal-bound, per-project). Different concerns, different commands. |
| D4 | Hub starts as additive surface; per-project servers still serve their own SPA | Decouple SPA serving into hub only | Preserves the "one process, just works" single-project story. Decoupling is a future option (per topology report §2.5) but adds CORS complexity that the v1 hub doesn't need. |
| D5 | Hub picker navigates the browser tab to each project's URL on click | Hub is itself the editor, connects to per-project Hocuspocus | Navigating keeps per-project servers as origin-of-truth; no CORS changes needed to per-project Hocuspocus; preserves the "one Hocuspocus connection per project" model. Vault-switcher UX shape A from topology report. |
| D6 | Register MCP user-scope for Claude Code only; per-project config for others | User-scope for all editors; never user-scope | Claude Code is the only editor with a documented `--scope user` today. Other editors either don't support it or use different conventions. User-scope where supported, per-project where not. |
| D7 | macOS launchd only in v1; Linux/Windows deferred | Ship all three at once | One platform at a time is tractable; macOS is the dominant dev environment for the target user; systemd/Task-Scheduler specs are enough work to belong in their own slices. |
| D8 | Hub install is interactive (prompts for Claude Code MCP registration) | Silent install + separate command | One-command setup is friendlier; the prompt is optional (user can say no); non-interactive mode uses the existing `--yes` flag convention to auto-accept. |
| D9 | `init` does NOT write to registry; `start` does | Write on init | "Enabled but never started" is clutter. Registry should reflect "projects you've actually used." Matches the landscape report's recommendation (§6). |
| D10 | Port auto-fallback uses `port: 0` on EADDRINUSE, not a second configured port | Retry with config.port + 1, + 2, ... | Kernel-assigned port is guaranteed free; linear-retry is guesswork. Real port goes in server.lock; MCP discovery already handles variable ports. |
| D11 | `hub uninstall` does NOT remove `~/.open-knowledge/projects.json` | Remove everything on uninstall | Preserves per-project history across hub reinstalls. User can `rm` the file manually if they want a clean slate. |
| D12 | Registry is keyed by project-id; path is mutable | Key by path; key by "name" | Identity must survive moves. Path-keyed registry is exactly the bug lifecycle-edge-cases E3 identifies. |

## 12) Assumptions

- **A1:** Users are on macOS 13+. Plist `launchctl bootstrap` subcommand is available. (Pre-Catalina launchctl `load`/`unload` is a different CLI; do not support.) Verification: test on macOS 13 and 14 before ship.
- **A2:** `bun install -g` or `npm install -g` places `open-knowledge` on the user's PATH in a location that's stable enough for launchd to find via `process.argv[0]` resolution. Verification: test both runtimes.
- **A3:** `claude mcp add -s user` is idempotent enough that a second `hub install` (with the user-scope flag re-run) does not corrupt existing registration. Verification: test Claude Code CLI behavior on repeat adds.
- **A4:** The Vite build that produces `packages/cli/dist/public/` bundles all SPA assets with relative paths so the same bundle works from `file://`, `http://localhost:5173/`, and `http://localhost:5100/` without re-bundling. Verification: visit all three URLs in a real browser post-build.
- **A5:** The hub `/api/projects` endpoint can synchronously enumerate the registry + probe server.locks in < 200ms for 50 projects. Verification: benchmark with synthetic fixture.
- **A6:** Advisory file locking on `projects.json` works reliably on every filesystem the user might have `$HOME` on (APFS, ext4, network filesystems). Verification: document NFS as unsupported for `$HOME`; don't test it.
- **A7:** The F1 fix (asset-path) is truly one line and does not interact with the Vite dev plugin's asset serving. Verification: after fix, `bun run dev` still serves assets correctly via the plugin path.

## 14) Risks & mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| F1 fix breaks some other codepath (Vite dev plugin, Playwright harness) | Low | High | Integration test matrix: dev-mode `bun run dev`, packed tarball install, Playwright e2e. Run all three before merge. |
| Launchd plist install fails silently due to System Integrity Protection or permissions | Low | Medium | Test on a clean macOS user account. Explicit error on `launchctl bootstrap` non-zero exit. |
| User already has an `open-knowledge` MCP registered at user scope under a different command | Medium | Low | R9 detects and refuses to overwrite with clear message. |
| Plist binary path goes stale after CLI upgrade | High over long time | Low | Document the "re-run `hub install` after upgrade" pattern. Consider `open-knowledge hub repair` as future work. |
| Advisory lock on `projects.json` leaves an orphan lock file after a process crash | Low | Low | Same `isProcessAlive` stale-detection pattern as `server.lock`. |
| Hub at `:5100` collides with user's other localhost service | Medium | Low | Fall back to free port, write real port to `hub.lock`, document how to configure. |
| Cwd-walk in MCP subprocess walks out of a sandbox on a remote dev environment | Low | Low | Cap walk at `$HOME` or `/`, whichever comes first. |
| Users on Windows/Linux see `hub install` do nothing useful and conclude the tool is broken | Medium | Medium | Explicit platform detection + clear "not yet supported" message pointing at manual fallback. |
| The hub SPA route accidentally connects to per-project Hocuspocus WS and hits a CORS wall, confusing users | Low (we're not building that in v1) | Low | v1 hub explicitly navigates to per-project URLs on click — no cross-origin WS. Document in D5. |
| First-time user runs `hub install` but doesn't have Claude Code — interactive prompt is confusing | Medium | Low | Detect `which claude`; skip the prompt entirely if not installed. |
