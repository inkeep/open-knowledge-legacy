## 1) Problem statement

**Situation.** PR #270 just merged 8 contract surfaces (S1-S8) spanning `core` / `server` / `app` / `desktop` — `POST /api/upload` + envelope (S1), `UploadWriteError` typed class (S2), `ClassifiedLinkTarget` discriminated union (S3, bidirectional — both app and server consume it), `DiskEvent` + `assertNeverDiskEvent` (S4, exemplary), `AssetViewerRegistry` (S5, empty at landing), `IpcChannelMap` (S6), `PropDef` (S7), `WikiEmbed*` compat descriptors (S8). The codebase has clear convergent patterns already (`{ ok, error }` envelope across **256 error sites + 30 success sites = 286 sites total in `api-extension.ts`**; typed error classes with `name` + `reason`-field discriminator; `z.discriminatedUnion` precedent in `packages/core/src/config/errors.ts:52` (`KnownConfigValidationErrorSchema`); `assertNever*` precedent in `assertNeverDiskEvent`). The new surfaces also shipped concrete drift: kebab-case enums mixed with English sentences in `error` fields (~98 distinct error strings), zero exhaustiveness gates on 4 of 5 shipped discriminated unions, `UploadWriteReason` flowing across the JSON boundary as an unstructured string with no typed consumption on the client (server throws a typed class; the wire body's `error` / `message` fields are read only as display strings — see `evidence/verified-current-state.md` C1), and field-name divergence (`ok` vs `found`). Three concrete next-round work items will touch these surfaces within days-to-weeks: typed-component-nodes Phase 2 viewers (S5's first registrar), cb-v2 prop file-upload server-endpoint split (S1's next consumer), and three open PRs on `JsxComponentMeta` (#380 / #374 / #372 — #380 has since merged).

**Complication.** Three pressures converge:

1. **Inherited drift.** Each new HTTP route, IPC channel, or descriptor copies from the nearest existing site. Without a defensible cut now, the next 5+ surfaces inherit the inconsistencies and re-derive the discipline per-surface.
2. **Cross-package contract erosion + display-only client.** `UploadWriteReason` is type-erased at the JSON boundary — the server's typed `reason` field flows out as an unstructured string in `error` / `message`, and the client today reads body fields purely for *display* (no branching on reason). The hardening tightens the wire contract with typed `type` tokens — primarily a server-side win (helper consolidation, telemetry, schema-as-SSOT). Client-side typed narrowing is option-value insurance for future branching, not a load-bearing current-day client benefit. There is no compile-time link from server-emitted reason to anything on the client today. `ClassifiedLinkTarget` is bidirectional (server + app); narrowing without exhaustiveness gates breaks consumers in distant packages silently.
3. **First-consumer window.** `AssetViewerRegistry` is empty at landing. Phase 2 viewers (B1 in worldmodel) are the first consumer. Whatever ordering / unregister / overlap policy this hardening picks lands as ground truth at zero friction cost; any later policy retrofit pays full cost.

**Resolution.** Adopt **RFC 9457 Problem Details** as the canonical HTTP error envelope (D22 LOCKED, batch 4). Pin per-handler request + success Zod schemas plus a shared `ProblemDetailsSchema` as SSOT in `core/src/schemas/api.ts`, with the typed `errorResponse(res, status, type, title, detail?, instance?)` helper extracted to `packages/server/src/http/error-response.ts` and adopted across **all 57 handlers in `api-extension.ts`** (~286 sites total — ~256 error-side + ~30 success-side migrations to drop the `{ ok: true }` wrapper). Wire shape: success → `{ ...data }` flat (no `ok: true` wrapper, content-type `application/json`); error → `{ type: 'urn:ok:error:<kebab-token>', title: required English, status: int, instance?: UUID, detail?: string }` (content-type `application/problem+json`; URN form per RFC 9457 §3.1.1's recommendation against path-only relative URIs — see D38). Client-side narrowing uses HTTP status (`if (!res.ok)`) to choose between `ProblemDetailsSchema` and the per-handler success schema (two-step parse). **Client-side migration is in lockstep with server migration**: 23 client sites in `packages/app/src` that read `data.ok` / `body.ok` / `raw.ok` migrate to HTTP-status discrimination (Reopen 1 = B, D32). Add `assertNever*` exhaustiveness gates on remaining shipped unions, with a single **structural meta-test** that scans for switch-statement consumer sites without `default: assertNever*` (Reopen 2 = B, D33; precedent #20 style; replaces per-DU type-test files that tested only themselves). Rename `AssetViewerLookupResult.found → ok` (D4 LOCKED, restored batch 7 after D35 reversal under deeper analysis — codebase converges 50+ sites on `ok`; the single `found` outlier eliminated); harden `AssetViewerRegistry` lifecycle (unregister fn, ordering model, collision warning, idempotent edges) before B1 registers; rename `handleUploadImage → handleUploadAsset` (D11 promoted to LOCKED via D24). Multipart and JSON request bodies validated via a `withValidation()` middleware wrapper at `packages/server/src/http/request-validation.ts` (Reopen 3 = B, D34) — handlers can't be added without going through the wrapper (structural enforcement). IS6 phasing includes: an **allowlist-based meta-test from PR1** that gates against regression of already-migrated handlers, an explicit **rollback unit** discipline (each cluster PR is a single revert unit), and a **streaming-endpoint checkpoint** after PR1+PR2 merge (validate canonical pattern handles NDJSON sites at `api-extension.ts:5054, 5359, 5569`; Reopen 6, D36). Defer S6 IPC reason-union work to PR #354 typed-IPC migration with sharpened expiry trigger; keep MCP envelope alignment, Idempotency-Key, branded IDs, and `PropDef` Zod-ification deferred for non-expediency reasons (see §3 confirmed-deferred rationale).

## 2) Goals

- **G1.** Establish one canonical RFC 9457-compliant HTTP error envelope as a Zod schema in `core`, importable by both server emission and client consumption, eliminating type-erasure at the JSON boundary across all 57 handlers (D12, D22).
- **G2.** Add `assertNever*` exhaustiveness gates on shipped discriminated unions consumed across package boundaries (`ClassifiedLinkTarget`, `ProblemTypeToken`), with per-DU compile-time exhaustiveness type-tests (D19, D25).
- **G3.** Normalize the `ok` vs `found` result-field-name divergence on `AssetViewerLookupResult` to `ok` (single reader site).
- **G4.** Harden `AssetViewerRegistry` lifecycle (`register()` returns unregister fn; document ordering model; warn on extension-key collision; idempotent re-registration and unregister semantics — Q8 deferred to a later batch but tracked) before its first consumer (typed-component-nodes Phase 2 viewers) registers.
- **G5.** Every handler in `api-extension.ts` is uniformly RFC 9457-compliant; each new HTTP route copies the canonical pattern without re-deriving the discipline. Per-handler request + success Zod schemas plus shared `ProblemDetailsSchema` are the SSOT.
- **G6.** Migrate request-body validation across handlers to per-handler Zod request schemas (D23, FR12) so the request side mirrors the response side's SSOT discipline.

## 4) Personas / consumers

### P1: Internal devs writing next-round capability work

- **JTBD:** When I write the next HTTP route / IPC channel / descriptor / discriminated union, but the existing patterns are inconsistent, help me copy a canonical shape from one place so I don't re-derive the discipline (and don't propagate drift).
- **Current workflow + workarounds:** Each PR author copies from the nearest existing site (`handleUploadImage` → next route copies its error shape; `assertNeverDiskEvent` → next DU author may or may not add a guard). No central canonical pattern.
- **Pain points:** PR review catches drift inconsistently. Cycle 47-49 of PR #270's review surfaced 3 such items (file-watcher loop ordering, `mintTempUploadPath` sync throw classification, agent-local path in comment) — pattern is "fresh-eyes audit catches drift after merge."
- **Trust/security sensitivities:** Hardening must not break agent-attribution boundary (precedent #24); must not break STOP rules (`fs-traced` wrappers, `extractAgentIdentity` at every mutating handler); must not invent envelope that breaks identity-extraction.
- **Success in their terms:** Next route they write goes through `withValidation(XyzRequestSchema, ...)` middleware + emits errors via `errorResponse(res, status, type, title, ...)` — both imported from canonical paths; next DU they author has its own `assertNever*` helper alongside, and the structural meta-test (D33) catches any consumer site that forgets `default: assertNeverXyz`; PropPanel for a new viewer doesn't re-derive collision policy.

### P2: LLM agents writing via `/api/agent-write*`

- **JTBD:** When I write markdown via `/api/agent-write-md`, help me know whether the call succeeded by reading the response shape, with errors I can route on.

- **Pain points:** None — agent-write envelope is already aligned.
- **Trust/security sensitivities:** Agent attribution must persist (precedent #24); identity extracted via `extractAgentIdentity` from query params (multipart precludes JSON body).

### P3: Operators triaging Pino structured logs

## 5) User journeys

Internal-only spec; surfaces are TypeScript types + JSDoc + tests, not user UI. Journeys are dev-experience micro-paths.

### P1.1 — Adding a new HTTP route after the hardening

1. **Discovery.** Dev opens `api-extension.ts`, sees neighboring route handlers wrapped by `withValidation(XyzRequestSchema, async (req, res, body) => { ... })` and emitting errors via `errorResponse(res, status, type, title, detail?, instance?)`.
2. **Setup.** Defines `XyzRequestSchema` and `XyzSuccessSchema` in `core/src/schemas/api.ts` alongside existing schemas (each with `satisfies StandardSchemaV1<...>` per D26; `.loose()` per repo convention).
3. **First use.** `errorResponse(res, 400, 'urn:ok:error:malformed-input', 'Field X is required')` — no string-concatenation, no inline `json(res, ...)`. Compile-error if `type` isn't in `ProblemTypeSchema`.
4. **Ongoing use.** Adding a new error type = add token to `ProblemTypeSchema` union + emit. Client-side `if (!res.ok) safeParse(ProblemDetailsSchema, body)` narrows automatically.
5. **Failure / debug.** New route author tries to ship `{ ok: false, error: '...' }` inline. CI's allowlist-based meta-test (D36) fails the build — handler is on the unmigrated allowlist OR uses `errorResponse`, no third path.
6. **Growth.** Future MCP `upload_asset` tool wraps the canonical envelope inside `{ content, isError? }`; same payload shape, transport-specific outer wrap.

### P1.2 — Registering a viewer against `AssetViewerRegistry` (Phase 2 PR)

1. **Discovery.** Phase 2 viewer author (Nick / Mike / similar) reads `asset-dispatch/types.ts` JSDoc.
2. **Setup.** Defines `viewer: AssetViewer` with `exts: ['pdf']` + `render: (ctx) => { ... }`.
3. **First use.** `const unregister = assetViewerRegistry.register(viewer)`. Returns unregister fn (matches React 19 ref-callback cleanup pattern).
4. **Ongoing use.** Hot-reload / test environments call `unregister()` to clean up.
5. **Failure / debug.** Two viewers register `'pdf'`. Console warning: "Viewer collision on `pdf`: replacing existing viewer." Last-registered wins (documented).
6. **Growth.** Image lightbox registers `['png', 'jpg', ...]`; video viewer registers `['mp4', 'webm']`. No friction.

### P1.3 — Switching on `ClassifiedLinkTarget`

1. **Discovery.** Dev writes `switch (target.kind)` for a new consumer (e.g., right-click context menu for asset hrefs).
2. **Setup.** Adds cases for `'doc'`, `'external'`, `'anchor'`, `'asset'`. Forgets `'asset'` initially.
3. **First use.** `default: assertNeverLinkTarget(target);` — TypeScript reports `'asset'` not assignable to `never`. Compile error points at the missing case.
4. **Ongoing use.** Future `ClassifiedLinkTarget` variant addition surfaces at every `assertNeverLinkTarget` site. Same discipline as `assertNeverDiskEvent`.

### Non-functional requirements

- **Performance:** Schema validation overhead on hot path (`POST /api/upload` response) is bounded — Zod v4 typical parse time for a 5-field flat object is sub-microsecond; acceptable on every request. No streaming-size sensitivity (response is small fixed shape).
- **Reliability:** Smoke-test (FR9) catches DU-member-validation footguns at module load before a real request can hit the broken schema. Compile-time discipline (FR5, FR2 closed union) catches drift at PR-author time.
- **Security/privacy:** No new attack surface. `error.instance` correlation ID must be a UUID or other unbounded-cardinality-safe value (not file path / agent name). Existing precedent #24 attribution boundary preserved (handlers still call `extractAgentIdentity`).
- **Operability:** `error.instance` enables log lookup. `error.cause` (ES2022) chain via Pino's std serializer surfaces underlying errno / syscall / path — already in place via PR #270's cycle-49 `UploadWriteError` constructor.
- **Cost:** Added dependencies: `@standard-schema/spec` (type-only; zero runtime cost) for the `satisfies StandardSchemaV1<T>` type assertion. Zod already in repo. Added bundle weight on app: minimal — `ProblemDetailsSchema` + per-handler success schemas + `withValidation` wrapper add < 5KB gzipped total (Zod's tree-shaking handles unused schemas; client only imports parse helpers it uses).

### System design

- **Architecture overview:** Schema-first SSOT in `core/src/schemas/api.ts`. Per-handler success schemas (`XyzSuccessSchema`) + per-handler request schemas (`XyzRequestSchema`) + shared `ProblemDetailsSchema` (RFC 9457). Server emits errors via `errorResponse(res, status, type, title, detail?, instance?)` helper that runs `.parse()` against `ProblemDetailsSchema`. Server validates requests via `withValidation(XyzRequestSchema, handler)` middleware wrapper at registration. Client uses HTTP-status discrimination (`if (!res.ok)`) + `safeParse` for narrowing.
- **Data model:** No DB changes. PM schema unchanged.
- **API/transport:** **Wire shape changes on both success and error paths.** Success: `{ ok: true, ...data }` → `{ ...data }` flat (drop `ok: true` wrapper, content-type `application/json`). Error: `{ ok: false, error: 'string' }` → `{ type: 'urn:ok:error:<kebab>', title: 'English sentence', status: int, instance?: 'uuid', detail?: 'longer message' }` flat (content-type `application/problem+json`). Only in-repo consumers (no third-party API today). Client-side migration of all 23 affected sites is in scope (FR4 / D32) and lockstepped with server cluster PRs.
- **Auth/permissions:** Unchanged. `extractAgentIdentity` continues at handler entry per precedent #24. Per F5 audit finding, the policy is now explicit: body-shape-failure errors (FR12 `withValidation` wrapper rejecting malformed input) emit pre-identity (anonymous, semantically OK); semantic-failure errors (handler logic) emit post-identity (attributed). The `attribution-sweep-coverage.test.ts` extension (FR17 + an ordering check) enforces this distinction.
- **Enforcement point(s):** Compile-time TypeScript narrowing on `errorResponse` signature + `ProblemTypeSchema` enum. CI: schema smoke test (FR9), allowlist meta-test (FR17), structural exhaustiveness meta-test (FR11 b), per-handler narrow-integration smoke tests (FR10), negative type tests (FR11 a). No runtime feature flag.
- **Observability:** Pino structured logs unchanged in emission shape; `error.instance` UUID added to error log lines for grep correlation. New counter `ok.api.error.count{type, handler}` (FR14 / D30) increments inside `errorResponse` helper.

#### Data flow diagram

- **Primary flow (upload success):** Client `POST /api/upload` (multipart) → server `withValidation(UploadRequestSchema, handleUploadAsset)` validates `parentDocName` etc. → handler receives typed `body` → `readUploadBody` streams binary → atomic link → server emits flat `{ src, deduped?, sha?, byteLength? }` (no `ok` wrapper) with `application/json` → 200 OK → client `safeParse(UploadAssetSuccessSchema, body)` after `if (res.ok)` → narrowed success path.
- **Primary flow (upload error):** Same up to error site → server calls `errorResponse(res, 4xx/5xx, 'urn:ok:error:<token>', 'English title', detail?, instance?)` → helper sets `application/problem+json`, generates UUID, increments `ok.api.error.count`, emits flat `{ type, title, status, instance, detail? }`, logs Pino → client sees `if (!res.ok)` → `safeParse(ProblemDetailsSchema, body)` → narrowed `result.data.type` (typed URN literal) for routing + `result.data.title` for display.
- **Shadow paths to test:**
  - **nil / missing:** Client receives a non-JSON response (proxy 502 HTML, network failure) → both `safeParse` calls fail → client throws `HttpResponseParseError` (FR15 / D31) — distinct from `ProblemDetails` parse failure.
  - **wrong type:** Server returns malformed JSON (server bug or version skew) → `safeParse` fails → client logs + throws `HttpResponseParseError`. The `instance` field (when present) aids triage.
  - **conflict:** N/A for upload (no concurrent-write semantics).
  - **partial failure:** Streaming pipeline fails mid-write → existing `UploadWriteError` path with `cause` chain → maps to typed `type` URN. Existing behavior, now typed.

### IS5 — Server-client typed `ProblemDetails` consumption + RFC 9457 two-step parse (D22 / D32)

- **Goal:** Server emits RFC 9457 Problem Details on errors; client uses HTTP-status discrimination (`if (!res.ok)`) to choose between `ProblemDetailsSchema` (errors) and per-handler `XyzSuccessSchema` (success) — the standard RFC 9457 two-step parse pattern. Today the client never reads a typed reason — the wire body is consumed as a display string (`evidence/verified-current-state.md` C1). **The hardening tightens the wire contract; client-side typed narrowing is option-value insurance for future branching, not a current-day client win.** The current load-bearing client benefit is preventing the silent `data.ok === undefined` regression that the wire-shape change would otherwise cause across 23 sites.
- **Non-goals:** No client-side schema generation tooling; no SDK gen.
- **Requirements:** FR3, FR4, FR15 (HttpResponseParseError), D8 (UploadWriteReason ↔ ProblemTypeSchema alignment), D22, D32.
- **Proposed solution:** Server emits via typed `errorResponse(res, status, type, title, detail?, instance?)` helper (D15 / D22) that produces RFC 9457-compliant bodies. Client uses `if (!res.ok) safeParse(ProblemDetailsSchema, body)` (errors) or `safeParse(XyzSuccessSchema, body)` (success). 23 client sites in `packages/app/src` reading `data.ok` / `body.ok` / `raw.ok` migrate per FR4 in lockstep with their handler's cluster PR. Hand-rolled response lookalike types (`UploadResponseBody`, etc.) retired in favor of `z.infer<typeof XyzSuccessSchema>`. Non-contract responses (network errors, proxy 502 HTML) → `HttpResponseParseError` per FR15.
- **Owner / DRI:** Nick (server) + per-handler client owners (per cluster PR ownership).
- **Next actions:** Implement helper (D15) + middleware (D34); migrate `handleUploadAsset` (D24) as canonical example in PR1; migrate other handlers + client consumers in stacked cluster PRs; retire lookalike types.
- **Risks + mitigations:** Network errors / non-JSON responses → `HttpResponseParseError` (FR15). Client-side migration must be lockstepped with server cluster PRs (D32) — without that, 23 client sites silently regress (covered in §14 risks).
