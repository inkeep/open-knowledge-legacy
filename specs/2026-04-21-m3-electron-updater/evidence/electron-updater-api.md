# electron-updater API — primary-source findings

Context: parent [[specs/2026-04-11-electron-desktop-app/SPEC]] §8.10 and milestone [[specs/2026-04-21-m3-electron-updater/SPEC]]. All `file:line` references below are against `electron-userland/electron-builder@master` (the package publishes from this repo). **Line-number caveat (F6 audit):** the line numbers in this file were gathered against `master` at research time and do not match the final pinned version. Claims (events exist, codes exist, guards exist) are correct; line numbers are traceability aids that will drift. At implementation time (when A4 pins the exact `electron-updater` version), re-cite line numbers from the pinned tag's blob URL (`blob/v<X.Y.Z>/packages/electron-updater/src/...`). Or drop line numbers and cite `file#symbol` only. ## 1. Event surface

Canonical event-type map, `AppUpdater.ts:50-59`:

```ts
export type AppUpdaterEvents = {
  error: (error: Error, message?: string) => void
  login: (info: AuthInfo, callback: LoginCallback) => void
  "checking-for-update": () => void
  "update-not-available": (info: UpdateInfo) => void
  "update-available": (info: UpdateInfo) => void
  "update-downloaded": (event: UpdateDownloadedEvent) => void
  "download-progress": (info: ProgressInfo) => void
  "update-cancelled": (info: UpdateInfo) => void
  "appimage-filename-updated": (path: string) => void
}
```

Emit sites (all in `AppUpdater.ts`):

| Event                       | Line                                 | Payload                                                                                        | Fires when                                                               | M3 handler                                                                                       |
| --------------------------- | ------------------------------------ | ---------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------ |
| `checking-for-update`       | 441                                  | `()`                                                                                           | `checkForUpdates()` starts the network request                           | log only                                                                                         |
| `update-available`          | 463                                  | `UpdateInfo` (`version`, `files`, `releaseNotes`, `stagingPercentage`, `minimumSystemVersion`) | Remote version > current version after manifest parse                    | log; download auto-starts when `autoDownload=true` (default, `AppUpdater.ts:72`)                 |
| `update-not-available`      | 454                                  | `UpdateInfo`                                                                                   | Remote version ≤ current version                                         | log only                                                                                         |
| `download-progress`         | 808 (guarded on `listenerCount>0`)   | `ProgressInfo` (`bytesPerSecond`, `percent`, `total`, `transferred`)                           | During download; rate-limited on listener presence                       | log at debug; M3 has no UI surface (NG: no progress toast)                                       |
| `update-downloaded`         | 628 via `UPDATE_DOWNLOADED` constant | `UpdateDownloadedEvent` (ver info + `downloadedFile` path)                                     | ZIP downloaded, cache + proxy-server ready, native Squirrel.Mac notified | fire Toast A ("will install when you quit") per [[specs/2026-04-21-m3-electron-updater/SPEC]] D3 |
| `error`                     | 338                                  | `(Error, message?)` with `error.code` = `ERR_UPDATER_*` (see §2)                               | Any failure in check/download/parse/verify                               | classify per §2; silent retry or log                                                             |
| `update-cancelled`          | 485                                  | `UpdateInfo`                                                                                   | User / app called `cancellationToken.cancel()`                           | not used; autoDownload is fire-and-forget in M3                                                  |
| `login`                     | 411                                  | `(AuthInfo, LoginCallback)`                                                                    | HTTP 401 basic-auth challenge                                            | NOT WIRED — public repo, no auth                                                                 |
| `appimage-filename-updated` | (Linux-only, via `AppImageUpdater`)  | `string`                                                                                       | AppImage renamed post-install                                            | NOT WIRED — macOS-only per parent D51                                                            |

**Do-not-wire:** `login`, `update-cancelled`, `appimage-filename-updated`. The first is dead for a public GitHub repo; the second would require exposing `CancellationToken` to the renderer (out of scope); the third is Linux-specific.

## 2. Error classes

`error` always fires with an `Error` whose `.code` is an `ERR_UPDATER_*` string set by `newError(message, code)` in `builder-util-runtime/src/error.ts`:

```ts
export function newError(message: string, code: string) {
  const error = new Error(message)
  ;(error as NodeJS.ErrnoException).code = code
  return error
}
```

Discriminate on `err.code`, not `err.message`. Catalog observed in source:

| `err.code`                                  | Origin                                                                                     | Likely cause                                                                                                                                                                                                                                                 | M3 response                                                                                                                                                                                                                                                                                                               |

Key architectural fact: `MacUpdater.ts` does **no** code-signature validation itself. All signature checks happen inside Electron's native `autoUpdater` (Squirrel.Mac), which emits a generic `Error` on `this.nativeUpdater.on("error", …)` (`MacUpdater.ts:31-33`). The shape "notarization changed between v1 and v2" therefore surfaces as an opaque `Error` without an `.code` — classifiable only by message-substring match, which is fragile. M3's rule: anything without a recognizable `ERR_UPDATER_*` / `HTTP_ERROR_*` code lands in the "silent retry + structured log" bucket.

## 3. GitHub provider asset-path requirements

The URL format depends on host, via `GitHubProvider.ts`:

- **github.com (our case)** — fetches the manifest from the release-download CDN, constructed by `getBaseDownloadPath(tag, channelFile)` (line \~208): `https://github.com/inkeep/open-knowledge/releases/download/<TAG>/latest-mac.yml`. The tag is resolved from `/inkeep/open-knowledge/releases/latest` (JSON) at lines \~160-175 — so there IS one `api.github.com`-equivalent request per check (the 302-redirected `/releases/latest`), but it's the unauthenticated HTML endpoint, not the rate-limited `api.github.com` one. Evidence: comment at line \~160 — `// do not use API for GitHub to avoid limit, only for custom host or GitHub Enterprise`.
- **GitHub Enterprise / custom host** — uses `api.github.com/repos/OWNER/REPO/releases/latest` (authenticated, rate-limited). Not our path.

**Filename is `latest-mac.yml` exactly**, derived from `util.ts`:

```ts
export function getChannelFilename(channel: string): string {
  return `${channel}.yml`
}
```

Combined with `Provider.ts:52-57` (`getChannelFilePrefix` returns `-mac` on darwin) and `getDefaultChannelName()` returning `"latest"`, the default channel is `latest` and the filename is `latest-mac.yml`. NOT `latest-mac-universal.yml` — the universal suffix is not a thing in the default channel-naming convention. If a tag ships multiple arch-specific DMGs, they are enumerated inside `files[]` of the one `latest-mac.yml`; the name of the yml itself does not change.

**Private repo auth.** Our target is a public repo, so no token is needed — the CDN `releases/download/...` path serves unauthenticated. For private repos, electron-builder's docs require `GH_TOKEN` in the **runtime** environment of the packaged app, and all checks route through `api.github.com` (5000 req/hr/user limit). See [`updates-and-distribution.md`](../../../reports/electron-desktop-app-operations-2025/evidence/updates-and-distribution.md) "Private repo" note.

**DMG name in `latest-mac.yml`.** Inside `latest-mac.yml`, each entry in `files[]` has a `url:` field (filename only, resolved relative to the release-download base path). The `MacUpdater` **does not download the DMG** — it downloads the **`.zip`** file (`MacUpdater.ts:89`, `findFile(files, "zip", ["pkg", "dmg"])`), pipes it through a local proxy HTTP server (`MacUpdater.ts:137` onward) to Squirrel.Mac, which performs the signature-verified swap. The `.dmg` is the first-install distribution; the `.zip` is required **in addition** for updates. electron-builder's `mac` target default is `["dmg", "zip"]` per [electron.build/auto-update](https://www.electron.build/auto-update.html) ("`zip` target for macOS is **required** for Squirrel.Mac"). M2 shipped `.dmg` only because auto-update was out of M2's scope (M2 NG4); M3 is the first consumer that needs `.zip`, so D6 lands the delta in M3 (F10 audit — neutral framing, not "M2 deviated"). Concretely: our release must attach `OpenKnowledge-<ver>-mac-universal.zip`, `OpenKnowledge-<ver>-universal.dmg`, `OpenKnowledge-<ver>-universal.dmg.blockmap`, and `latest-mac.yml`. The `url:` fields inside `latest-mac.yml` must match those asset filenames exactly — electron-builder writes them consistently, but a hand-edited `stagingPercentage` injection must not touch `files[].url` or `path`.

## 4. Dev-mode dry-run approaches

`AppUpdater.ts:429-438` gates `checkForUpdates()` on `app.isPackaged || forceDevUpdateConfig`. When inactive, logs: `Skip checkForUpdates because application is not packed and dev update config is not forced`.

Three documented approaches, ranked:

1. **`autoUpdater.forceDevUpdateConfig = true` + `dev-app-update.yml`** — canonical. Set the flag, place `dev-app-update.yml` at `app.getAppPath()` mirroring the `publish:` block from `electron-builder.yml` (YAML form). The updater loads it via the same code path as production's `app-update.yml` (`AppUpdater.ts` `loadUpdateConfig`) and points at real GitHub Releases. Highest fidelity — exercises `GitHubProvider`, real network, real `latest-mac.yml`, real `.zip` fetch. Install step still fails in dev (unpackaged `.app`), but every event up through `update-downloaded` fires correctly. Primary source: [electron.build/auto-update](https://www.electron.build/auto-update.html) "Debugging" section.
2. **`setFeedURL(urlOrConfig)` pointing at a local HTTP server** — `AppUpdater.setFeedURL(options: PublishConfiguration | AllPublishOptions | string)` routes a bare-string URL through `GenericProvider`. Spin up a local server serving a hand-crafted `latest-mac.yml` + fake `.zip` with a valid `sha512`. Tests provider plumbing and local event fan-out without GitHub involvement. Still requires `forceDevUpdateConfig = true` to bypass the `isPackaged` gate. Medium fidelity — exercises the download pipe and `update-downloaded` dispatch, but uses `GenericProvider` not `GitHubProvider`, so the URL-resolution logic of §3 is not exercised. Good for M3's `smoke-mock-update.mjs` harness (SPEC §4).
3. **Event-stub unit tests** — construct an `AppUpdater` subclass, override `doDownloadUpdate` + `getUpdateInfoAndProvider`, drive `this.emit(...)` directly. Lowest fidelity but fastest — good for asserting IPC shape + error-classification logic in isolation (the `auto-updater.test.ts` file in [[specs/2026-04-21-m3-electron-updater/SPEC]] §4).

Recommendation for M3: use approach **2** in `smoke-mock-update.mjs` (no GitHub dependency, exercises end-to-end except Squirrel.Mac swap) AND approach **3** in `auto-updater.test.ts` (fast feedback on routing + error code mapping). Approach 1 becomes the end-state smoke post-M2-creds-procurement — it's the real-world dry-run against a staged pre-release tag.
