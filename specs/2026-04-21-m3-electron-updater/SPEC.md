## 1) Problem statement

**Situation.** M2 produced the signed-DMG scaffolding (electron-builder.yml with `publish: github`, `afterSign.mjs` notarize+staple+fuse-verify, `desktop-build.yml` with signed-vs-unsigned mode detection, `latest-mac.yml` + `.dmg.blockmap` already generated by electron-builder and uploaded as artifacts). `electron-updater` is NOT yet a dependency of `@inkeep/open-knowledge-desktop`, and the main process has no auto-update wiring. M2's end-state DOD is gated on (a) Apple Developer Program cert procurement (1–6 weeks) and (b) the bun-workspace universal-merge SHA-parity gap (`@napi-rs/keyring` prebuilt binaries identical across arches — see M2 §6). `packages/desktop/package.json` is `private: true` + version `0.0.0`; changesets skips it. `release.yml` publishes npm packages + creates a GitHub Release from the changesets-published PR, but nothing attaches the macOS DMG or `latest-mac.yml` to that release.

**Complication.** Without M3, every DMG is terminal — users must manually re-download to update, which breaks the "install once, live there" UX parent §6 J6 promises and defeats the P1-persona bet ("zero terminal contact"). M7 (first design-partner build for Nick + Miles) has M3 as a hard dependency. And because the native Squirrel.Mac framework (which Electron's native `autoUpdater` + `electron-updater` sit on top of) signature-verifies every update against the installed `.app`'s signing identity, the end-state smoke (v0.1.0 → v0.1.1 silent upgrade on a fresh Mac) requires **two** signed+notarized+released DMGs — gating M3's DOD on M2's procurement **and** the universal-merge fix. But the scaffolding (main-process autoUpdater wiring, toast + "What's new" UX, desktop-package version bootstrap, J7a failure handling, structured logging) is independent of that gate — it can be built, type-checked, unit-tested, and dry-run-smoked today against a mock update manifest. The release workflow itself (`desktop-release.yml`) lands today but its execution path verifies post-M2-creds — same creds gate as M2 FU-2 (C8 audit — §1 framing tightened).

**Resolution.** Mirror M2's two-tier decomposition. Ship M3's scaffolding now: wire `electron-updater`, bootstrap the desktop version to 0.1.0, add a release-tag-triggered workflow that attaches DMG + `latest-mac.yml` to a GitHub Release, add the toast layer (update-available-after-download + "What's new" post-update first-launch) keeping persistent affordances at zero (Obsidian-strict for non-ephemeral state), and cover the J7a failure modes with structured logs and silent retry. Gate the end-state DOD (real v0→v1 silent upgrade on a fresh Mac) on M2's creds + universal-merge fix, same pattern M2 used for its own creds-gated DOD items. The moment M2 produces a first signed DMG, the M3 release path flips on and the DOD closes.

## 2) Goals

- **G1.** Every piece of electron-updater wiring, release-pipeline config, toast plumbing, and smoke harness for the M3 path is in place and verified-where-possible, so M3 closes within minutes of the first signed-release DMG landing — not days.
- **G2.** Toast UX satisfies parent §6 J6 "no nag" invariant AND promotes the parent's *optional* "What's new" toast (parent J6 step 7) to an always-shown affordance via D3 (F18 audit — scope delta explicit). Gives P1 enough signal that an update has been staged: permanent-until-clicked toast on `update-downloaded` (D11), "What's new" toast on first launch post-update (D9). Zero persistent chrome affordance (no menu entry, no status-bar badge, no Dock badge) per D2.
- **G3.** `update-error` events from `electron-updater` are handled gracefully per parent §7 J7a — silent retry on next launch, structured log, never user-visible breakage in the common case.
- **G4.** Dry-run smoke works in dev mode against a **mock update manifest** (local HTTP server + pre-built fake DMG) so the wiring is verified-wired before signed DMGs exist.
- **G5.** `bun run check` (canonical quality gate) stays green — no regressions to M1/M2 flows.

## 3) Non-goals

- **NG1.** Producing a **real signed** v0.1.0 → v0.1.1 upgrade this session — blocked on M2 end-state (cert procurement + universal-merge fix). The dry-run mock-manifest smoke is the deliverable; the real smoke is FU.
- **NG2.** Closing the bottom two items of M3's parent §14 DOD (actual silent-upgrade on a fresh Mac, kill-updater-mid-download failure smoke against real notary service). Both are creds-gated and verified post-procurement.
- **NG3.** **Beta channel / multi-channel routing** (`autoUpdater.channel = 'beta'`). Single `latest` channel only.
- **NG4.** **Windows/Linux.** macOS-only per parent D51.
- **NG5.** **Auto-rollback on crash loop.** If v0.1.1 crashes on launch, users re-download v0.1.0 manually from GitHub Releases. Rollback automation is a later milestone.
- **NG6.** **Sentry / crash-reporting integration.** Parent §8.10 + M7 DOD mention it; out of scope for M3.
- **NG7.** **Fixing the M2 universal-merge blocker.** Tracked in M2 FU-1; not M3's problem.
- **NG8.** M4 (URL scheme), M5 (keyring end-to-end), M6 (MCP wiring), M7 (design-partner build).

## 6) Known gaps / blockers (documented, not fixed here)

### External: Apple Developer Program + notary creds

Same blocker as M2. M3's end-state DOD cannot be verified until creds land in GitHub secrets. This spec ships scaffolding; FU closes the DOD.

### External: Universal-merge SHA-parity gap

Same blocker M2 §6 documents. M3's release workflow triggers `build:mac` (signed path), which will hit the same `@electron/universal` failure until M2 FU-1 lands. The `build:mac:unsigned --arm64` per-arch path works, but produces an unsigned DMG which `electron-updater` refuses to accept as an upgrade target (signature-verify fails). M3's release path is therefore serialized behind M2 FU-1.

> Decisions will be captured here as the iterative loop resolves open questions. Each decision gets a resolution status (LOCKED / DIRECTED / DELEGATED).

### D1 (DIRECTED, 2026-04-21). Scope = scaffolding tier, mirroring M2

Ship electron-updater wiring, release-pipeline, version bootstrap, toast UX, and dry-run smoke now. Gate real v0.1.0 → v0.1.1 smoke on M2 end-state (creds + universal-merge fix). **Why:** decouples work that can land today from work blocked by procurement. **Source:** intake §4 batch, user-confirmed.

### D2 (DIRECTED, 2026-04-21). Persistent-affordance level = Obsidian-strict (option a)

No menu entry, no status-bar badge, no persistent dock/menu/chrome indicator. **Scope of "persistent indicator"** (clarified per C5 challenger): menu-bar / status-bar / Dock badge only. **NOT included** in the ban: renderer-overlay toasts (including D11's permanent-until-clicked ones), which are a different UX primitive with a clear user-owned dismissal affordance. Only ephemeral or user-dismissable toasts (D3). **Why:** matches parent §6 J6 "no nag" invariant; P1 persona discovers new version on next quit; power users who care can check About box. **Source:** intake §4 batch, user-confirmed.

### D4 (DIRECTED, 2026-04-21). Dev-mode dry-run = three-tier validation

- **Tier 1 (unit):** `packages/desktop/tests/integration/auto-updater.test.ts` — `AppUpdater` event-stub subclass; drives `this.emit(...)` to assert IPC shape + error-class routing. Fast CI feedback.
- **Tier 2 (integration, in-scope):** `packages/desktop/scripts/smoke-mock-update.mjs` — local HTTP server serving a hand-crafted `latest-mac.yml` + fake `.zip` with valid sha512. Exercises `GenericProvider` + download pipe + `update-downloaded` dispatch. `autoUpdater.forceDevUpdateConfig = true` + `setFeedURL('http://localhost:…')` bypasses the `app.isPackaged` gate.
- **Tier 3 (end-state, post-creds):** `packages/desktop/build/dev-app-update.yml` + `forceDevUpdateConfig = true`. Canonical approach per electron.build/auto-update. Exercises real `GitHubProvider` URL resolution + real staged pre-release tag. Cannot run until signed DMGs exist.

**Why:** evidence/electron-updater-api.md §4 recommends exactly this layering. Tier 2 does NOT exercise `GitHubProvider`'s URL resolution (uses `GenericProvider`) — Tier 3 covers that gap after creds procurement. **Source:** decision batch #1 item 6, user-confirmed ("go with rec").

### D5 (DIRECTED, 2026-04-21). Error-routing policy = classified → silent retry + structured log; unclassified → same

```ts
if (err.code?.startsWith('ERR_UPDATER_') || err.code?.startsWith('HTTP_ERROR_')) {
  // 13 known codes per evidence/electron-updater-api.md §2 — structured log with code, message, timestamp
} else {
  // Squirrel.Mac native errors (signature / notarization / FS) forward as bare Error with no .code
  // structured log with full stack; same silent-retry behavior
}
```

Zero per-error toast, zero dialog for any error class in M3. **D12 closes the 7-calendar-day escape hatch** (Toast C) so the failure is no longer terminal-per-installation — users see a one-time hint pointing at manual re-download after a week of silent failure. Parent §7 J7a explicitly accepts this. Promote to a limited "update unavailable" toast if FW5 (Sentry telemetry) shows persistent failure clusters on real users. **Source:** decision batch #1 item 7, user-confirmed ("continue with no toast in that situation").

### D12 (DIRECTED, 2026-04-21, post-audit). Stuck-update escape hatch = one-time-ever hint after 7 consecutive failed days

Middle path between D5's silent-retry-everything and a per-error toast (which would violate parent J6 "no nag"). **Mechanism:** state-store tracks `lastSuccessfulCheckAt: string | null` + `stuckHintShown: boolean`. On every update-check outcome, update `lastSuccessfulCheckAt` (when `checking-for-update` or `update-available` or `update-not-available` fires successfully) or leave unchanged (when `error` fires). Each time main evaluates: if `now - lastSuccessfulCheckAt > 7 days` AND `!stuckHintShown` → fire IPC `ok:update:stuck-hint` to renderer → Toast C: `"This app may not be receiving updates. Visit [inkeep.com/open-knowledge/download] to install a fresh copy."` with `shell.openExternal` link affordance. Set `stuckHintShown = true` — fires once ever, per installation. Resets on any successful check (user downloads fresh copy, or update cycle resumes, and counter clears automatically). **Why:** closes parent J7a's orphaned "manual-fallback" clause (`"user can re-download DMG from website"`) by giving users a signal + remediation path. Does NOT violate D2 "no nag" — Toast C fires at most once per installation. **Accepted cost:** 7 calendar days of silent failure before signal reaches user; if pairs with FW5 Sentry telemetry, Inkeep sees failures faster than users do. **Source:** C3 challenger raised the middle path; user-confirmed post-audit ("3b").

### D8 (DIRECTED, 2026-04-21). Release trigger = new tag-triggered `.github/workflows/desktop-release.yml`

Fires on `on: release: types: [published]` — the Release event is emitted by `release.yml`'s `gh release create "v${VERSION}"` regardless of which token created it (F14 audit revision; more robust than the original `push.tags: ['v*']` design because it does not depend on App-token-vs-GITHUB\_TOKEN push semantics). Runs the signed DMG+ZIP build on `macos-14`, then uploads `.dmg` / `.dmg.blockmap` / `.zip` / `.zip.blockmap` / `latest-mac.yml` to the already-existing Release (the `release: published` trigger guarantees the Release exists — no wait-for-release polling needed). Does NOT modify `release.yml` (keeps npm publish path insulated from macOS runner cost and M3-specific failure modes). ### D9 (DIRECTED, 2026-04-21). "What's new" source-of-truth = bare version string for v1; GitHub-Release-body fetch is Future Work

Toast B ships as `"Updated to v${VERSION} — see what's new"` with a link affordance that calls `shell.openExternal('https://github.com/inkeep/open-knowledge/releases/tag/v${VERSION}')` (uses D47 scheme allowlist). No GitHub REST call at app boot for the first release. **Promote path (FW1a):** after the first release validates the update flow in real users' hands, replace the bare string with a GitHub Release body fetch (authenticated or unauthenticated — within 60/hr/IP unauth limit at realistic usage). Promote trigger: first successful user-hands silent upgrade confirmed. **Why:** bare-string-plus-link is trivially simple, zero network dependency at boot, and the link is the "rich What's new" escape hatch for users who want details; deferring (a) avoids shipping a REST dependency we don't yet need to prove. **Source:** decision batch #1 item 3, user-confirmed ("d for first release then a").

### D10 (DIRECTED, 2026-04-21, revised post-audit). Update-check cadence = launch + periodic every 1 hour (matches Obsidian)

`checkForUpdatesAndNotify()` at the end of the `app.whenReady().then(...)` handler (after the first window is requested, regardless of which branch in `main/index.ts:405-421` ran), then every 1 hour via `setInterval(…, 60 * 60 * 1000)`. **Interval rationale (revised per C2 challenger):** Parent §6 J6 cites Obsidian as the UX model. Obsidian's update cadence is hourly per the Obsidian team on their forum (`forum.obsidian.md/t/frequency-of-up-to-date-checks-is-insane/15170`). Matching the parent's cited UX model removes the "outlier" critique from the original 6h proposal. Cost delta is noise: `checkForUpdatesAndNotify()` is a HEAD for `latest-mac.yml` metadata, not a full download; 24 vs 4 checks/day is negligible network cost. CDN path per `evidence/electron-updater-api.md` §3 is not rate-limited for public repos. Interval cleared on `app.on('will-quit')` (parent D40 canonical ordering — F17 audit). One updater instance per app launch (singleton, not per-window). **Source:** decision batch #1 item 4, user-confirmed ("B"); C2 challenger surfaced 1h as the principled interval matching parent's Obsidian UX model; user-confirmed post-audit ("2a").

### D11 (DIRECTED, 2026-04-21). Toast behavior = permanent-until-clicked

Both toasts use sonner's `duration: Infinity`. User dismisses by clicking the toast or the built-in dismiss button. Toast A (`"Update downloaded"` + "Relaunch now" action per D3 revised) is shown exactly once per pending-update state (tracked via `versionPendingInstall` on `AppState` in `state-store.ts`; cleared when install completes). Toast B (canonical copy per D9: `"Updated to v${VERSION} — see what's new"`) is shown exactly once per version transition (tracked via `lastSeenVersion`). **Risk (R5):** if a new update downloads while the previous toast is still visible (user left app running through multiple releases), the toast does NOT auto-update to reflect the latest version — content stays as originally rendered. Mitigation: toast text is version-agnostic ("Update downloaded") rather than version-specific, so stale content is still accurate. **Source:** decision batch #1 item 5, user-confirmed ("perma until clicked pls").

### D6 (DIRECTED, 2026-04-21). `.zip` target added to `electron-builder.yml` as part of M3

Change `mac.target: [{ target: 'dmg', arch: ['universal'] }]` → `mac.target: [{ target: 'dmg', arch: ['universal'] }, { target: 'zip', arch: ['universal'] }]`. `afterPack`/`afterSign` hooks from M2 already iterate over `appOutDir`; no changes needed. M2 shipped `.dmg` only because auto-update was out of M2's scope (M2 NG4); M3 is the first consumer that needs `.zip`, so the scope delta lands here rather than in a separate M2-patch PR. **Coordination note (C7 challenger):** because M2 is `Implementation Complete — pending review`, the M3 PR touching `electron-builder.yml` should coordinate with M2's reviewer — flag the `.zip` addition in the PR description so M2's review context is preserved. ### D3 (DIRECTED, 2026-04-21, revised post-audit). Notification layer = three toasts total (A/B/C)

- Toast A — `"Update downloaded"` body + an explicit action button `"Relaunch now"` (sonner `action` prop) on `autoUpdater.on('update-downloaded')`. Clicking "Relaunch now" → IPC `ok:update:relaunch-now` → main calls `autoUpdater.quitAndInstall()` for immediate install (not waiting for user-initiated quit). User can still dismiss the toast without acting — install-on-quit remains active via `autoUpdater.autoInstallOnAppQuit = true`, so Cmd+Q / File→Quit still installs even if toast is dismissed.
- Toast B — canonical copy per D9: `"Updated to v${VERSION} — see what's new"` on first launch post-update (version-change detection at boot).
- Toast C (added per D12) — `"This app may not be receiving updates. Visit [inkeep.com/open-knowledge/download] to install a fresh copy."` Fires once ever, per installation, after 7 consecutive calendar days without a successful update check. Link affordance uses `shell.openExternal` via D47 scheme allowlist.

  All three toasts **superseded by D11** for duration behavior (`duration: Infinity`, permanent-until-clicked); D3's original "auto-dismiss after N seconds" language is historical. **Why:** satisfies both "no nag" and "signal that something happened" — closes the gap where a user who never quits the app for weeks would never notice an update had been staged. **Source:** intake §4 batch, user-confirmed with Q1 expansion (toasts added to In Scope).

## 10) Risks

- **R1.** Toast UX could leak beyond ephemeral — if the toast component shares state with a persistent notification center, the "no nag" invariant breaks silently. **Mitigation:** toast component is purpose-built for M3, not a reusable notification surface.
- **R2.** `latest-mac.yml` name/path mismatch between artifact-upload paths and Release-asset expectations. electron-updater fetches from `https://github.com/inkeep/open-knowledge/releases/latest/download/latest-mac.yml` — asset must be uploaded exactly there. **Mitigation:** smoke-test asset URLs before first real release.
- **R3.** Version-bootstrap (0.0.0 → 0.1.0) could cascade: if desktop `package.json` version is imported by other workspaces, the bump might need coordinated changes. **Mitigation:** grep confirms no importers of desktop's version.
- **R4.** Update-check cadence on app launch could race with the Navigator-window boot sequence — checking too early (before project is loaded) wastes a network call; too late (after BrowserWindow is gone) misses the window. **Mitigation:** trigger after `app.whenReady()` + `createNavigatorWindow()`; only once per launch + every 6h thereafter (D10).
- **R5.** Permanent-until-clicked toast (D11) can go stale across multiple downloads if the user leaves the app running through multiple releases — Toast A stays rendered as first seen even though a newer version is now the pending install. **Mitigation:** toast text is intentionally version-agnostic ("Update downloaded — quit to install") so stale content is still accurate; on install at quit, whatever version is currently pending gets installed (electron-updater replaces the pending package when a newer one downloads).
- **R6.** 1-hour periodic check (D10 revised) fires while the user is editing or on a metered connection. `checkForUpdatesAndNotify()` metadata HEAD is negligible (\~1-2 KB per check, \~24/day); full `.zip` download only triggers when a new version is actually published (rare event). **Mitigation (minimal for M3):** structured `[updater]` log documents each check; if real usage surfaces bandwidth complaints, promote an opt-out toggle (OQ8 FW) or metered-network suppression (new FW item).
- **R7.** Silent update failure goes undetected by the user for up to 7 calendar days before Toast C (D12) fires. A user with a broken update chain (cert rotation, notarization staple stale, filesystem corruption, persistent network block) is frozen on an old version with no user-visible signal until D12's 7-day timer elapses. **Mitigation:** D12 closes the fallback path for users; FW5 Sentry closes the detection path for Inkeep. Threshold is tunable — if real failure patterns emerge with faster signal needs, drop to 3 days.
