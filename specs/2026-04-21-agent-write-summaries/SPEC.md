## 1) Problem statement

**Situation.** [[Open Knowledge]]'s [[TimelinePanel]] renders shadow-repo commits per document, with author + doc-list attribution per row. MCP-driven agent writes (`write_document`, `edit_document`, `rename_document`, `rollback_to_version`) appear as rows like `Claude · foo.md`. The shadow-log line `ok-contributors:` already carries `{id, name, colorSeed, docs[]}` per writer, parsed by `parseContributors` and surfaced as `entry.contributors` on each `TimelineEntry`.

**Complication.** Rows convey *who* and *which doc* but not *what*. The 15-30s L2 debounce coalesces multiple agent writes into one shadow commit, so a single timeline row can hide 5 unrelated edits. To understand recent agent activity a reader must open each row's diff — a constant low-grade tax that compounds as MCP-mediated agent collaboration scales (the project itself uses it heavily for spec/report authoring). The tax also degrades the timeline's secondary use cases: pre-Save-Version auditing, "what just happened?" diagnosis, and (future) intent-aware undo.

**Resolution.** Add an optional `summary: string` parameter to all four MCP write tools. The server accumulates summaries into the existing `ok-contributors:` JSON line as an additive `summaries?: string[]` field (per D23 — shape simplified from prior `summaries?: Record<string, string[]>`) (no v-bump — additive parsing per `parseContributors` tolerance (cf. `ok-checkpoint-v1:` sibling-line prior art for the v-bump path if one is ever needed)). [[TimelinePanel]] `EntryRow` renders summaries as a bullet list under the row, falling back to the current doc-list rendering when no summaries exist. Multiple writes in a debounce window contribute multiple bullets to the same row.

## 2) Goals

- **G1.** Make agent edits scannable in the timeline — a reader can understand the intent of any commit row without opening the diff in ≥80% of cases (post-ship measurement).
- **G2.** Zero regression for legacy commits and summary-less writes — the timeline renders identically for any commit lacking the new `summaries` field.
- **G3.** Cheap for agents to comply — adding a summary costs ≤1 short string, no new tool calls, no API ergonomics changes for non-summary fields.
- **G4.** Forward-compatible storage — the schema admits future enrichment (per-bullet author identity, structured intent, hover preview content) without another migration.

## 3) Non-goals

- **\[NEVER] NG1.** Required-summary enforcement that fails the MCP write when summary is absent. The friction would punish trivial edits (typo fixes) and break backward compatibility with existing agent integrations. Tool-description nudge only.
- **\[NEVER] NG2.** Server-side auto-generation of summaries from diffs. The whole point is *agent intent*, not structural diff description. Diff descriptions are noise.
- **\[NEVER] NG3.** Storing summaries in any side-channel separate from the existing `ok-contributors:` line (e.g. a new commit trailer, sidecar file, Y.Map). Maintaining two attribution channels invites drift.
- **\[NOT NOW] NG4.** Per-bullet contributor identity in the timeline UI when multiple agents contribute to one commit. v1 keeps contributor info in the row header; bullets are flat. — Revisit if: multi-agent-per-commit becomes common (>10% of commits) AND user reports confusion about attribution.
- **\[NOT NOW] NG5.** Structured `{verb, target}` summary schema. v1 is free-form string with character cap. — Revisit if: post-ship measurement shows agents producing inconsistent summary styles that hurt scannability AND structured form would demonstrably help.
- **\[NOT NOW] NG6.** Searchable timeline summaries (filter by summary text). v1 just renders. — Revisit if: timeline accumulates >100 entries per doc and users report needing to find specific past edits.
- **\[NOT NOW] NG7.** Fixing the "History unavailable" error shown in the original screenshot. Out of scope per user direction. — Revisit when: this spec ships and the rendering layer needs the data.
- **\[NOT UNLESS] NG8.** Plumbing summaries into the parent-git commit message produced by `/api/save-version` and `/api/rollback`. — Only if: a stakeholder explicitly asks for summaries to leak into the user's project git history. Default is shadow-only.
- **\[NOT UNLESS] NG9.** Surfacing summaries on non-MCP write surfaces (browser WYSIWYG, source-mode CodeMirror). — Only if: human-authored writes prove to need a similar affordance. Humans have direct UI to express intent (commit messages on Save Version); agents do not.
- **\[NOT NOW] NG10.** Unifying `summary` with the existing ephemeral `ActivityEntry.description` field on Y.Map('activity'). Worldmodel F2: `description` is currently auto-derived using the format `"<verb> (${agentName}): <truncated>"` at `api-extension.ts:1095` (Added — handleAgentWrite), `:1180` (Added — handleAgentWriteMd), `:1715` (Patched — handleAgentPatch). Read by `TiptapEditor.tsx:319` for a substring heuristic (`includes('prepend')` to set flash position direction); the CodeMirror plugin (`agent-flash-source.ts`) ignores it. Neither consumer surfaces the description text to users today. v1 keeps the channels independent — presence is ephemeral, shadow `ok-contributors:` is durable. — Revisit if: a flash-tooltip or live-activity surface wants to display agent intent in real time AND the unification cost is justified by that consumer. See [[specs/2026-04-21-multi-agent-presence/SPEC]] Cluster A for the natural future consumer.
- **\[NOT NOW] NG11.** Carrying summaries from shadow `ok-contributors:` lines into project-git via the `/api/save-version` checkpoint commit body. Worldmodel F3: today the project-repo commit body is `userMessage ?? \`Checkpoint v$\{n}\`` (`api-extension.ts:1880-1900`); contributor lines do NOT aggregate into it. Summaries are shadow-local and do not cross to GitHub via the existing sync path. — Revisit if: a stakeholder explicitly wants summaries to leak into project git history (e.g. for code-review attribution alongside `Co-Authored-By: Claude\` trailers).
- **\[NEVER] NG12.** Attributing UI-driven (non-MCP) rollback or rename actions to any agent. Per D22: `handleRename` and `handleRollback` only call `recordContributor` when the body carries an explicit `agentId`. UI Restore button at `EditorPane.tsx:155` (and any future UI affordance for rename/rollback) stays anonymous in the timeline as today. Audit H2 + Challenger F3 verified that without this guard, every human Restore click would attribute to Claude via `extractAgentIdentity` defaults — that is the explicit failure mode this NG forbids.

## 4) Personas / consumers

- **P1 — Open Knowledge user with multi-agent workflow** (primary). A developer or knowledge worker editing docs alongside Claude Code, Cursor, or similar AI tools that drive MCP. Opens [[TimelinePanel]] to: (a) understand what an agent did during a recent session, (b) audit changes before promoting WIP to a Save Version, (c) diagnose "what just happened?" when a doc looks different than expected.
- **P2 — The agent itself** (the producer). Needs the tool ergonomics to be cheap. Tool description must clearly signal: *when* to summarize (every write, ideally), *how* to phrase (user-outcome language), *what happens* (renders in timeline) and *how it interacts with defaults* (rename/rollback have defaults; agent-provided summary replaces).
- **P3 — Next-turn agent reading prior history** (immediate via `exec`/`get_history`). Worldmodel finding F1: `packages/cli/src/content/shadow-log.ts:121` already calls `parseContributors`, and `packages/cli/src/content/enrichment.ts:160` exposes `history.contributors[]` on every `exec` listing/read response. The moment summaries land in the JSON shape, agents calling `exec("ls reports/")` or `read_document(...)` see prior summaries automatically — zero extra wiring. This pattern (agent reading prior agent's narrative before writing) is common in this repo's spec/report workflow. The future "agent activity dashboard" / "intent-aware undo" use cases are deferred but inherit the same data shape.

## 5) User journeys

### P1 — User reading the timeline (happy path)

1. User opens a doc in the editor.
2. User opens the Timeline panel (right-side Sheet via [[TimelinePanel]]).
3. Panel polls `/api/history?docName=X&limit=100` every 10s while open.
4. Each row shows: author name, relative timestamp, doc list, **and (new) bullet list of summaries** when present.
5. User scans bullets to understand recent agent activity without diff-opening.
6. **Aha moment:** "Oh, Claude restructured the auth section and added two examples — that's exactly what I wanted."

### P1 — Failure / recovery

- **Summary unreadable** (truncated, ambiguous, missing): user falls back to clicking the row to view the diff. Same as today.
- **No summaries in any rows** (legacy commits, agents not yet adopting): timeline renders exactly as today (doc-list line). Zero regression.

### P2 — Agent producing a summary (happy path)

1. Agent intends to make an edit (e.g. find/replace via `edit_document`).
2. Agent reads the tool description, sees: *"`summary` — One-line user-outcome description (≤50 chars). Optional but strongly recommended. Appears as a bullet in the document timeline."*
3. Agent passes `summary: "Fixed token-refresh race in middleware"`.
4. MCP tool forwards to `/api/agent-patch` with `summary` in the body.
5. API handler validates + truncates if needed, calls `recordContributor(docName, agentId, name, colorSeed, summary)`.
6. Tool response includes `{ok: true, summary: { value: "Fixed token-refresh race in middleware", truncatedFrom?: number } }`.
7. On the next L2 debounce flush, the contributor accumulator emits the summary in `summaries[]` (flat per-contributor array per D23) on the `ok-contributors:` line.

### P2 — Failure / recovery

- **Summary too long** (>50 chars): server truncates with `…`, returns `truncatedFrom: <originalLength>` in the response. Agent learns to self-limit on next call.
- **Server down**: write fails for transport reasons, summary never lands. No special handling — same failure mode as any MCP write.
- **Same agent makes 5 edits in 30s window**: all 5 summaries accumulate into one shadow commit's `ok-contributors:` line for that agent. Timeline row shows all 5 as bullets.

### Non-functional requirements

- **Performance:** No measurable change to MCP tool latency; truncation is O(string length), JSON serialization grows linearly with summary count. Shadow commit message size grows by ≤(50 chars × N summaries × M docs) per coalesced commit — for typical N=1-3, M=1-2, this is <500 bytes per commit, well within git's natural body capacity.
- **Reliability:** Backward-compatible parser must never throw on legacy commits; type guard silently drops malformed `summaries` array (per D27 deliberate divergence) rather than failing the whole contributor parse.
- **Security/privacy:** Summary text is agent-controlled and surfaces in the timeline UI verbatim. No XSS risk in the existing TimelinePanel rendering (React text nodes). No PII concerns beyond what agents would write into a doc body. No size DoS — capped at 50 chars × write count × debounce-window writes.
- **Operability:** Existing logging unchanged; consider adding a single counter `metric.summariesProvided` (post-ship) to track adoption.
- **Cost:** Token cost for agents is small (one short string per write). Storage cost is tiny. No infra cost.

## 8) Current state (how it works today)

- **Storage:** [[shadow repo]] at `.git/openknowledge/` (integrated mode). Per-writer WIP refs at `refs/wip/<branch>/<writer-id>` with commits authored by the writer identity. Commit message format: `WIP auto-save <iso-timestamp>\n\nok-contributors: <json>\n[ok-contributors: ...]`.
- **Attribution accumulator:** `pendingContributors: Map<agentId, ContributorEntry>` in `packages/server/src/contributor-tracker.ts`. `recordContributor(docName, agentId, displayName, colorSeed)` adds to the map after each agent-write API call. `swapContributors()` atomically drains; `formatContributorsFrom(snapshot)` serializes; `restoreContributors(snapshot)` merges back on commit failure (D16).
- **L2 debounce:** 15-30s after last L1 disk write, `commitToWipRef()` in `packages/server/src/persistence.ts:164` swaps the contributor map and emits one commit message containing all accumulated `ok-contributors:` lines.
- **Read path:** `GET /api/history?docName=X` → `getDocumentHistory` in `packages/server/src/timeline-query.ts` parses `git log` output, runs `parseContributors(body)` per entry, returns `TimelineEntry[]`.
- **Render:** `TimelinePanel.tsx:256-264` — `EntryRow` shows `allDocs = contributors.flatMap(c => c.docs)` as the second line, or falls back to `entry.message` when `allDocs.length === 0`.
- **Key constraint:** No existing UI surface for per-write intent. Awareness/`activity` Y.Map carries flash text (`Added (Claude): <first 50 chars of content>`) for live presence flash, but it's ephemeral and not persisted to the shadow log.

### Gaps discovered during workshop

- `handleRollback` (`api-extension.ts:2129`) does NOT call `recordContributor`. Rollback content lands in the L2 cycle as an anonymous server-attributed commit. To plumb a rollback summary, the handler must either start calling `recordContributor` (with what agent identity?) or use a different attribution path.
- `handleRename` (`api-extension.ts:2657`) similarly does NOT call `recordContributor`. The rename mutates many docs (the renamed doc + every doc with backlinks). All those mutations land in the L2 cycle without per-contributor attribution.
- `handleAgentWrite` (the legacy non-md endpoint at `api-extension.ts:1041`) IS used in the agent-sim script and tests; it should support summary for parity even though the MCP layer doesn't expose it.

## 9) Proposed solution (vertical slice)

### User experience / surfaces

- **TimelinePanel `EntryRow`:** new bulleted summary list rendered between the author header and the doc-list (or replacing the doc-list when summaries cover all touched docs — see OQ3). Uses existing typography scale (text-xs, text-foreground for bullets).
- **MCP tool descriptions:** updated for all four tools to include the summary param + behavior clearly. Tool descriptions are getting long; restructure deferred (NG4 / Future Work).
- **MCP tool response:** new `summary` field in structured response when summary was provided. Includes `truncatedFrom` hint if applicable.
- **API endpoints:** five endpoints (`agent-write`, `agent-write-md`, `agent-patch`, `rename`, `rollback`) accept optional `summary` body field.

### System design

```
Agent (MCP client)
  │
  ▼ tool call: write_document(docName, markdown, position, summary?)
MCP tool handler (packages/cli/src/mcp/tools/write-document.ts)
  │
  ▼ HTTP POST /api/agent-write-md  body: {…, summary?}
API handler (packages/server/src/api-extension.ts:handleAgentWriteMd)
  │ - extract summary (extractAgentIdentity helper extended OR new extractWriteMetadata)
  │ - validate type (string), truncate to 50 chars with '…' if needed
  │ - record `truncatedFrom` for response shape
  │
  ▼ recordContributor(docName, agentId, name, colorSeed, summary?)
contributor-tracker.ts
  │ - push summary into ContributorEntry.summaries[] (flat append per D23)
  │
  ▼ (15-30s later)  L2 debounce timer fires
persistence.ts:commitToWipRef
  │ - swapContributors() → snapshot
  │ - formatContributorsFrom(snapshot) emits ok-contributors line(s) with summaries[] (per D23)
  │ - commitWip(shadow, defaultWriter, contentRoot, message, branch)
  │
  ▼ git commit-tree
shadow repo: refs/wip/<branch>/<writer-id>
  │
  ▼ (UI poll: every 10s while TimelinePanel open)
GET /api/history?docName=X
  │
  ▼ getDocumentHistory → parseContributors(body) → TimelineEntry[]
TimelinePanel.tsx EntryRow
  │ - allSummaries = contributors.flatMap(c => c.summaries ?? [])
  ▼ render bullet list
```

### Data model

```ts
// packages/core/src/shadow-repo-layout.ts
export interface ShadowContributor {
  v?: number;
  id: string;
  name: string;
  colorSeed?: string;
  docs: string[];
  summaries?: string[];  // NEW per D23: flat per-contributor array, oldest first
}

// packages/server/src/contributor-tracker.ts
interface ContributorEntry {
  agentId: string;
  displayName: string;
  colorSeed: string;
  docs: Set<string>;
  summaries: string[];  // NEW per D23: flat array, append on each summarized write
}
```

Storage on the wire (one `ok-contributors:` line per contributor in the commit body):

```
ok-contributors: {"v":1,"id":"agent-abc","name":"Claude","colorSeed":"...","docs":["foo.md","bar.md"],"summaries":["Fixed typo","Added example","Scaffolded doc"]}
```

### API/transport

- Body field: `summary?: string` on five endpoints
- Response shape addition (when summary was provided):
  ```ts
  // Existing fields preserved
  summary: { value: string; truncatedFrom?: number }
  ```

### Auth/permissions

No new auth surface. Agents already authenticate via the existing MCP/HTTP boundary; summary inherits that.

### Enforcement point(s)

- **Truncation:** API boundary in `api-extension.ts` (single source of truth for the 50-char cap).
- **Type validation:** Zod schemas on MCP tools (200-char hard cap as transport-safety bound).
- **Backward compat:** `parseContributors` type guard (drop field if malformed, contributor still parses).

### Observability

- Increment counters `summariesProvided`, `summariesTruncated` in `metrics.ts`.
- No new log lines (existing `[agent-write*]` log lines unchanged).

#### Data flow shadow paths

- **nil:** no `summary` field on body → no `summaries` field on contributor → row renders as today (zero regression)
- **empty:** `summary: ""` → treated as absent (FR9/FR10 default applies for rename/rollback; no-op for write/patch)
- **wrong type:** `summary: 42` → API returns 400 invalid input
- **timeout:** transport timeout — same as today
- **conflict:** N/A — additive
- **partial failure:** L2 commit fails → `restoreContributors(snapshot)` already merges back including summaries (per existing failure-recovery pattern at `persistence.ts:182`)

#### Failure modes and handling

| Component               | Failure                                                 | Detection                                 | Recovery                                                                | User Impact                                             |
| ----------------------- | ------------------------------------------------------- | ----------------------------------------- | ----------------------------------------------------------------------- | ------------------------------------------------------- |
| API truncation          | summary not a string                                    | type check at body parse                  | 400 invalid input                                                       | Agent sees structured error, retries without summary    |
| Contributor accumulator | shadow commit fails after summary recorded              | existing failure path in `commitToWipRef` | `restoreContributors(snapshot)` merges summaries back into next attempt | None — transparent retry                                |

| TimelinePanel           | summaries field missing on contributor (legacy commits) | undefined check                           | fall back to existing doc-list rendering                                | None — zero regression                                  |

### Alternatives considered

- **(A) New commit trailer (e.g. `Summary: <text>`)** instead of extending `ok-contributors:`. Rejected: maintains parallel attribution channels, invites drift. Single source of truth (one JSON line per contributor) is simpler.
- **(B) Side-channel storage in Y.Map('activity')** instead of shadow repo. Rejected: Y.Map activity is ephemeral (presence flash); shadow repo is the durable attribution journal. Summary is durable intent — belongs in the journal.
- **(C) Auto-derive summary from diff** server-side. Rejected: defeats the goal (agent intent, not diff structure). Diff descriptions are noise.
- **(D) Required summary** on every write. Rejected: friction tax on trivial edits, breaks backward compat.
- **(E) Per-bullet contributor identity** in v1. Deferred to NG4. v1 keeps bullets flat, attribution in row header.
- **(F) Single rendering: comma-joined "Fixed typo; Added example; …"** instead of bulleted. Rejected by user in workshop in favor of bullets / newline separation.

## 15) Future Work

### Explored

- **Per-bullet contributor identity in multi-agent commits (NG4 / Q7)**
  - What we learned: When N agents contribute to one commit (rare today, possible as multi-agent collab grows), bullets are flat — header shows all contributors but a reader can't tell which agent wrote which bullet.
  - Recommended approach: extend `summariesByDoc` to `summariesByDocWithAuthor: Array<{doc: string, summary: string, agentId: string}>` OR keep the current shape but add a sibling `bulletAuthors: string[]` aligned by index.
  - Why not in scope now: rare today; v1 should ship without it to avoid premature schema growth.
  - Triggers to revisit: multi-agent-per-commit observed in >10% of commits AND user reports confusion about attribution.
  - Implementation sketch: parser extension + EntryRow renders authorDot per bullet alongside the bullet text.

- **Structured `{verb, target}` summary schema (NG5)**
  - What we learned: Free-form is simpler but invites inconsistency. Structured form would enable verb-icons in the timeline (✏️ edit, ➕ add, 🗑️ remove) and intent-aware undo.
  - Recommended approach: extend with optional `intent?: { verb: 'fix' | 'add' | 'rewrite' | 'remove' | 'rename' | 'restructure', target?: string }` alongside `summary` string.
  - Why not in scope now: Adds tool DX cost; unclear if free-form will produce inconsistent enough output to motivate.
  - Triggers to revisit: M3 qualitative shows users squinting to interpret bullets.
  - Implementation sketch: additive field on `ContributorEntry` + parse + render verb-icon in EntryRow.

- **Searchable timeline summaries (NG6)**
  - What we learned: As timeline accumulates, finding "the commit where Claude added the auth example" becomes a needle-in-haystack problem. Summaries are searchable text but TimelinePanel doesn't expose search today.
  - Recommended approach: client-side filter input above the entry list; matches against `entry.contributors[].summariesByDoc[*]` (and existing fields).
  - Why not in scope now: requires UI design + accessibility consideration; v1 ships rendering only.
  - Triggers to revisit: user reports timeline-as-archive is hard to search.

- **Save-Version commit message includes summary digest (NG8)**
  - What we learned: `/api/save-version` creates a parent-git commit that's user-visible in `git log`. Summaries from the WIP refs since the last save could be condensed into a digest in the save-version commit body.
  - Recommended approach: when saving, walk WIP refs since last save, collect all `summariesByDoc[*]` arrays, summarize as a multi-line commit body.
  - Why not in scope now: separate UX surface (git log) with different audience (project teammates, not the doc author); nuanced default behavior decisions.
  - Triggers to revisit: user explicitly asks for summaries to leak into project git history.

### Identified

- **Cluster A activity sidebar — live agent narrative consumer (NG10 follow-up)**
  - What we know: [[specs/2026-04-21-multi-agent-presence/SPEC]] proposes `AgentPresenceEntry` substrate on `__system__` awareness for live agent state. Cluster A "Activity sidebar" (NOT NOW in that spec) would consume agent intent in real time.
  - Why it matters: Today the only summary consumer is TimelinePanel (durable, opened on demand). A sidebar would surface "Claude is currently working on: <summary>" live. Natural second consumer.
  - What investigation is needed: Coordinate with the multi-agent-presence spec on whether `AgentFocusEntry` or `AgentPresenceEntry` should grow a `summary` field, and whether MCP tools should broadcast it to `__system__` awareness in addition to the shadow log.

- **`get_history` MCP tool first-class exposure**
  - What we know: `/api/history` is HTTP-only today; consumed by TimelinePanel poll. A first-class MCP tool would let next-turn agents query prior history with structured filtering (currently they get bulk via `exec` enrichment).
  - Why it matters: Once exposed, agents become the primary consumer of summaries (P3 promotes from "immediate via exec" to "first-class MCP user"). Reinforces structured-shape design choice.
  - What investigation is needed: Tool ergonomics (filter params, pagination), MCP description, scoping vs. existing `exec` overlap.

- **Description restructure across MCP tools (NG4 follow-up)**
  - What we know: Tool descriptions are >300 words each, growing as features accumulate. Token cost is real per tool call.
  - Why it matters: Higher description tokens means worse model attention to the *important* constraints; bloated descriptions are an anti-pattern.
  - What investigation is needed: Audit all MCP tool descriptions in this repo for redundancy; propose a shared "preamble" pattern or move static behavior to a docs link.

- **Per-write summary on browser/source-mode writes (NG9)**
  - What we know: Humans typing in WYSIWYG/source mode have direct UI affordance (Save Version commit message); agents do not.
  - Why it matters: Symmetry might be desirable if humans use timeline auditing as much as agent-driven workflows.
  - What investigation is needed: User research on whether human writes in TimelinePanel feel under-attributed.

- **Intent-aware undo** — UndoButton driven by summary intent rather than transaction origin.
