## 1) Problem statement

**Situation.** Open Knowledge ships today as `@inkeep/open-knowledge` on npm — a CLI + MCP stdio server + headless Hocuspocus collab server + bundled React SPA. The parent Electron desktop spec ([[specs/2026-04-11-electron-desktop-app/SPEC|2026-04-11-electron-desktop-app]], Draft, gating V0-20 in Andrew's "Later" bucket) locks the desktop direction: electron-builder + signed DMG + GitHub Releases + electron-updater + opt-in-only telemetry (NG3) + CLI coexistence (G5). A research pass just landed ([[reports/mastra-speakeasy-cli-install-recommendations/REPORT|mastra-speakeasy-cli-install-recommendations]]) surveying Mastra and Speakeasy across 9 dimensions and produced 8 ranked recommendations for the CLI's install UX, distribution, and telemetry posture. The companion PR (this session) also shipped the `ok` + `open-knowledge` dual-bin alias plus README install-path updates.

**Complication.** Those recommendations and the new `ok` alias live only in a research report's Conclusions section and in the shipped README — not in any durable decision record. Specifically:

- **Silent violations are possible.** A future contributor could add Mastra-style unconditional PostHog telemetry, or a postinstall-binary wrapper for the CLI, and no authoritative artifact says "don't." The research report's Conclusions section is non-authoritative — it's a Conclusions lens, not a spec.
- **Conflict with an older spec.** `specs/2026-04-08-cli-packaging/SPEC.md` §3 NG1 reads "**NEVER:** GUI/Electron packaging, Docker distribution" — which the parent Electron spec reverses. The parent spec flags "NEEDS RECONCILIATION — see Decision D1" but the reconciliation was never written.
- **Telemetry posture is locked at the posture level but undesigned at the implementation level.** Parent Electron spec NG3 commits to "opt-in-only, Obsidian model" but the full implementation (DO\_NOT\_TRACK honoring, `OK_TELEMETRY_DEBUG`, docs page, crash/usage split via `@sentry/electron` + PostHog) is not designed anywhere. When the trigger to add telemetry fires, whoever implements will re-derive from scratch.
- **Trigger-gated future work is informal.** Homebrew Cask for the desktop app (5/7 reference Electron apps ship this) and the `ok update` self-update subcommand are both "we could do it later" without triggers documented anywhere that would ungate activation.

**Resolution.** Codify the 8 research-derived recommendations plus the `ok` bin decision as a decision-codification spec. This spec is primarily a durable record — most items are LOCKED (already-shipped current state) or Future Work (pre-designed with triggers). It complements (does not duplicate) the parent Electron spec, reconciles the contradiction with `2026-04-08-cli-packaging` NG1, and pre-designs telemetry implementation so activation is mechanical when a concrete trigger fires.

## 2) Goals

- **G1** — Every one of the 8 research-derived recommendations is recorded as a numbered decision in §10 with resolution status (LOCKED / DIRECTED) and evidence pointer. A contributor reading this spec cold can tell which decisions are immutable vs which are trigger-gated Future Work.
- **G2** — The `2026-04-08-cli-packaging` §3 NG1 contradiction is resolved via a corrigendum breadcrumb (per the repo AGENTS.md post-ship corrigendum protocol) pointing to this spec's D3.
- **G3** — Trigger-gated Future Work items (Homebrew Cask for desktop, telemetry implementation, `ok update`, shell completions) have explicit, observable trigger conditions documented — not vague "later." Activation is mechanical, not judgment-laden.
- **G4** — NEVER / NOT NOW non-goals close the door on known anti-patterns from the D1-D9 evidence (postinstall-binary for CLI, account-gate install, email-only telemetry opt-out, npm-distributed desktop app, vendor CI Docker action).
- **G5** — Discoverability: README.md and CLAUDE.md contain pointers to this spec so a contributor making a CLI-packaging / install-UX / telemetry PR finds it before (not after) making design choices.

## 3) Non-goals

- **\[NEVER]** NG1: **Postinstall-binary distribution for the CLI.** The CLI is pure JS. Native deps are handled via the Pattern A / optionalDependencies + pure-JS fallback pattern already shipped in `2026-04-11-zero-config-bunx-packaging` for `@parcel/watcher`. Adding a postinstall script that downloads from a non-npm CDN reintroduces the supply-chain surface, offline brittleness, and ephemeral-install fragility that Pattern A eliminates (D8 evidence). Revisit if: The CLI acquires a native dep whose per-platform payload exceeds npm's practical tarball limit AND Pattern A tooling for that dep is unavailable.

- **\[NEVER]** NG2: **Account gate before install.** Open Knowledge is local-first (parent Electron G9: no outgoing network calls in default config except the auto-update check). Install must never require signup. Revisit: never — violating this breaks the parent Electron spec's G9 contract.

- **\[NEVER]** NG3: **Email-only telemetry opt-out.** If telemetry is ever added, opt-out must be programmatic (env var + CLI subcommand + `DO_NOT_TRACK=1`). Speakeasy's "email `info@speakeasy.com` to opt out" is the worst-case UX (D9 evidence). Revisit: never.

- **\[NEVER]** NG4: **Desktop app installable via npm for end users.** 0 of 7 reference Electron apps do this (D8 evidence). Desktop distribution is electron-builder → DMG/EXE/DEB + Homebrew Cask (see D13). The `@inkeep/open-knowledge` npm package remains the headless/dev surface only. Revisit: never — npm is wrong distribution for the P1 persona (docs author, no terminal).

- **\[NEVER]** NG5: **Re-litigate the `ok` alias naming choice.** The `ok` + `open-knowledge` dual-bin shipped in the companion PR. Alternatives (`okb`, `okm`, no alias) were evaluated in [[reports/cli-command-name-ok-okb/REPORT|cli-command-name-ok-okb]] with collision audit + [[reports/mastra-speakeasy-cli-install-recommendations/REPORT]] §D7 reinforcing the long-name→alias precedent. Revisit: never — removing `ok` breaks user muscle memory after adoption; adding a third bin has no driver. Note this NG forbids re-debating the *naming* choice; it does NOT forbid implementation mechanics like a future `ok config alias`-style user-configurable override.

- **\[NOT NOW]** NG6: **Vendor CI GitHub Action (`inkeep/setup-open-knowledge` or similar).** Open Knowledge is homogeneous Node/TS; `actions/setup-bun@v2` + standard turbo invocations is sufficient. Speakeasy's polyglot-toolchain Docker action is over-engineered for this shape (D6 evidence). Revisit: if open-knowledge adopts a polyglot toolchain (unlikely given the Electron+MCP+CRDT focus).

- **\[NOT NOW]** NG7: **`ok update` self-update subcommand.** Mastra ships none; the package-manager upgrade path (`npm i -g @inkeep/open-knowledge@latest` etc.) works. A `speakeasy update`-style subcommand is standard for Go-binary CLIs but redundant for an npm package (D4 evidence). Revisit if: User-reported confusion about how to upgrade, OR installed-version drift causes support tickets.

- **\[NOT NOW]** NG8: **Shell completions for the CLI (bash / zsh / fish).** open-knowledge CLI has \~13 top-level subcommands today (start, ui, init, mcp, status, stop, clean, preview, auth, clone, sync, push, pull — verified against `packages/cli/src/cli.ts`). Speakeasy ships completions and has a comparable subcommand surface, but the flag surface driving the tab-completion ROI is not yet here (most `ok` subcommands have ≤3 flags). Revisit if: the flag surface grows such that tab-completion materially reduces friction (heuristic: when a subcommand regularly uses 5+ flag combinations in typical workflows).

- **\[NOT UNLESS]** NG9: **`create-open-knowledge` scaffolder split.** Mastra ships both `mastra` (CLI) and `create-mastra` (scaffolder) to optimize for the discoverable `npm create mastra` one-liner. `ok init` works for the CLI AND the future Electron app's first-launch flow (parent Electron G4: auto-register MCP on first launch reuses `init` codepath). Only revisit if: A distinct onboarding flow emerges that `ok init` genuinely cannot absorb.

## 4) Personas / consumers

This spec's consumers are **contributors**, not end users. The spec is a decision record, not a feature spec.

- **P1 — Future contributor touching CLI packaging, install docs, or telemetry code (PRIMARY).**
  - **Role:** Engineer making a PR that modifies `packages/cli/`, `README.md` install paths, bin configuration, or adds/removes telemetry.
  - **Needs:** Know which decisions are LOCKED vs which are open, find the rationale, find the evidence, understand what trigger would unfreeze a NOT NOW item.
  - **Success:** Reads this spec before designing their change; aligns with its decisions OR proposes amendments via Decision Log rather than silently violating.

- **P2 — Reviewers of future PRs touching these surfaces (SECONDARY).**
  - **Role:** Engineer reviewing a PR that affects distribution / install UX / telemetry code.
  - **Needs:** Point the author at the relevant decision if their change conflicts; know when to escalate (LOCKED decision change = spec amendment needed).
  - **Success:** PR either aligns with the spec or explicitly amends it; no silent drift.

- **P3 — AI agents running `/ship`, `/spec`, or `/review` on this surface (SECONDARY).**
  - **Role:** Agent dispatched to implement a related task (e.g., add telemetry, add Homebrew Cask) or extend this spec.
  - **Needs:** Read §16 Agent Constraints to know SCOPE / EXCLUDE / STOP\_IF / ASK\_FIRST boundaries; read §15 Future Work to know which items are Explored (implementable from the sketch) vs Identified (need their own spec pass).

- **P4 — Indirect: CLI end users + future desktop end users.**
  - **Role:** Benefit from the spec only insofar as consistent install UX + documented commands + honored telemetry posture flows from the decisions here.
  - **Interaction:** None directly with this spec.

## 5) User journeys

Since the spec is decision-codifying, user journeys are contributor journeys.

### P1 Happy path — Contributor making a CLI-packaging PR

1. Contributor picks up a ticket to, e.g., "add a `--telemetry-endpoint` CLI flag for enterprise telemetry forwarding."
2. Before designing: runs `grep -rn "telemetry" specs/ reports/` OR reads CLAUDE.md's CLI-ops section; finds pointer to this spec.
3. Reads §10 D14 (telemetry implementation design). Realizes: (a) telemetry is NOT NOW by default; (b) when added, DO\_NOT\_TRACK + env var + subcommand are required; (c) crash/usage are separate consent axes.
4. Decides their change is actually a telemetry implementation (which triggers D14's "Future Work — Explored" status → promote to a spec amendment + implementation PR).
5. Submits amendment to this spec alongside the implementation PR. Reviewers have a durable reference.

### P1 Failure path — Silent violation

1. Contributor adds `postinstall` to `packages/cli/package.json` that downloads a binary from a CDN.
2. Did not read this spec; did not find the D5 LOCKED decision.
3. Reviewer catches it: "This violates `specs/2026-04-20-cli-distribution-and-install-ux/` §10 D5 (NEVER postinstall-binary for CLI). Amend the spec or reframe the PR."
4. Contributor either justifies + amends spec (with evidence) or withdraws the pattern.

### "Aha moment"

The spec's value materializes the first time it prevents a silent violation or the first time a Future Work trigger fires (e.g., V0-20 ships signed DMG → Homebrew Cask PR lands in \~1 day because D13 implementation sketch is pre-written).

### Debug experience

- **"Why is this decision locked?"** Every D-row in §10 has an Evidence column pointing to the research report's evidence file or a parent spec reference. The rationale is one click away.
- **"Did someone already propose this?"** `meta/_changelog.md` records every change to the spec with date + change.

## 9) Proposed solution (vertical slice)

### User experience / surfaces

- **No new end-user surface.** Spec is a contributor-facing decision record.
- **CLAUDE.md pointer** (minimal edit) — one line in the "related specs" or CLI-ops section referencing this spec.
- **Corrigendum breadcrumb** — one-line edit to `specs/2026-04-08-cli-packaging/SPEC.md` §3 per the repo's post-ship-corrigendum protocol.
- **Research report cross-link** — already present in [REPORT.md Related Research](../../reports/mastra-speakeasy-cli-install-recommendations/REPORT.md); verify bidirectional link at finalization.

### System design

- **Architecture overview:** N/A (no runtime system being designed).
- **Data model:** N/A.
- **Decision log as the artifact.** §10 is the payload.
- **Evidence references:** All evidence lives in the research report and the parent Electron spec. This spec cites; it does not duplicate.

### Alternatives considered

- **Option A — Amendments only (no new spec).** Edit parent Electron spec + `2026-04-08-cli-packaging` spec + CLAUDE.md in place. **Rejected** for three concrete reasons: (1) the 8 decisions fragment across 3 artifacts, erasing the single citation anchor a reviewer needs in a PR comment ("this violates spec X §Y" is cheaper than "this violates parent spec §Z and old spec §W and AGENTS.md STOP rule #N"); (2) the D14 telemetry implementation sketch is too large to embed in parent Electron spec without distorting its scope (Nick owns the parent spec's desktop focus; telemetry implementation is a cross-cutting concern with its own playbook surface); (3) the decision provenance fragments — a future contributor auditing "why is telemetry opt-in?" would have to trace across three artifacts to reconstruct the reasoning, vs one spec with inline evidence pointers. The challenger (design-challenge pass) flagged that this rejection reasoning was thin in the original draft; expanded here for durability.
- **Option B — Consolidate by superseding `2026-04-08-cli-packaging` entirely.** **Rejected:** the older spec covers other concerns (monorepo restructure, initial npm publish plumbing) that don't belong in this spec's scope. Reconciling §3 NG1 via corrigendum is cheaper and preserves the older spec's other decisions.
- **Option C — Narrower scope (just `ok` alias + reconciliation).** **Rejected by user** during scope confirmation — the user wants telemetry implementation design included as Future Work so it's ready when the trigger fires.
- **Chosen — Option D (proposed solution).** Full decision-codification spec: 4 LOCKED (D1-D4), 4 NEVER (D5-D8), 3 NOT NOW (D9-D11), 1 NOT UNLESS (D12), 2 Future Work Explored (D13-D14). Corrigendum breadcrumb to older spec. CLAUDE.md pointer. **Why chosen:** single source of truth for the 8 recommendations + the `ok` alias; complements (doesn't duplicate) parent Electron spec; reversible (can be collapsed into another spec later if maintenance cost materializes).

## 15) Future Work

### Explored

- **Homebrew Cask for the desktop app** — **D13**
  - **What we learned:** 6 of 7 reference Electron apps surveyed in [[reports/electron-desktop-app-operations-2025/REPORT]] §4 ship a vendor-maintained Cask (VS Code, Obsidian, Slack, Discord, Linear, Cursor); Claude Desktop ships a community-maintained cask. The submission process is a PR to `homebrew/homebrew-cask` with a `Casks/<name>.rb` file. `auto_updates true` tells `brew upgrade --cask` to SKIP the cask and let electron-updater handle in-app upgrade; it does NOT automate the cask formula's SHA256 updates that homebrew-cask requires on each new release. Those are a separate ongoing operational concern.
  - **Recommended approach:**
    - **Initial submission (one-time):** After V0-20 ships signed DMG with stable URL pattern, fork `homebrew/homebrew-cask`, add `Casks/open-knowledge.rb` with: URL template `https://github.com/inkeep/open-knowledge/releases/download/v#{version}/Open-Knowledge-#{version}.dmg`; SHA256 of current release; `auto_updates true`; `name "Open Knowledge"`; `desc "Local-first knowledge base with CRDT collaboration"`; `homepage "https://github.com/inkeep/open-knowledge"`; `uninstall quit: "com.inkeep.open-knowledge"`; app-bundle zap entries for `~/Library/Application Support/Open Knowledge/` + `~/Library/Caches/com.inkeep.open-knowledge/` + `~/Library/Preferences/com.inkeep.open-knowledge.plist`.
    - **Ongoing SHA256 bumps (every release):** Three options — pick one at activation time, don't pre-commit here.
      - **Option A (manual):** Maintainer runs a local script on each release to compute the new SHA256 and open a PR to `homebrew/homebrew-cask`. Lowest infra cost; highest per-release toil. Obsidian pattern.
      - **Option B (GitHub Actions automation):** Workflow triggered on `release` event publishes a PR to `homebrew/homebrew-cask` with updated SHA256 using a tool like [dawidd6/action-homebrew-bump-formula](https://github.com/dawidd6/action-homebrew-bump-formula) (casks supported via the `formula-cask` input) or [mislav/bump-homebrew-formula-action](https://github.com/mislav/bump-homebrew-formula-action). Best fit if releases are already GitHub Actions–driven (likely per parent Electron CI spec). \~30 min setup; near-zero ongoing toil.
      - **Option C (delegate to a dedicated tap):** Maintain `inkeep/homebrew-tap` as a private tap with automation; skip upstream `homebrew-cask` entirely. Faster release propagation; user install command changes to `brew install inkeep/tap/open-knowledge`. Worth it only if upstream `homebrew-cask` review turnaround becomes a blocker.
  - **Why not in scope now:** No DMG exists yet (V0-20 is "Later" bucket, gated on parent Electron spec promoting from Draft).
  - **Triggers to revisit:** V0-20 ships signed DMG AND GitHub Release URL pattern is stable (versioned, documented) AND the release-triggering CI workflow is decided (affects Option A vs B vs C choice). Expect initial submission activation within 7 days of those conditions; ongoing automation option picked at first release post-ship.
  - **Unresolved at spec time (to decide at activation):** which of Option A/B/C for SHA256 bumps. The spec records the options; the activation-time decision belongs with whoever owns the Electron release process (likely Nick + whoever inherits V0-20 after Andrew).

- **Telemetry implementation (desktop + CLI)** — **D14**
  - **What we learned:** Industry gold-standard is Turbo's three-opt-out-path + debug-mode + docs-page + data-exclusion-list pattern. VS Code's 4-level `telemetry.telemetryLevel` separates crash from usage consent. Mastra's implementation (unconditional PostHog, no docs page, single env var) is below industry bar — specifically worth *not* copying.
  - **Recommended approach:**
    - **Posture (from parent Electron NG3):** Opt-in only. First-launch consent prompt in the desktop app ("Help improve Open Knowledge by sharing anonymous usage data? \[Yes / No / Learn more]"). CLI first-run has no banner (CLIs should not banner by default — silent-until-asked is the cleaner UX).
    - **Three opt-out paths (enforce per D7):**
      - `ok telemetry enable | disable | status` subcommand (persistent; writes to `~/.open-knowledge/config.yml`)
      - `OPEN_KNOWLEDGE_TELEMETRY_DISABLED=1` env var (redundant when opt-in-default but useful for CI / "don't even ask" contexts)
      - `DO_NOT_TRACK=1` honored — Turbo is the only surveyed tool doing this; easy differentiator
    - **Debug mode:** `OPEN_KNOWLEDGE_TELEMETRY_DEBUG=1` prints payload to stderr without transmission.
    - **Docs page:** `/docs/telemetry` (fumadocs) with: data collection list (event names + fields), example payload JSON, exclusion list (env vars, file paths, file contents, logs, serialized errors, PII), data retention period, endpoints, contact.
    - **Crash / usage split (VS Code 4-level equivalent):**
      - Crash: `@sentry/electron` for desktop (bridges main + renderer via Electron's Crashpad); the CLI can optionally use `@sentry/node` or be crash-free.
      - Usage: separate pipeline — vendor TBD at activation time; explicitly NOT pre-selected by this spec. D9 evidence shows no single vendor is ecosystem-dominant (Mastra uses PostHog; Vercel/Turbo/Next.js self-host; VS Code uses Azure App Insights; Speakeasy vendor-hosts). **Vendor compatibility matrix against D14 posture constraints** (activation-time implementer uses this as research input, not a directive — all four clear the posture bar, pick based on cost + data-residency + team fit):
        - **PostHog (cloud or self-hosted, AGPL)** — honors programmatic opt-out via client-side toggle; supports self-hosting (EU data residency achievable); event schema is open; free tier 1M events/mo. Known caveat: Mastra uses it with hard-coded US-region key + no opt-out — don't copy Mastra's implementation; replicate VS Code's telemetry-level pattern with PostHog as the sink.
        - **Amplitude (hosted only)** — respect opt-out via `setOptOut(true)`; no self-host; strong analytics features; paid tiers. Compatible with the posture but introduces a vendor-hosted-US dependency EU users would need disclosed.
        - **Segment (pipeline / source-of-truth)** — honors opt-out via `reset()` + library-level disable; routes to any downstream (including PostHog / Amplitude / self-hosted). Best if the team wants to A/B vendors without touching client code. Adds a pipeline layer (cost + latency).
        - **Self-hosted (Plausible / Umami / Tinybird / InfluxDB)** — strongest data-residency story; highest ops cost. Homebrew uses InfluxDB self-hosted with 365-day retention as their published reference. Compatible only if ops capacity exists.
        - **Sentry (`@sentry/electron`) for CRASH pipeline ONLY** — crash pipeline (non-negotiable for an Electron app). Bridges main + renderer via Crashpad. Free tier 5K errors/mo, team $26/mo annual for 50K. Claude Desktop ships `@sentry/electron ^7.0.0` — precedent-aligned. Pair with any of the above for usage.
        - **Not compatible with posture (DO NOT USE):** vendors that don't honor programmatic opt-out (none of the above fall here, but verify at activation); vendors without published data-retention policies; vendors that require email-to-opt-out (this reinforces D7 NEVER).
      - Independent consent axes: settings UI toggle in desktop; subcommand-level control in CLI.
    - **Data exclusion list (from Next.js precedent):** event names + version + OS + CPU count + CI detection + anonymous install ID are in; env vars + file paths + file contents + logs + serialized errors + PII are out. Publish this list verbatim.
    - **Data retention:** State explicitly (e.g., 365 days, matching Homebrew's published retention).
    - **GDPR posture:** Opt-in default satisfies EU consent requirements. If usage vendor is US-hosted (PostHog US region), state it; offer EU-region alternative if vendor supports.
  - **Why not in scope now:** No concrete motivating use case. Adding telemetry in anticipation would be premature.
  - **Triggers to revisit:** Team decides to add telemetry for a named use case (e.g., measuring MCP-stdio adoption, crash baseline for Electron GA). When this trigger fires, amend this spec with a new D15+ entry or supersede with a telemetry-implementation spec.
  - **Implementation sketch:** Full design above. Estimated effort: \~1 week for the full playbook (subcommand + env + DO\_NOT\_TRACK + debug + docs page + Sentry integration + PostHog integration + settings UI + tests).

  - **Why it matters:** Reduces MCP setup friction for Claude Desktop users from "find `.mcp.json` and hand-edit" to "double-click `.mcpb`." Parent spec G4 already auto-registers MCP on desktop first-launch for users who install the Electron app, but `.mcpb` serves Claude Desktop users who want ONLY the MCP server without the Electron app.
  - **What investigation is needed:** Whether `ok mcp` fits cleanly into the `user_config` schema (config dir picker, content path); whether the bundle can co-exist with `ok` CLI install (single source of MCP server implementation); how distribution channels (`.mcpb` direct download, MCP registry) compare to the existing `ok init` flow.

  - **What investigation is needed:** Azure Trusted Signing (\~$120/yr per parent spec) setup for Windows; code-signing policy for Linux AppImage; distribution channel choices (Scoop vs WinGet vs Chocolatey for Windows — similar "wrapper of wrappers" question to D2/D8 for macOS). Parent Electron spec is the owner; this spec's D8 NEVER-npm-for-desktop-end-users generalizes to all three platforms.
  - **Trigger:** Parent Electron spec NG4 promotes from NOT NOW to In Scope.

### Noted

- **Corporate / air-gapped install guidance** — open-knowledge has no documented offline-install story for either the CLI or the desktop app. When a customer asks, this is the gap. For the CLI: `npm install` from a private registry or tarball cache works today (esbuild's Pattern A / `@parcel/watcher`'s optionalDep + chokidar fallback survive air-gap). For the desktop app: `.dmg` download + manual copy + auto-update disabled via electron-builder config (`publish: null` per-instance or `ELECTRON_UPDATER_DISABLE` env check).

If that resolves, a Bun-compiled single-file `ok` binary (\~57 MB, self-contained — no Node runtime needed) becomes viable for users who don't have Node installed. Serves a narrow but real slice of the P1 + corporate-air-gapped overlap. Related to "Corporate / air-gapped install guidance" above. Not revisiting until: Bun #29120 is closed AND corporate / air-gapped demand materializes.
