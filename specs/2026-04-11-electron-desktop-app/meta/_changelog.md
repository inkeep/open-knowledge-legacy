- **R2 landed (Hocuspocus flush + reconnect) — BOUNCED BACK CRITICAL FINDING.** Two verdicts + a bug:
  - **OQ-08 → D17:** Pre-existing data-loss bug in OK's current `createServer().destroy()` at `packages/server/src/standalone.ts:399-424`. `hocuspocus.flushPendingStores()` is fire-and-forget (returns `void`, not Promise) — verified against `@hocuspocus/server/src/Hocuspocus.ts:165-177` source. Additionally, L2 git-commit flush is called BEFORE L1 markdown flush (wrong order). Up to 2s of L1 writes lost on every graceful shutdown. **This would bite CLI SIGINT users today** — it's not a new desktop concern, but the desktop spec work surfaced it. Fix: implement `flushAllStoresAndWait()` helper using a one-shot `afterUnloadDocument` hook (mirrors `Server.destroy()`'s internal pattern), reorder `destroy()` to run L1 before L2. Concrete patched code in §8.3.1. Must ship before desktop launches. Filed as new risk **R1-a**.
  - **OQ-08 / OQ-11 → D19:** Renderer-side drain barrier `ProviderPool.flushAllProviders()` using `HocuspocusProvider.forceSync()` + `hasUnsyncedChanges === 0` event wait with 5s timeout. Required because server-side L1 flush can only persist what the server's Y.Doc already has — client-buffered updates need to be ACK'd first. Two-phase shutdown: renderer `shutdown:begin` → `flushAllProviders` → `shutdown:client-drained` IPC → main posts `shutdown` to utilityProcess → patched `destroy()` runs. Added to §8.4.1 + IPC channel inventory §8.5.
  - **OQ-11 → D18:** Crash recovery is commutative + lossless via y-protocols bidirectional SyncStep1/SyncStep2 handshake. Zero custom renderer-side code — `HocuspocusProvider.onOpen` fires on every WS open (including reconnects), `startSync()` sends SyncStep1, server replies SyncStep2+SyncStep1, client replies SyncStep2 → Yjs commutative merge preserves every local-only update. Main just detects exit, respawns utilityProcess, shows a "Reconnecting..." banner that auto-dismisses on `synced`. Add crash-loop guard (max 3 in 60s → hard fail dialog). J7g rewritten to reflect this.

- **2026-04-11 — D18 REVISED via cross-spec drift check.** While investigating open PRs during the concurrent `2026-04-11-server-destroy-flush-fix` spec's finalization pass, PR [#56 "Prevent whole-document duplication after server restart" (mike-inkeep)](https://github.com/inkeep/open-knowledge/pull/56) was found to **directly invalidate D18's "commutative + lossless y-protocols merge" claim**. PR #56's body documents the exact failure mode: *"the browser kept the existing HocuspocusProvider and Y.Doc in memory; the server restarted and reconstructed the document from markdown into a fresh Y.Doc; the client reconnected with the old CRDT state; Yjs merged both copies and duplicated the document. This was especially visible with repeated restarts, where the file could double again on each reconnect."* The root cause is that the client's stale Y.Doc and the server's disk-reloaded Y.Doc have different Yjs item identities for the same content — the merge is lossy, not lossless. PR #56's fix is client-side **provider recycling**: on clean disconnect with `synced && unsyncedChanges === 0`, the pool recycles the provider (fresh Y.Doc from server's disk state). Offline-edit preservation only kicks in when `unsyncedChanges > 0`. **Files updated:**
  - **D18 revised** from "LOCKED — zero-loss via commutative merge" to "REVISED 2026-04-11 — bounded-loss via provider recycling per PR #56." Data-loss budget is no longer zero; it's bounded by the L1 debounce window (2–10s of typing that was in the dead server's Y.Doc but not yet on disk).
  - **§6 J7g rewritten** to reflect the recycle-on-reconnect model. Removed the "commutative Yjs merge, lossless" phrasing. Explicit "this is a significant correction from earlier session framing" note inline so future readers see the reason for the revision.
  - **§11 OQ-11 resolution** updated to cite PR #56 alongside the R2 report and note the original R2 framing is superseded by PR #56's behavioral evidence.
  - **Lesson for the spec audit trail:** source-code reading of Hocuspocus internals (R2) correctly identified the y-protocols message flow but missed the Yjs item-identity mismatch that only surfaces behaviorally under server-restart-from-disk. PR #56's production debugging caught what code-reading didn't. Future specs that claim "lossless CRDT merge" should cross-check against behavioral evidence (PRs, incident reports) not just protocol source.
