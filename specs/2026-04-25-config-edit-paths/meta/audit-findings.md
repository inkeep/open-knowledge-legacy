## TL;DR

The release-pivot reframe is substantively coherent — the architectural shift from HTTP-centric to Hocuspocus + TypeScript API is internally consistent in §1–§5, §10 D39–D63, and §16. However, the spec carries significant **textual debt from the old draft that was not swept**:

1. The most critical defect is **§9 has duplicate sections**: post-pivot `### System design` / `#### Architecture overview` / `### Alternatives considered` / `#### Failure modes` etc. (lines 303-449) coexist with the **original HTTP-centric versions** (lines 466-573) that were supposed to be superseded. The §9 head note at line 268 only superseded §9.5-9.8, not the in-§9 duplication.
2. **D56 is missing from the Decision Log table** but referenced 4 times (FR-14b, FR-39, FR-37 implementation step, etc.). Its content appears to be smashed into the D63 row in line 1148, alongside ghost text from the original D38.
3. **Seven P0 NQs (NQ1, NQ2, NQ4, NQ8, NQ11, NQ12, NQ18) are still marked Open** — the orchestrator's claim that all P0 are RESOLVED with Q9 the only retained P2 is **contradicted** by the spec's own §11 table.
4. Multiple references to dropped concepts (`applyConfigPatch`, "Modal Settings UI", "Settings dialog", "POST /api/config/patch") survive in §7 metrics, §9 (both halves), §9.5–9.8, §15 Future Work, and §16 ASK_FIRST.

Factual code references are mostly accurate (10 of 11 spot-checked passed). The Zod registry / yaml@2 round-trip / persistence atomic-write claims hold up under verification.
