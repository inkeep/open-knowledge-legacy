## Findings

### Finding: Bun recommends `moduleResolution: "bundler"` which permits extensionless imports
**Confidence:** CONFIRMED
**Evidence:** https://bun.com/docs/typescript

Bun's recommended tsconfig.json (generated by `bun init`):

```json
{
  "compilerOptions": {
    "lib": ["ESNext"],
    "target": "ESNext",
    "module": "Preserve",
    "moduleDetection": "force",
    "jsx": "react-jsx",
    "allowJs": true,
    "types": ["bun"],
    "moduleResolution": "bundler",
    "allowImportingTsExtensions": true,
    "verbatimModuleSyntax": true,
    "noEmit": true,
    "strict": true,
    "skipLibCheck": true
  }
}
```

**Implications:** The `bundler` moduleResolution is what allows extensionless imports to pass type-checking. Switching to `node16` or `nodenext` would require `.js` extensions.

### Finding: `allowImportingTsExtensions` enables explicit `.ts` in import paths
**Confidence:** CONFIRMED
**Evidence:** https://bun.com/docs/typescript

From the documentation:

> Bun supports things like top-level await, JSX, and extensioned `.ts` imports, which TypeScript doesn't allow by default.

`allowImportingTsExtensions: true` is required to write `from "./foo.ts"` without TypeScript errors. It requires `noEmit: true` because TypeScript can't emit `.ts` import specifiers to JavaScript.

### Finding: `verbatimModuleSyntax` does not affect extension resolution
**Confidence:** CONFIRMED
**Evidence:** https://www.typescriptlang.org/tsconfig/verbatimModuleSyntax.html

`verbatimModuleSyntax` controls type-only import/export elision behavior — it enforces that `import type` is used for type-only imports. It has no effect on how module specifiers or extensions are resolved.
