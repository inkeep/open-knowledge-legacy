# Bun Module Resolution: File Extension Requirements in Import Paths

**Purpose:** Determine whether Bun requires file extensions in import paths, how its resolution algorithm works, and how TypeScript configuration and build tools interact with extension handling.

---

## Executive Summary

Bun does **not** require file extensions in import paths — they are optional. When an import path omits an extension (e.g., `from "./foo"`), Bun resolves it by checking a fixed sequence of extensions: `.tsx`, `.jsx`, `.ts`, `.mjs`, `.js`, `.cjs`, `.json`, then `index.*` variants. When an extension is present, Bun resolves only that exact file (with a special `.js` → `.ts` fallback for TypeScript ESM compatibility).

This differs from Node.js ESM, which **requires** explicit file extensions for relative imports. Bun deliberately follows the "bundler" convention rather than strict ESM spec.

With the recommended TypeScript config (`moduleResolution: "bundler"` + `allowImportingTsExtensions: true`), both extensionless and `.ts`-extensioned imports are valid at the type-checking level. Build tools like tsdown bundle source imports away entirely, so the choice of extension style in source code does not affect published output.

**Key Findings:**
- **Extensions are optional:** Bun resolves extensionless imports via a fixed probe sequence (`.tsx` first, `.json` last)
- **Both styles work:** `from "./foo"` and `from "./foo.ts"` are equivalent when targeting Bun with `moduleResolution: "bundler"`
- **Build tools are unaffected:** tsdown/esbuild resolve imports at build time — source extension style doesn't reach published output
- **Node.js is stricter:** Node.js ESM requires explicit extensions; Bun is deliberately more lenient

## Detailed Findings

### 1. Bun's Default Module Resolution

**Finding:** File extensions are optional in Bun import paths. Bun probes a fixed sequence of extensions when none is specified.

**Evidence:** [evidence/bun-resolution-algorithm.md](evidence/bun-resolution-algorithm.md)

When resolving `import { foo } from "./bar"`, Bun checks files in this order:

1. `./bar.tsx`
2. `./bar.jsx`
3. `./bar.ts`
4. `./bar.mjs`
5. `./bar.js`
6. `./bar.cjs`
7. `./bar.json`
8. `./bar/index.tsx` through `./bar/index.json` (same extension order)

When an extension **is** present (`from "./bar.ts"`), Bun only checks for that exact file. A special case: importing `"./bar.js"` will also check for `"./bar.ts"`, matching TypeScript's ESM convention where source files use `.js` extensions that map to `.ts` files.

**Implications:**
- Extensionless imports work out of the box — no configuration needed
- TypeScript files (`.ts`, `.tsx`) are checked **before** JavaScript files, so TypeScript source always wins in ambiguous cases
- The `.js` → `.ts` fallback means TypeScript's `node16`/`nodenext` `.js`-extension convention also works in Bun

**Confidence:** CONFIRMED

### 2. TypeScript Configuration Interaction

**Finding:** The recommended Bun tsconfig (`moduleResolution: "bundler"`) permits both extensionless and `.ts`-extensioned imports at the type-checking level.

**Evidence:** [evidence/typescript-config.md](evidence/typescript-config.md)

Three TypeScript settings control extension behavior:

| Setting | Value | Effect |
|---------|-------|--------|
| `moduleResolution` | `"bundler"` | Allows extensionless imports (unlike `node16`/`nodenext` which require `.js` extensions) |
| `allowImportingTsExtensions` | `true` | Permits `from "./foo.ts"` (TypeScript normally forbids `.ts` extensions in imports) |
| `verbatimModuleSyntax` | `true` | Enforces `import type` for type-only imports; does **not** affect extension resolution |

Bun's recommended tsconfig (generated by `bun init`) uses all three. With this configuration:
- `from "./foo"` — valid (bundler resolution)
- `from "./foo.ts"` — valid (allowImportingTsExtensions)
- `from "./foo.js"` — valid (TypeScript's standard ESM convention)

If `moduleResolution` were changed to `node16` or `nodenext`, extensionless imports would produce TypeScript error `TS2835` ("Relative import paths need explicit file extensions").

**Implications:**
- The `moduleResolution: "bundler"` setting is what makes extensionless imports work with TypeScript's type checker — Bun's runtime would resolve them regardless, but `tsc --noEmit` would fail without it
- `allowImportingTsExtensions` is only needed if you want to write `.ts` extensions explicitly; it requires `noEmit: true` (since TypeScript can't emit `.ts` imports to JavaScript)

**Confidence:** CONFIRMED

### 3. Build Tool Interaction (tsdown/esbuild)

**Finding:** tsdown bundles source imports at build time, so import extension style in source code does not affect published output.

**Evidence:** [evidence/build-tool-interaction.md](evidence/build-tool-interaction.md)

tsdown (built on esbuild) resolves all relative imports during bundling. The published output contains fully-resolved, bundled code — no relative import paths remain (except for external dependencies marked in `neverBundle`). This means:

- Source can use extensionless imports, `.ts` extensions, or `.js` extensions — all are resolved identically at build time
- The `outputExtension` config controls output file naming (`.js`, `.d.ts`), not source import resolution
- External dependencies (`@parcel/watcher`, `simple-git`) remain as bare specifiers — no extension issue

**Implications:**
- Import extension style is purely a source-code convention when using a bundler
- If the project ever switched to a non-bundling build (e.g., `tsc` with `noEmit: false`), extension choice would matter

**Decision triggers:**
- If migrating away from tsdown to a non-bundling transpiler, import extensions would need to be standardized to `.js` for Node.js ESM compatibility

**Confidence:** CONFIRMED

---

### 4. Comparison with Node.js ESM

**Finding:** Node.js ESM requires explicit file extensions for relative imports; Bun is deliberately more lenient.

**Evidence:** [evidence/nodejs-comparison.md](evidence/nodejs-comparison.md)

| Behavior | Bun | Node.js ESM |
|----------|-----|-------------|
| `from "./foo"` | Resolves via probe sequence | **Error:** ERR_MODULE_NOT_FOUND |
| `from "./foo.ts"` | Resolves directly | **Error:** unknown file extension ".ts" (without `--experimental-strip-types`) |
| `from "./foo.js"` | Resolves (+ `.ts` fallback) | Resolves (exact match only) |
| Resolution strategy | Bundler-style (lenient) | Strict ESM spec |

Bun's approach matches what bundlers (webpack, esbuild, Vite) do — probe for extensions. Node.js follows the browser ESM spec, which requires explicit extensions because there's no filesystem to probe.

**Implications:**
- Code written for Bun with extensionless imports won't run directly on Node.js ESM without a bundler or loader
- Code using `.js` extensions (TypeScript's `node16` convention) works on both Bun and Node.js

**Confidence:** CONFIRMED
