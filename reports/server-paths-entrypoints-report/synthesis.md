Synthesis
Reading all three together: the reports/server-paths/REPORT.md is the "reality map," the SPEC.md is the remediation, and the audit-findings.md is an honest self-audit that actually caught a real implementation bug (H1). The three cohere cleanly — the REPORT's §5 links to the spec, the spec's Problem Statement references the REPORT's entry-point enumeration, and the audit has been fully applied to the spec with disclosure of which findings were deferred (only L5, appropriately).

Where the final design diverged from our conversation — and why I think the divergences are correct
1. Kill-and-restart instead of hard-refuse-only (G4). I recommended hard-refuse in all mismatch directions. The spec goes further: newer-desktop-vs-older-lock gets a user-consented SIGTERM-and-restart path; only older-desktop refuses. I think this is better than what I proposed. The direction asymmetry is load-bearing (D4's state-integrity argument — killing a newer server to boot an older one could corrupt state the newer one already wrote), the consent dialog is a firebreak, and the kill-and-restart is using the desktop's own bundled server, which sidesteps the cross-binary codesign concern that was the entire reason we rejected re-exec. The "refuse-only" position I landed on was shaped by the codesign obstacle; once kill-and-restart routes around that, the UX argument clearly wins. D3 correctly scopes this as the user's proposal and credits it.

2. executablePath kept in the lock, but scoped to diagnostics. I explicitly pushed back on storing a launch vector in v1 ("shipping a field no one reads is how schemas rot"). The spec kept executablePath but bound it to a concrete consumer — describeLockHolder(lock) produces the "Open Knowledge CLI vX.Y" / "Open Knowledge desktop vX.Y" labels for dialog copy. That gives it a live consumer without creating a delegation surface. Acceptable. The audit's M4 finding — "G4 dialog copy assumes the holder is always a CLI" — is exactly what this field exists to fix.

3. state.json naming retained. My concern was the collision with desktop's state-store.ts and server's sync-engine.ts (sync-state.json). In practice the collision is weaker than I argued: desktop's state.json lives under Electron's app-support directory (per-user state), not inside .open-knowledge/. The spec's file is at <contentDir>/.open-knowledge/state.json, fully namespaced by parent dir. Reasonable hold.

What the audit caught that mattered
H1 (missing SIGTERM handler) was a real implementation gap, not a doc issue. packages/server/src/boot.ts has zero process.on('SIGTERM', …) registrations — the kill-and-restart flow as originally written would always time out. The audit correctly proposed both (a) add a graceful-shutdown handler to bootServer and (b) relax the kill-poll to process-death only. Both applied (§5 boot.ts row, §6.4 poll condition, AC A6). This is the kind of finding that justifies the whole audit pass.

H2 (rollout adoption) was a coherence bug that would have caused silent state-version lies at rollout. Writing stateSchemaVersion = current on an existing project with a pre-existing shadow repo would permanently encode "current binary authored all of this." Splitting into fresh-vs-adopt with a stateSchemaVersion: 0 sentinel is the right fix and AC B6 tests it.

M1 + M2 together straightened out an implementability bug in the return-type design — the tagged-union ReadLockResult with distinct 'incompatible' from 'stale' unblocks the desktop mismatch matrix.

Gaps I'd still flag
The audit is thorough but was conducted by the author, with acknowledged bias. A few things survived the pass that are worth naming:

1. Strict-equality protocol compatibility with no minCompatibleProtocol range. The spec chose strict protocolVersion equality. Every protocol bump forces every sibling process on the machine to restart in lockstep — editor MCPs, CLI ok start, desktop utilities, Vite-plugin dev servers. For a pre-release greenfield project this is fine; post-release it's a noticeable UX tax on every breaking change. A single minCompatibleProtocol: number field on the lock (server declares the oldest client it accepts) buys real headroom for almost no complexity. Retrofitting later means parsing two lock schemas. Worth a conscious Q-level decision; I don't see it raised.

2. ownerKind derived from path parsing rather than explicitly stored. describeLockHolder parses executablePath for .app, cli.mjs, /ok suffixes. This is brittle in unusual install layouts — custom prefixes, symlinked binaries, monorepo dev paths, bun-installed bins, future package managers. An explicit ownerKind: 'cli' | 'desktop' | 'dev' field on the lock would be strictly more robust; the parsing layer could fall back to it when executablePath doesn't match known patterns. Low severity, but it's one field.

3. The SIGTERM handler's shutdown-ordering hazard isn't explicitly specified. §12 flags "Hocuspocus SIGTERM drain time" as a verification item, but the real concurrency hazard is ordering: to avoid losing in-flight writes, the handler must close WS connections → drain pending y-doc updates → flush persistence → release lock → exit. Getting that sequence wrong races writes against disposal. AC A6 checks "lock file absent + exit invoked" but not write-flush completeness. Worth making the invariant explicit in §6 or as an AC refinement.

4. bun run dev is a fifth lock writer that the audit caught (M5), but I'd verify the Playwright fixture chain transitively. The audit notes the per-worker fixture writes through the Vite plugin → acquireServerLock, so fixing the plugin fixes both. But createTestServer() (integration harness) reuses createServer() directly, not bootServer() — so it doesn't go through the new assertCompatibleStateManifest path. Q3 in the spec flags this. Needs a conscious decision: either the test harness writes version fields too (scope expansion), or integration tests skip the compatibility gate (controlled opt-out via an env var or constructor option). The spec raises the question but doesn't answer it.

5. The interaction with M6 (/usr/local/bin/ok bundled-CLI shim) isn't cross-referenced for drift implications. If M6 writes /usr/local/bin/ok into editor MCP configs from the DMG, and that shim's version is "whichever DMG was installed last," there's still latent drift: DMG v1.5 installs the shim, user opens a project, the CLI server holds a lock at v1.5; user later installs DMG v2.0 which overwrites the shim; editor MCPs now launch v2.0 against a v1.5 server. The mismatch protocol catches this (G6 refuses), but the UX is "your editor MCP stops working until you restart the CLI server." Worth calling out explicitly — M6's shim is not a drift eliminator, just a drift-channeling mechanism that the handshake protocol catches.

Net judgment
The spec is in genuinely shippable shape. The decision log is complete, the AC groups cover the surface, the implementation order is coherent (six small PRs), and §12's "non-obvious things to verify" captures the right hazards. The audit's self-disclosed bias is the strongest reason to run a cold independent audit before PR A lands — specifically for the items above (strict-equality protocol policy, test-harness gate behavior, SIGTERM ordering).

The architectural center ("exactly one runtime owns each project at each moment; everyone else speaks compatibly or refuses loudly") held across all three rounds of the conversation and all three artifacts. The spec didn't drift from that invariant; it just added a humane escape hatch (kill-and-restart) that I had wrongly excluded, and picked the right direction-asymmetry to make that escape hatch safe.

Two concrete next actions I'd suggest:

Decide strict-equality vs minCompatibleProtocol now, before PR A. This is the one schema decision that's expensive to retrofit.
Cold-audit by an agent that was not in this conversation, specifically probing (a) the SIGTERM ordering invariant, (b) createTestServer() compatibility-gate behavior, (c) the M6 shim interaction. The current self-audit was high-quality but has acknowledged bias limits.
