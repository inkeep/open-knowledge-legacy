## Key Findings

### Page References
- Syntax: `[[page-name]]`
- Creates a bidirectional link from the current block to the referenced page
- Functions identically to MediaWiki-style wikilinks at the page level
- Tags (`#tag`) are equivalent to `[[tag]]` — they create the same page reference

### Block References
- Syntax: `((block-uuid))`
- Uses triple parentheses around a UUID (not double brackets)
- The UUID is an internal identifier generated by Logseq for each block
- When pasted, shows the main node content with the rest visible on hover
- Created via right-click > "Copy block ref" or typing `((`

### Block Embeds
- Syntax: `{{embed ((block-uuid))}}`
- Embeds the full block content inline (not just a reference)

### File Storage Format
- Logseq supports both **Markdown** and **Org-mode** syntax
- Markdown is the default; can be switched per-graph
- Data stored as local files on disk — plain markdown `.md` files
- A graph = a local directory of `.md` files
- Block UUIDs are stored as properties within the markdown (e.g., `id:: 64a1b2c3-...`)

### Parsing Engine
- Core parser: **mldoc** (OCaml-based, compiled to JS)
- Converts markdown/org-mode text into an Abstract Syntax Tree (AST)
- Source: [logseq/mldoc on GitHub](https://github.com/logseq/mldoc)

### Portability Concerns
- `[[page references]]` are wikilink-compatible (similar to Obsidian, MediaWiki)
- `((block-uuid))` references are **Logseq-specific** — not standard markdown, not recognized by any other tool
- Block UUIDs are opaque identifiers — they don't contain semantic information about the block content
- From 99rabbits.com: Block references are fragile because the UUIDs can change if blocks are edited in certain ways outside of Logseq
