## Findings

### 1. VitePress `themeConfig.sidebar` (navigation only; path-prefix-based, not folder-based)

**Confidence:** High (type file public; multiple doc pages converge).

**Config location:** `.vitepress/config.{ts,js,mjs}` — returned by `defineConfig({ themeConfig: { sidebar: ... } })`. Pure JS/TS config, no JSON layer.

**Schema (canonical — `types/default-theme.d.ts`, paraphrased from doc + deepwiki mirrors):**

```ts
export type Sidebar = SidebarItem[] | SidebarMulti

export interface SidebarMulti {
  [path: string]: SidebarItem[] | { items: SidebarItem[]; base: string }
}

export interface SidebarItem {
  text?: string
  link?: string
  items?: SidebarItem[]
  collapsed?: boolean        // undefined = non-collapsible; true/false = collapsible
  base?: string              // base path prepended to each child link
  docFooterText?: string
  rel?: string
  target?: string
}
```

**Path-matching primitive:** **URL-path prefix**, not glob and not filesystem folder. `SidebarMulti`'s keys (`'/guide/'`, `'/reference/'`) are URL-prefix keys: the currently displayed page's URL longest-prefix-matches into the table to pick which sidebar tree to render. `base` on a group prepends to child `link`s — it operates on routed URLs, not on disk paths.

**Precedence:** Longest URL prefix wins for multi-sidebar selection. Within a tree, order is explicit (arrays are rendered in order). No globs, so no glob-precedence problem.

**Merge with page frontmatter:** None. Page frontmatter (`title`, `outline`, `sidebar: false`, etc.) controls the current page's rendering; the `themeConfig.sidebar` controls the *tree structure*. A page can hide itself from the sidebar (`sidebar: false` in frontmatter) but cannot inject new tree nodes from frontmatter. **Config tree is the only source of tree structure.**

**Override / escape hatch:** Page-level `sidebar: false` frontmatter suppresses the sidebar on that page. No per-file way to "override the group label for just this folder" — relabeling requires editing config.

**Type-safety:** TS-native. `import { defineConfig } from 'vitepress'` gives full inference; no Zod.

**Consumer:** Navigation rendering only. No SEO/route-generation implications — routes are file-based from the source tree regardless of sidebar config.

**Evidence:** "The `Sidebar` type is defined as `SidebarItem[] | SidebarMulti`, and the `SidebarMulti` interface is structured as `{ [path: string]: SidebarItem[] | { items: SidebarItem[]; base: string } }`" — WebSearch snippet quoting `types/default-theme.d.ts`. `base` example: `sidebar: { '/guide/': [{ text: 'Guide', base: '/guide/', items: [{ text: 'One', link: 'one' }] }] }` resolves to `/guide/one`.

**Key observation for OK:** VitePress chose not to model "folders" at all. Everything is a URL-tree expressed as nested `{ text, link, items }` objects in config. The disk structure is irrelevant to the sidebar model.

### 2. Docusaurus `sidebars.js` + `_category_.json` (hybrid: central config + colocated file)

**Confidence:** High.

**Config location:** `sidebars.ts` / `sidebars.js` at site root. Plus **optional** `_category_.json` / `_category_.yml` colocated in each `docs/**/` folder when using `type: 'autogenerated'`.

**Schema (central config, from `packages/docusaurus-plugin-content-docs/src/sidebars/types.ts`, paraphrased):**

```ts
type SidebarItemConfig =
  | SidebarItemDocConfig             // { type: 'doc', id, label?, className?, customProps? }
  | SidebarItemRefConfig             // { type: 'ref', id }
  | SidebarItemLinkConfig            // { type: 'link', href, label, description?, className? }
  | SidebarItemCategoryConfig
  | SidebarItemAutogeneratedConfig
  | SidebarItemHtmlConfig            // { type: 'html', value, defaultStyle?, className? }
  | string                           // shorthand for { type: 'doc', id: '<string>' }

type SidebarItemCategoryConfig = {
  type: 'category'
  label: string
  items: SidebarItemConfig[]
  link?:
    | { type: 'doc'; id: string }
    | { type: 'generated-index'; slug?; title?; description?; keywords?; image? }
  collapsible?: boolean
  collapsed?: boolean
  className?: string
  customProps?: Record<string, unknown>
}

type SidebarItemAutogeneratedConfig = {
  type: 'autogenerated'
  dirName: string                    // e.g. 'tutorials/easy' — directory under docs/
}
```

**`_category_.json` schema (colocated; only read in autogenerated mode):**

```json
{
  "position": 2.5,
  "label": "Tutorial",
  "key": "unique-sidebar-item-key",
  "collapsible": true,
  "collapsed": false,
  "className": "red",
  "link": { "type": "generated-index", "title": "Tutorial overview" },
  "customProps": { "description": "..." }
}
```

**Path-matching primitive:** `dirName` (string, docs-relative directory path), expanded recursively. Filesystem folders → categories, files → doc entries. Not a glob; explicit relative path.

**Precedence:** In explicit sidebars, order is declarative (array order). In autogenerated mode, sorting is by (a) `sidebar_position` frontmatter or `position` in `_category_.json`, then (b) alphabetical file/folder name. `index.md` / `readme.md` in a folder is special-cased as that category's index document.

**Merge with page frontmatter:** Explicit **three-way merge** for categories: `label`, `className`, `key`, `position`, `customProps` **default to values from the linked doc's frontmatter** (`sidebar_label`, `sidebar_class_name`, `sidebar_key`, `sidebar_position`, `sidebar_custom_props`) when the category has a linked index doc. For plain doc items, page frontmatter (`sidebar_label`, `sidebar_position`) cascades into the sidebar automatically.

**Override / escape hatch:** Authors can opt a doc out of autogeneration by excluding the file from the autogenerated dir's tree (no frontmatter flag for this — structural only) or by listing it explicitly in `sidebars.js`. `sidebar_label: 'X'` in page frontmatter overrides the autogenerated label for a single file.

**Type-safety:** TS-first. `SidebarsConfig` type exported from `@docusaurus/plugin-content-docs`. `_category_.json` is validated at build time via a JSON Schema (Joi-based historically); runtime validation throws on malformed fields.

**Consumer:** Sidebar navigation rendering; also drives "previous / next" links between docs. `_category_.json`'s `link.type: 'generated-index'` additionally **generates a new route/page** (the category index), so the sibling file is consumed by routing, not just navigation.

**Evidence:** Source paths: `packages/docusaurus-plugin-content-docs/src/sidebars/types.ts` (types), `src/sidebars/normalization.ts` (normalizes user input), `src/sidebars/generator.ts` (autogenerated-mode filesystem walk). Full `_category_.json` example quoted above is the canonical doc sample on https://docusaurus.io/docs/sidebar/autogenerated.

**Key observation for OK:** Docusaurus's hybrid is the reference design for **"central config + per-folder override file."** The central `sidebars.js` has an `autogenerated` escape hatch that delegates to `_category_.json` siblings — which is the sibling-file pattern the *other* subagent covers. For OK's centralized path, Docusaurus's lesson is that the central config needs explicit-list **and** directory-autogenerate modes to scale; pure explicit listing does not work beyond ~50 pages.

### 3. Starlight `starlight({ sidebar: [...] })` (discriminated union; Zod-validated)

**Confidence:** High.

**Config location:** `astro.config.mjs` — passed as the `sidebar` option to the `starlight()` Astro integration.

**Schema (paraphrased from `packages/starlight/schemas/sidebar.ts` — Zod discriminated union; shape confirmed by docs pages):**

```ts
// Three item variants in a union:

// Link:
{ label: string; link: string; translations?: Record<string, string>;
  badge?: BadgeConfig | string; attrs?: HTMLAttributes }

// Manual group:
{ label: string; items: SidebarItem[]; collapsed?: boolean; translations?; badge?; }

// Autogenerated group:
{ label: string;
  autogenerate: { directory: string; collapsed?: boolean; attrs?: HTMLAttributes };
  collapsed?: boolean;                  // top-level: controls the parent group
  translations?; badge?;
}
```

**Path-matching primitive:** `autogenerate.directory` — a string path **relative to `src/content/docs/`**. Not a glob. Discovery walks the filesystem under that directory and produces nested groups from subfolder structure.

**Precedence:** Explicit-list order in config wins; within an autogenerated group, order is by `sidebar.order` frontmatter, then alphabetical by file id. `autogenerate.collapsed` applies to *nested* subgroups; top-level `collapsed` on the parent group controls its own state (so a parent can be expanded while children start collapsed).

**Merge with page frontmatter:** Heavy. Per-page frontmatter under `sidebar:` **customizes how that page appears** in an autogenerated group:

```yaml
sidebar:
  label: 'Custom Label'
  order: 1
  hidden: false
  badge: { text: 'New', variant: 'success' }
  attrs: { class: 'red' }
```

Config-level `label` / `badge` / `collapsed` apply to the **group** (category), not individual pages. Pages control themselves; the config controls the group-around-them. No true "cascade" where a folder-level rule defaults all child pages — unless you use the community `starlight-auto-sidebar` plugin or `_meta.json`-style sibling files (covered by the other subagent).

**Override / escape hatch:** `hidden: true` in page frontmatter removes a page from the autogenerated sidebar. `sidebar.order` overrides alphabetical ordering. No way to opt a page out of having config-level group styling applied.

**Type-safety:** Zod. Entire config object runs through Zod schemas at Astro build time. Invalid sidebar shapes fail the build with Zod error messages. Users get full TS inference in `astro.config.mjs` via the `StarlightUserConfig` type.

**Consumer:** Sidebar rendering; also drives the search index's section grouping and the previous/next page links.

**Evidence:** "autogenerated group: `{ label: 'Guides', autogenerate: { directory: 'guides', collapsed: true } }`" (docs, accessed 2026-04-16). "Badges can include `text`, `variant` (note/tip/danger/caution/success), and `class`." Zod discriminated union is inferred from schema error shape and doc ref to `docsSchema()` extensibility.

**Key observation for OK:** Starlight is the **closest analog to OK's technology stack** — Zod-based, cascades from a single central config, supports directory-autogenerate as a first-class citizen. The big gap: Starlight has no concept of a folder-level **frontmatter cascade** in the config — per-folder defaults (e.g., "all pages under /api get tag:api") need plugins or sibling files.

### 4. Astro Content Collections `defineCollection` (collection-level schema, not folder-level)

**Confidence:** High.

**Config location:** `src/content.config.ts` (Astro v5) or `src/content/config.ts` (earlier). TypeScript only.

**Schema:**

```ts
import { defineCollection, z } from 'astro:content'
import { glob, file } from 'astro/loaders'

const blog = defineCollection({
  loader: glob({ pattern: '**/*.{md,mdx}', base: './src/data/blog' }),
  schema: z.object({
    title: z.string(),
    pubDate: z.coerce.date(),
    updatedDate: z.coerce.date().optional(),
    tags: z.array(z.string()).default([]),
  }),
})

export const collections = { blog }
```

**Path-matching primitive:** **Glob** (via `micromatch` under the hood). `glob({ pattern, base })` or `file({ file })`. Pattern is relative to `base`. A single collection can span nested folders (`**/*.md`); different collections point at different `base` directories.

**Precedence:** Each collection has exactly one schema. If two collections matched the same file (not normally allowed because `base` partitions the filesystem), behavior is undefined. Per-entry override of generated `id`/`slug` via frontmatter `slug: 'custom'`.

**Merge with page frontmatter:** Schema **validates** frontmatter; it does not provide defaults across files unless each Zod field has `.default(...)`. Defaults from Zod fill in missing fields per-entry. No folder-level cascade: a file in `src/data/blog/subfolder/` gets the same schema as a file directly under `src/data/blog/`. **The schema is per-collection, not per-folder.**

**Override / escape hatch:** Frontmatter can override `slug`. No mechanism to opt a specific file out of its collection's schema short of moving it to a different collection.

**Type-safety:** Zod. Astro generates `.astro/types.d.ts` with full inference from each collection's `schema`. Query results are typed; `getCollection('blog')` returns entries typed to the blog schema.

**Consumer:** Validation + type-generation for `getCollection` queries. Does not drive navigation, routing, or sidebar. Pages/routes are separate concerns (file-based routing lives in `src/pages/`; content collections populate those pages).

**Known limitation:** "The current Astro setup does not support nested collection structures with different schemas per folder" (see Astro roadmap discussion #801). **Astro deliberately chose one-schema-per-collection** instead of folder-cascading schemas. v5's Content Layer (`loader`) unlocks some flexibility (multiple collections with different loaders/bases can overlap in the `src/` tree) but not true per-folder cascade.

**Evidence:** `docs.astro.build/en/guides/content-collections/` (accessed 2026-04-16); `glob({ pattern, base })` signature from `docs.astro.build/en/reference/content-loader-reference/`; withastro/roadmap#801 explicitly documents the per-folder schema gap.

**Key observation for OK:** Astro's model is **deliberate restraint** — one schema per collection, no folder cascade. If OK wanted the simplest mental model, "per-collection schema" maps to "per-top-level-folder schema." But the primary use case Astro serves is validation, not metadata authoring, so the tradeoff differs from OK's.

### 5. Mintlify `docs.json` navigation (flat file list inside groups; no folder concept)

**Confidence:** High.

**Config location:** `docs.json` at project root. JSON only (no TS/JS).

**Schema (from navigation docs):**

```json
{
  "navigation": {
    "tabs": [
      {
        "tab": "Guides",
        "icon": "book",
        "groups": [
          {
            "group": "Getting Started",
            "pages": [
              "quickstart",
              "introduction",
              {
                "group": "Editing",
                "icon": "pen-paintbrush",
                "pages": ["development", "web-editor"]
              }
            ]
          }
        ]
      }
    ]
  }
}
```

**Path-matching primitive:** **Explicit file-path list**, one string per page, extension stripped (`"some-folder/file-1"` maps to `some-folder/file-1.mdx`). Groups can be nested recursively. No globs, no autogenerate mode.

**Precedence:** Array order is rendering order. If the same path appears twice, both render (no dedup). Tabs > groups > pages > nested groups — strict nesting where each level holds one type of child.

**Merge with page frontmatter:** Page frontmatter (`title`, `description`, `icon`, `sidebarTitle`, etc.) populates page metadata. `docs.json` controls the *tree*; frontmatter controls the *page*. `sidebarTitle` in frontmatter overrides the `"pages"` string's display label.

**Override / escape hatch:** Add/remove pages from `docs.json`; there is no "this folder maps to this group" shorthand. Folders are **not** a primary concept — they are incidental to the filesystem. A single group can mix pages from `folder-a/`, `folder-b/`, and root.

**Type-safety:** JSON Schema published at `https://mintlify.com/docs.json` (`$schema` reference). Runtime-only validation at CI/build time; no TS inference (JSON-only config).

**Consumer:** Site navigation (sidebar + top tabs + anchors + dropdowns — all one unified recursive tree).

**Evidence:** Navigation docs (accessed 2026-04-16): "In `docs.json` there is no separation between tabs, groups, and pages — they are integrated into one large recursive structure." Mintlify blog "Refactoring mint.json into docs.json" explains the 2024 migration.

**Key observation for OK:** Mintlify **deliberately rejects folder-driven navigation**. Pages are grouped by authorial intent (`"groups"`), not by disk layout. This is the opposite end of the spectrum from Docusaurus autogenerate. Trade-off: Mintlify's approach scales poorly past a few hundred pages (every page is listed explicitly); benefit is total authorial control and the ability to reorder content without moving files.

### 6. Turborepo `turbo.json` + Biome `files.includes` (non-docs config-driven path rules)

#### Turborepo `turbo.json` (glob-array per task)

**Config location:** `turbo.json` at repo root; optional per-package `turbo.json` extends the root.
**Schema (summarized):**

```jsonc
{
  "$schema": "https://turbo.build/schema.json",
  "tasks": {
    "build": {
      "dependsOn": ["^build"],
      "inputs": ["$TURBO_DEFAULT$", "src/**", "!src/**/*.test.ts"],
      "outputs": ["dist/**", ".next/**", "!.next/cache/**"]
    },
    "test": { "outputs": [] },
    "dev": { "cache": false, "persistent": true }
  }
}
```

**Path-matching:** glob arrays; negated globs prefixed with `!` subtract. `dist/` is equivalent to `dist/**`. `$TURBO_DEFAULT$` is a sentinel meaning "Git-tracked files."
**Precedence:** Later negations subtract from earlier inclusions within the same array. Package-level `turbo.json` **extends** root's task definition (merging); can override any field.
**Consumer:** Cache key derivation; not navigation.
**Key lesson for OK:** Turborepo is the archetypal **glob-array with negation** model. It interoperates with `.gitignore` semantically but requires explicit opt-in via `!pattern`. Renamed `pipeline` → `tasks` in v2.

#### Biome `files.includes` (ordered negation; overrides with first-match)

**Config location:** `biome.json` / `biome.jsonc`.
**Schema (excerpt):**

```jsonc
{
  "files": {
    "includes": ["**", "!**/node_modules/**", "!dist/**"]
  },
  "overrides": [
    { "includes": ["**/*.test.ts"], "linter": { "rules": { ... } } },
    { "includes": ["packages/legacy/**"], "linter": { "enabled": false } }
  ]
}
```

**Path-matching:** Biome-dialect globs (superset of gitignore). `**` must come first when using `!` negations. Docs: "you should always specify `**` first to match all files and folders, otherwise the negated pattern will not match any files."
**Precedence:**
- **`files.includes`** — evaluated in order; last match wins (per Biome issue #6645: "the order of patterns matters, and the last one takes priority").
- **`overrides[]`** — evaluated in order; **first match wins** ("if a file can match three patterns, only the first one is used").
- **Filesystem-walker vs config-layer distinction:** at `files.includes` level, ignoring a folder means the walker never enters it. At `overrides[].includes` level, files are always read and the match only controls which config applies. This is the critical footgun Biome calls out in its docs.

**Merge:** `overrides` patches the base config layer. Deep merge per tool (linter rules, formatter options).
**Consumer:** Lint/format/assist tool selection per file.

**Key lesson for OK:** Biome is the cleanest articulation of **"two globbing layers with different semantics"** — one controls filesystem inclusion (walker), one controls config-layer application (overrides). Precedence direction differs between the layers (last-match for `includes`, first-match for `overrides`). This is a real footgun worth explicit documentation in any OK design.

### 7. Obsidian Bases (YAML `.base` files; query-over-folders, not config-of-folders)

**Confidence:** Medium (fast-moving — Bases is native as of 2026, replacing Dataview in many flows).

**Config location:** Per-`.base` file in the vault. Not a central config — each base is its own file. Included here because the model (YAML + `file.inFolder()` filter) is the closest to OK's YAML-native world.

**Schema (from `obsidian.md/help/bases/syntax`):**

```yaml
filters:
  and:
    - file.inFolder("Projects")
    - status != "archived"
    - file.hasTag("active")
views:
  - type: table
    name: Active projects
    order:
      - file.name
      - status
      - priority
    filters:
      file.inFolder("Projects/active")
```

**Path-matching:** First-class `file.inFolder("path")` predicate; also `file.path.startswith("Projects/")`, glob-style via `file.path` regex.
**Precedence:** View-level filter **intersects** base-level filter (conjunction). Explicit, no hidden merge.
**Merge with page frontmatter:** Reverse direction — bases *read* frontmatter properties as query inputs. Bases don't *set* properties on files. One-way projection.
**Consumer:** Projected table/board/card views in the Obsidian UI. Not navigation, not routing.

**Key lesson for OK:** Obsidian's design statement is "folders are just another property you can filter by." This is the **query-driven alternative** to folder-configured metadata — instead of "this folder has these defaults," you write a query that says "for every file in this folder, show it with these columns." Obsidian does not attempt to *inject defaults* into files from a central config.

## Cross-system synthesis

### Config location patterns

| System | Location | Format | Why |
| --- | --- | --- | --- |
| VitePress | `.vitepress/config.ts` | TS/JS | Full programmatic power; types ship with package |
| Docusaurus | `sidebars.ts` + `_category_.json` siblings | TS/JS + JSON | Central for explicit; JSON sibling for per-folder overrides in autogenerate |
| Starlight | `astro.config.mjs` | TS/JS | Integrated with Astro's existing config; Zod-validated |
| Astro Content | `src/content.config.ts` | TS only | Zod schemas compile-time-typed; one file defines all collections |
| Mintlify | `docs.json` | JSON | No JS/TS — writable by non-engineers, CI-validatable via JSON Schema |
| Turborepo | `turbo.json` | JSONC | JSON Schema publishable; per-package extension supported |
| Biome | `biome.json(c)` | JSONC | No JS — stable config for lint/format tool |
| Obsidian Bases | `*.base` | YAML | End-user editable; no build step |

**Two camps.** **Engineer-facing docs tools (VitePress / Starlight / Astro)** pick TS for schema-as-code + type inference. **Editor-facing / validation tools (Mintlify / Turborepo / Biome)** pick JSON(C) for IDE JSON-schema validation without a build step. Docusaurus straddles — central is TS/JS, per-folder is JSON (readable by non-engineers committing category labels).

### Path-matching primitives

| Primitive | Systems | Trade-off |
| --- | --- | --- |
| **Explicit link/page list** | VitePress, Mintlify, Docusaurus explicit mode | Total authorial control; doesn't scale past ~hundreds of pages |
| **Directory-autogenerate (named relative path)** | Docusaurus `dirName`, Starlight `autogenerate.directory` | Scales infinitely; requires sibling files (or frontmatter) for per-folder metadata |
| **Glob arrays with negation** | Astro `glob({ pattern })`, Turborepo `inputs`/`outputs`, Biome `files.includes` | Maximum expressiveness; has ordering-semantics footguns |
| **URL-prefix-keyed table** | VitePress `SidebarMulti` keys (`'/guide/'`) | Simple for multi-sidebar selection; doesn't describe tree shape |
| **Property-predicate queries** | Obsidian Bases `file.inFolder()` | Declarative and composable; requires a query engine |

**Key insight.** None of the docs-framework systems use pure **glob-arrays** for their sidebar config. VitePress/Mintlify use explicit lists; Docusaurus/Starlight use **named directories** (effectively a single-segment glob). Glob-arrays appear only where the config drives **file-walker inclusion** (Astro collections, Turborepo, Biome) rather than navigation structure. This is a meaningful divergence — explicit named directories are legible and performant; glob-arrays with negation are powerful but carry precedence footguns.

### Precedence models

| Model | Systems | Failure mode |
| --- | --- | --- |
| **Array order, explicit** | VitePress, Mintlify, Docusaurus explicit mode | Safest; no hidden rules |
| **Alphabetical + position-override** | Docusaurus autogenerate, Starlight autogenerate | Requires frontmatter or sibling file to override defaults |
| **First-match wins** | Biome `overrides`, Docusaurus sibling merge | Authors must order rules specific-first |
| **Last-match wins** | Biome `files.includes`, Turborepo inputs | Authors must order rules general-first |
| **Conjunction of filters** | Obsidian Bases (AND across layers) | Requires mental model of intersection, not override |

**Mixed-direction precedence is the most common footgun** — Biome ships both (first-match for `overrides`, last-match for `includes`) and documents the distinction explicitly. Any design for OK should pick one direction and stick to it.

### Merge with per-file frontmatter

| Direction | Systems | Behavior |
| --- | --- | --- |
| **Config wins over frontmatter** | None of the surveyed systems do this by default |
| **Frontmatter wins over config (override)** | Docusaurus (`sidebar_label` overrides autogenerated label), Starlight (`sidebar.label`), Astro (`slug`) | The dominant pattern |
| **Frontmatter populates defaults when config is silent** | Docusaurus `_category_.json` cascading from linked `index.md` frontmatter | Partial pattern |
| **No merge — config is structure, frontmatter is content** | VitePress, Mintlify | Clean separation |

**Takeaway:** when the config specifies a value, file frontmatter can either **override** (most systems) or **ignore** it (VitePress tree structure). No surveyed system makes the central config override per-file frontmatter silently — that would be too surprising. The open design question is whether a config *default* can be overridden by a per-file value **without restating the config key** on each file.

### Type-safety landscape

| System | Validation | Inference |
| --- | --- | --- |
| VitePress | TS types | Full TS inference |
| Docusaurus | TS + Joi (runtime) | Full TS inference; Joi validates `_category_.json` |
| Starlight | Zod | Full Zod-derived TS types |
| Astro Content | Zod | Full Zod-derived TS types; `.astro/types.d.ts` generated |
| Mintlify | JSON Schema | No TS (JSON config); IDE completion via `$schema` |
| Turborepo | JSON Schema | No TS; IDE completion via `$schema` |
| Biome | JSON Schema | No TS; IDE completion via `$schema` |
| Obsidian Bases | YAML (validated at load) | No TS (end-user format) |

**Zod is dominant in the Astro/Starlight axis** — unsurprising, since both are Astro-stack. **JSON Schema dominates for config files meant to be edited by non-engineers** (Mintlify, Turborepo, Biome).

## Implications for Open Knowledge

Open Knowledge already ships with:
- **YAML config** (`.open-knowledge/config.yml` at user + workspace levels)
- **Zod schema** (`packages/cli/src/config/schema.ts:3-13`) validating `content.include` / `content.exclude`
- **`picomatch`** glob engine (`packages/server/src/content-filter.ts:13,58`)
- Exclusion supersedes inclusion, gitignore-aware

This puts Open Knowledge architecturally between **Astro Content Collections** (Zod + collection-level rules) and **Biome** (YAML/JSON + glob-array files.includes). The idiomatic extension for folder-level frontmatter would be:

1. **YAML + Zod** (matches existing `config.yml` + `ConfigSchema` disciplines).
2. **Glob-array pattern match** (matches existing `content.include` / `content.exclude`; picomatch is already a dependency).
3. **Ordered rules with explicit first-match or last-match semantics** (Biome's explicit documentation of which direction is the only survivable pattern).
4. **Cascade direction: config provides defaults; per-file frontmatter overrides** (Docusaurus / Starlight / Astro consensus).
5. **Validation layer separate from application layer** (Zod validates shape; the frontmatter-merge step applies defaults at read time).
