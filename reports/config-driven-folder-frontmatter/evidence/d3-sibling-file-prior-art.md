## Findings

### 1. Fumadocs `meta.json`

**Confidence:** CONFIRMED (read from shipped source + shipped `.d.ts`)

**Filename convention:** `meta.json` (no leading underscore) placed at the folder root, alongside `index.mdx` and sibling pages. Supported formats in practice: `.json`, `.yaml`, `.ts`, `.tsx`, `.mdx` (via Fumadocs MDX collections — the `meta` collection defaults to the `meta.{json,yaml}` pattern; the Zod schema is agnostic to serialization).

**Schema (verbatim from `node_modules/fumadocs-mdx/dist/config/index.js:11-18`):**

```js
var metaSchema = z.object({
  title: z.string().optional(),
  pages: z.array(z.string()).optional(),
  description: z.string().optional(),
  root: z.boolean().optional(),
  defaultOpen: z.boolean().optional(),
  icon: z.string().optional()
});
```

Corresponding typed interface (`node_modules/fumadocs-core/dist/loader-DgL6G4CA.d.ts:12-19`):

```ts
interface MetaData {
    icon?: string | undefined;
    title?: string | undefined;
    root?: boolean | undefined;
    pages?: string[] | undefined;
    defaultOpen?: boolean | undefined;
    description?: string | undefined;
}
```

Every field is optional. `pages: string[]` is the key ordering primitive.

**`pages[]` entry grammar** (from `resolveFolderItem`, `source/index.js:335-386`):
- `"guide"` — reference a sibling page or folder by basename
- `"..."` (alphabetical) / `"z...a"` (reverse) — "splat the rest of the filesystem items here"
- `"...folder"` — extract (flatten) a specific sub-folder's children inline
- `"!file"` or `"!folder"` — exclude from the auto-assembled list
- `"---My Separator---"` or `"---[icon]My Separator---"` — inject a visual separator
- `"[icon][External Name](https://external)"` or `"external:[..]..."` — inject an external link

**Inheritance:** NOT inherited in the Hugo sense. Each folder's `meta.json` is independent — it drives only that folder's page-tree node. A nested folder without a `meta.json` falls back to filesystem order. There is no frontmatter-injection-to-descendants like Hugo's `cascade`.

**Merge with page frontmatter (`source/index.js:428-439`):**

```js
let name = meta?.data.title ?? index?.name;
// ...
let node = {
  type: "folder",
  name,
  icon: meta?.data.icon ?? index?.icon,
  root: meta?.data.root,
  defaultOpen: meta?.data.defaultOpen,
  description: meta?.data.description,
  index,
  children,
  // ...
};
```

Merge is **shallow** and **field-by-field for folder-node fields only**: `meta.title ?? index.frontmatter.title`, `meta.icon ?? index.icon`. Meaning: **`meta.json` wins over the folder's `index.mdx` frontmatter for folder-tree display** (title, icon, description). Individual child page frontmatter is never read by `meta.json` — it only affects that child's `Page` node.

**Consumer:** Single consumer — the `PageTreeBuilder` (`createPageTreeBuilder` in `source/index.js:255-296`) which produces a `Root` tree the Fumadocs UI renders as the sidebar/nav. No search, no frontmatter injection into page bodies.

**Root semantics:** `root: true` on a folder's `meta.json` marks it as a "docs root" (top-level section with its own page-tree). Affects multi-section docs where navbar has multiple independent trees. Interplay with the `index.mdx` at that folder is that when `isRoot`, the `index` page reference is NOT attached (line 406: `if (!isRoot && !visitedPaths.has(indexPath)) { index = this.file(indexPath); }`).

**Extension:** The schema is Zod and `fumadocs-mdx` exports `metaSchema` explicitly. Users extend via `metaSchema.extend({ custom: z.string() })` on the `meta` collection — the extended schema runs at build time and types propagate. Confirmed via Fumadocs docs on Collections. Storage remains JSON/YAML — build-time Zod validates.

**Notes/limitations (from discussions):**
- `root: true` does NOT "hide" other roots in the same tree — discussed in `fuma-nama/fumadocs#728`.
- Mixing `pages[]` with index-doc interaction has edge cases around whether the index doc itself appears in the list (it doesn't, by design — it's folded into the folder node).

### 2. Docusaurus `_category_.json` / `_category_.yml`

**Confidence:** CONFIRMED (from official docs, backed by multiple issues/discussions)

**Filename convention:** `_category_.json` OR `_category_.yml` (both supported, same schema, same folder).

**Schema (composed from docs and issues; no single upstream Zod/TS schema since Docusaurus uses Joi):**

```jsonc
{
  "label": "Tutorial - Basics",
  "position": 2,                    // supports floats (2.5)
  "className": "red-category",
  "collapsed": true,                // default true
  "collapsible": true,              // category expandable
  "description": "Category description",  // used by DocCard in generated-index
  "link": {                         // category-root behaviour
    "type": "doc",
    "id": "intro"                   // relative to containing dir
    // OR
    // "type": "generated-index",
    // "title": "Tutorial overview",
    // "slug": "/category/tutorial",
    // "description": "..."
    // OR
    // null  → opt out of auto index
  },
  "customProps": {                  // arbitrary passthrough
    "any": "shape"
  },
  "key": "unique-sidebar-key"
}
```

YAML equivalent:

```yaml
label: Tutorial - Basics
position: 2
link:
  type: generated-index
  title: Tutorial overview
```

**Inheritance:** NOT inherited. Strictly per-folder. A grandchild folder is entirely unaffected by an ancestor `_category_.json`.

**Merge with page frontmatter:** Docusaurus defines a **dual surface** — when a folder has `index.md` / `<dirname>.md` with sidebar-prefixed frontmatter, `_category_.json` and the index doc's frontmatter merge with specific precedence:

From docs: "By default, an index.mdx doc's `sidebar_label` and `sidebar_position` will be applied for directories. To override, you can add a `_category_.json`."

Frontmatter keys that participate:
- `sidebar_label` ↔ `label`
- `sidebar_position` ↔ `position`
- `sidebar_class_name` ↔ `className`
- `sidebar_custom_props` ↔ `customProps`
- `sidebar_key` ↔ `key`

**Precedence:** `_category_.json` > index-doc frontmatter > filesystem default. The merge is shallow and per-field.

**Page frontmatter on individual pages:** For a non-index page, `sidebar_label` / `sidebar_position` on the page's own frontmatter takes precedence over any position implied by filename prefix (`01-foo.md`) or by the sidebar item config. `_category_.json` does not affect pages inside the folder other than via ordering context.

**Ordering semantics:** Pages are ordered by (precedence, ascending):
1. explicit `sidebar_position` frontmatter
2. numeric filename prefix (`01-foo.md`)
3. `_category_.json.position` for folder-as-sidebar-item
4. alphabetical fallback

**Consumer:** `@docusaurus/plugin-content-docs` — builds the autogenerated sidebar. Not read by search, not injected into page frontmatter, not visible at render time except as the sidebar shape.

**Observed limitations:**
- Cannot set sidebar_class_name via category index metadata for certain edge cases (issue #10095).
- Difficult to hide the category itself while showing children (feature request on canny board).
- Description surface for generated-index cards has been a recurring feature request (issues #7818, #7598).

### 3. Nextra `_meta.{json,js,jsx,ts,tsx}`

**Confidence:** CONFIRMED (from official docs; schema described directly in Nextra docs + GitHub)

**Filename convention:** `_meta.js`, `_meta.ts`, `_meta.tsx`, `_meta.jsx`, or `_meta.json`. JS variants are preferred (supports JSX for separator content). Deprecated: `_meta.json.js`. A special `_meta.global.{js,ts}` at the root can declare the entire tree in one file.

**Schema (roughly, composed from Nextra 3/4 docs):**

```ts
// Nextra exposes this as a Record<string, string | Schema>
type MetaItem =
  | string                          // shorthand: just the display title
  | PageItemSchema
  | MenuItemSchema
  | SeparatorSchema;

type PageItemSchema = {
  type?: 'page' | 'doc';            // default 'doc'; 'page' hoists to navbar
  title?: string;
  display?: 'normal' | 'hidden' | 'children';  // default 'normal'
  theme?: Partial<ThemeConfig>;     // inherits to children if set on folder
  href?: string;                    // external link
  newWindow?: boolean;
  sidebarTitle?: string;
};

type MenuItemSchema = {
  type: 'menu';
  title: string;
  items: Record<string, MenuItemSchema | string>;
};

type SeparatorSchema = {
  type: 'separator';
  title?: string | ReactNode;        // JSX allowed in .tsx variants
};
```

Key constraint from docs: "The type of _meta keys should always be a string and not a number because numbers are always ordered first in JavaScript objects."

**Inheritance:** Partial. The `theme` field is the documented inheritable field — "This option will be inherited by all child pages if set to a folder." Other fields (`title`, `display`, `type`) are per-entry and do NOT cascade.

**Merge with page frontmatter:**
- Page order: the key order in `_meta` dictates sidebar order; any file not listed is appended (or hidden, configurable).
- Page title: `_meta` entry title overrides frontmatter `title` for sidebar display, but `sidebarTitle` in frontmatter can override it again for the sidebar-specific label.
- `display: 'hidden'` hides from sidebar but keeps route accessible.
- `display: 'children'` — shows children in parent's place (folder is transparent).

Precedence for sidebar label: `_meta` value < frontmatter `sidebarTitle` < explicit sidebar config. For content title: frontmatter `title` wins.

**Ordering semantics:** Strictly the key order in `_meta`. Files not declared in `_meta` still appear (at the end) unless hidden. There's no `...rest`-style sentinel like Fumadocs — position is solely by object key order.

**Consumer:** Nextra builds a `pageMap` (exposed via `nextra/page-map`). Functions `normalizePageMap` and `mergeMetaWithPageMap` are the documented merge primitives. Consumed by the navigation/sidebar components.

**Observed limitations:**
- Object-key ordering is fragile when keys are numeric strings (JS re-orders integer-like keys).
- Pre-v3 only JSON was supported; v3+ added `.js/.ts` for JSX separators — this caused churn during migration.
- Dynamic routes require careful `_meta` handling (issue #4814).

### 4. Hugo `_index.md` with `cascade`

**Confidence:** CONFIRMED (from official docs; `cascade` has been Hugo's canonical inheritance mechanism since v0.76.0, Sept 2020)

**Filename convention:** `_index.md` (not `index.md`) at a section directory root. Branch bundle = directory with `_index.md`; leaf bundle = directory with `index.md`. The difference is structural:
- `_index.md` = section, has descendants (other branches, leaves, pages)
- `index.md` = leaf, terminal, no descendants as pages

**Cascade front matter (verbatim example from Hugo docs):**

```yaml
---
title: Books
cascade:
  - params:
      banner: images/books.jpg
    target:
      kind: page
      path: /books/**
  - params:
      color: blue
    target:
      environment: production
      kind: page
      path: /films/**
---
```

Or single-map form without `target`:

```yaml
---
title: Posts
cascade:
  params:
    banner: default-banner.jpg
  sitemap:
    priority: 0.8
---
```

**Schema (shape):**

```ts
type CascadeBlock = {
  // Any front-matter field (title, params, sitemap, layout, etc.) except
  // 'cascade' itself (prevents recursive cascade).
  [frontMatterKey: string]: unknown;
  // Optional targeting
  target?: {
    path?: string;         // glob against logical path, e.g. "/books/**"
    kind?: string;         // glob against page kind, e.g. "{home,section}"
    lang?: string;         // glob against language code, e.g. "{en,de}"
    environment?: string;  // glob against Hugo build environment
  };
};

type CascadeField = CascadeBlock | CascadeBlock[];
```

(Note: `_target` is the deprecated alias for `target` — removal planned.)

**Inheritance:** TRUE deep cascade. From the docs: "values are passed down to the page's descendants unless overwritten by self or a closer ancestor's cascade."

Key rules:
- Cascade flows to ALL descendants (arbitrary depth), not just immediate children.
- A descendant's own explicit frontmatter field overrides the cascaded value.
- A closer ancestor's cascade overrides a farther ancestor's cascade for the same field.
- Intermediate sections without their own `_index.md` do NOT block cascade — it flows past them to descendant leaves.
- Cascade is single-field-level: if you cascade `params: { banner: x }` and a descendant sets `params: { author: y }`, the result depends on Hugo's param merging (which is a **merge**, not a replace, for maps; arrays replace).

**Merge with page frontmatter:** Descendant frontmatter takes precedence over cascaded values, per field. For map-valued fields (`params`), maps merge recursively; leaf values in descendant win. For array-valued fields, descendant replaces (no concat).

**Consumer:** The Hugo template engine — cascaded fields are visible as ordinary frontmatter in `.Params`, `.Title`, etc. from templates. Search, RSS, sitemap, layouts all consume the effective (post-cascade) frontmatter.

**`target` targeting:** Lets a cascade block apply only to a subset of descendants. `path`, `kind`, `lang`, `environment` are all glob patterns. Multiple cascade blocks with different targets can coexist (array form).

**Site-level cascade:** Hugo 0.86+ allows `cascade` at the site `config.toml/yaml` level, which cascades to ALL pages. This is the "default defaults" layer.

**Observed limitations:**
- Cascade array merging from imported Hugo Modules has edge cases (issue #13869).
- Kind-target globs had pattern issues (issue #8888).
- When cascade changes, Hugo's incremental rebuild graph sometimes misses descendants (issue #6538).

### 5. Obsidian folder notes (plugin ecosystem)

**Confidence:** CONFIRMED — Obsidian core does NOT support folder-level frontmatter; plugin ecosystem fills the gap with varying semantics.

**Core Obsidian:** No first-class folder metadata. Folders are pure filesystem entities. All metadata lives in individual note frontmatter.

**Plugin landscape:**

**Folder Notes (LostPaul, most popular):**
- Filename convention: attaches a note to a folder, conventionally named `<folderName>.md` or `_folder.md` (configurable), living either inside or beside the folder.
- Purpose: makes clicking the folder open a note (Notion-style). Does NOT propagate frontmatter to children.
- Features reading folder note frontmatter: the plugin reads the folder note's frontmatter `title` / filename sync to display folder name, but this is for display aliasing — not child-inheritance.
- Inheritance semantics: **None.** The folder note is just a note that happens to be linked by convention to the folder display.

**alx-folder-note (aidenlx):**
- Similar filename convention, adds "add description, summary, and more info to folders via folder notes."
- No cascade to children. Folder note is a standalone doc.

**Waypoint:**
- Auto-generates dynamic MOCs (table-of-contents) inside folder notes.
- Integrates folders into graph view.
- Again, no metadata cascade — it's the opposite direction (folder note summarizes children, not vice versa).

**Metadata Menu (mdelobelle) — the closest Obsidian has to inheritance:**
- Defines `fileClass` — a named schema of fields that can be attached to notes.
- A fileClass can `extends` another fileClass (inherits its fields).
- Inheritance is **schema-level** (field definitions), not **value-level** (values). A note of fileClass `course` gets the fields its parent `fileClass` declares, but it still has to fill them in.
- Not folder-scoped — it's fileClass-scoped. You can attach a fileClass to a folder via plugin settings, but inheritance is by fileClass lineage, not by filesystem hierarchy.

**Summary:** Obsidian has NO folder→descendant metadata cascade in Hugo's sense. Plugins either (a) anchor a single folder-note for display/index purposes without cascade, or (b) implement inheritance at the fileClass/schema level disconnected from filesystem structure. The pattern is closer to "folder has an associated note" than "folder injects metadata into descendants."

**Observed limitations:**
- Community has repeatedly requested folder-level frontmatter with child inheritance (forum threads), but no plugin has shipped true cascade semantics in a stable form.
- Dataview (the query plugin) can synthesize folder-derived fields at query time, but this is read-side virtual, not storage-side inheritance.

## Cross-system synthesis

### Filename conventions

| System       | Filename                                    | Format                  |
| ------------ | ------------------------------------------- | ----------------------- |
| Fumadocs     | `meta.json`                                 | JSON / YAML / TS        |
| Docusaurus   | `_category_.json` / `_category_.yml`        | JSON / YAML             |
| Nextra       | `_meta.{js,ts,tsx,jsx,json}`                | JS (preferred) / JSON   |
| Hugo         | `_index.md`                                 | Markdown with YAML/TOML |
| Obsidian FN  | `<folderName>.md` or `_folder.md` (plugin)  | Markdown                |

**Convergence:** Everyone agrees on a sibling-file-per-folder pattern; nobody embeds folder metadata elsewhere. Everyone prefixes or uniquely names the file (leading `_`, or reserved basename like `meta`) so it doesn't collide with a real content page.

**Divergence:** Hugo uniquely uses the same markdown file format for folder metadata as for pages (`_index.md`), which means the "folder frontmatter" lives in YAML/TOML at the top of a normal markdown doc. All other systems use a dedicated non-markdown data file (JSON/YAML/JS). This is a meaningful split — Hugo's `_index.md` is both a renderable section landing page AND the folder metadata carrier.

### Inheritance patterns

| System       | Inheritance to descendants                                 |
| ------------ | ---------------------------------------------------------- |
| Fumadocs     | **None** — per-folder only                                 |
| Docusaurus   | **None** — per-folder only                                 |
| Nextra       | **Partial** — only `theme` cascades from folder to pages   |
| Hugo         | **Full deep cascade** — via explicit `cascade` opt-in      |
| Obsidian     | **None** in core/common plugins                            |

**Critical observation:** Hugo is the outlier. Every other documentation framework treats folder metadata as strictly local. Hugo is the only one that treats folder metadata as a mechanism for descendant-frontmatter injection, AND it makes that injection **opt-in via an explicit `cascade` key** rather than implicit. The implicit cascade pattern (every folder field auto-applies to children) does NOT appear in any mature system examined.

### Merge semantics

Across all systems that read both folder-metadata and page-frontmatter:

**Universal: folder metadata wins over folder-index-doc frontmatter for the folder-as-sidebar-node representation.**
- Fumadocs: `meta.title ?? index.frontmatter.title`
- Docusaurus: `_category_.label` overrides `index.md`'s `sidebar_label`
- Nextra: `_meta` entry title overrides folder-index title for sidebar

**Universal: page frontmatter wins for that page's own representation.** A child page's own frontmatter always beats any folder-level declaration for the child's own title, description, etc.

**Merge depth:** All systems do **shallow, per-field** merge. No system does deep object merge with one exception: Hugo's `params` map merges recursively (because `params` is already known to be a map-valued bag).

**Arrays:** Universal — arrays REPLACE, not concat. No system concats arrays at the folder/page merge boundary.

### Schema looseness

| System       | Typing                                                              |
| ------------ | ------------------------------------------------------------------- |
| Fumadocs     | **Strict Zod schema** — `metaSchema.extend()` for custom fields     |
| Docusaurus   | **Joi-validated, documented** — known fields; `customProps` bag     |
| Nextra       | **TS interfaces** — typed entries; JS variant allows dynamic values |
| Hugo         | **Loose** — cascade accepts any frontmatter field                   |
| Obsidian FN  | **Plugin-dependent** — typically loose                              |

**Observation:** The two systems most architecturally similar to an agent-edited CRDT environment — Fumadocs and Docusaurus — both lean toward a **small, tightly-typed, well-known field set** with an explicit `customProps` (Docusaurus) or Zod-extension (Fumadocs) escape hatch for arbitrary user fields. Nextra's JS-first approach trades type safety for expressive power (JSX in separators). Hugo maximally delegates to user-defined schemas (no schema at all).

### Primary consumer

Every system has a **single primary consumer** for folder metadata: the navigation/sidebar builder. No system uses folder metadata for search, rendering, or content transformation. This suggests the "folder metadata" problem space is 90% a navigation-shape problem.

Hugo is again the exception — cascaded frontmatter is visible to all template code, not just the nav. That's because Hugo's cascade is a frontmatter-injection primitive, not a nav-config primitive.
