# Evidence: D4 — Side-by-Side: Same Conceptual Surface in Each Product

**Dimension:** For each conceptual configuration surface that exists in either product, name the equivalent in both and note semantic differences.
**Date:** 2026-04-25
**Sources:** D2 + D3 evidence files. Direct cross-walks; no new claims beyond what those files document.

---

## Approach

Each row of the table below is a *conceptual surface* — a use case the reader might want to satisfy ("set a preference that follows me everywhere", "give the team a shared formatter rule"). For each, the table answers: what is it called and where does it live in each product? Where the products diverge semantically (not just in name), the "Notes" column flags it.

Rows are ordered roughly by how often the use case comes up.

---

## What the table shows

**Both products have a 4-5 layer scope ladder + a project-tree directory + first-class admin layer.** That much is consensus.

**VS Code has more *categories* of file-per-feature** within the same scope: `.vscode/` holds 5 distinct files for 5 different subsystems (settings, launch, tasks, extensions, MCP). Claude Code consolidates to one `settings.json` per scope, with separate top-level files for category surfaces VS Code doesn't have (CLAUDE.md, agents, commands, skills, .mcp.json).

**The asymmetric rows cluster:**
- VS Code-only: per-folder (multi-root), per-language, per-machine sync tags, snippets, tasks, debug, profiles, first-party sync, recommended extensions
- Claude Code-only: per-developer per-project local override (named/supported), text-as-context (CLAUDE.md), markdown-defined commands/skills/agents, hooks, permissions, CLI-as-scope, managed MCP gating

The pattern: VS Code's scope topology is richer along the *editor-feature* axis (per-language, per-folder, snippets, tasks); Claude Code's is richer along the *agent-trust* axis (permissions, hooks, managed-MCP, project-local override).
