# Shrinking AGENTS.md below the 40k performance threshold

**Status:** Applied. `AGENTS.md` reduced from 159,093 → 35,775 chars (78% reduction, 88% of the 40k limit).

## Problem

`AGENTS.md` (and its symlinked `CLAUDE.md`) had grown to 159,093 characters / 1,185 lines. Claude Code emitted a startup warning on every session:

```
Large CLAUDE.md will impact performance (153.8k chars > 40.0k)
```

The 40k threshold is the harness's performance guardrail. Above it, every session pays the context cost in both latency and instruction adherence — per Anthropic's own guidance, **"bloated CLAUDE.md files cause Claude to ignore your actual instructions."**

## What the evidence says

### Hard limits

- **OpenAI Codex AGENTS.md:** `project_doc_max_bytes` defaults to **32 KiB (~32,768 bytes)**. Content beyond that is silently truncated ([docs](https://developers.openai.com/codex/guides/agents-md)). At 159k chars, open-knowledge was ~5× over the Codex cap — any Codex user working in this repo was already getting truncated instructions.
- **Claude Code:** No published hard cap, but the [memory docs](https://code.claude.com/docs/en/memory) give a **200-line target** and explicitly warn that longer files reduce adherence. `@path` imports **do not reduce context cost** — they expand at load time alongside the root file.
- **This repo's harness:** 40,000-char warning threshold (shown in the session startup banner).

### Soft guidance (convergent across 5+ sources)

| Source | Target |
|---|---|
| [Anthropic — memory docs](https://code.claude.com/docs/en/memory) | < 200 lines |
| [HumanLayer](https://www.humanlayer.dev/blog/writing-a-good-claude-md) | < 60 lines (their production CLAUDE.md) |
| [aihero.dev](https://www.aihero.dev/a-complete-guide-to-agents-md) | 150–200 instructions (frontier LLM instruction-following ceiling) |
| [agentsmd.net](https://agentsmd/) | < 150 lines |
| [Anthropic — best practices](https://code.claude.com/docs/en/best-practices) | "Keep it short and human-readable" |

### The test Anthropic recommends

From [best practices](https://code.claude.com/docs/en/best-practices):

> For each line, ask: *Would removing this cause Claude to make mistakes?* If not, cut it.

## Patterns applied

Per-pattern breakdown, drawn from the research in [evidence/external-patterns.md](./evidence/external-patterns.md):

### 1. Point, don't embed

**Anti-pattern:**
> The bridge uses hybrid diff3+DMP three-way merge with per-document baseline tracking via `lastSyncedXmlMd` in `server-observers.ts`. Observer A dispatches via `afterAllTransactions`... [200 more lines]

**Pattern applied:**
> **Full observer design** (server-authoritative Path A/B, settlement dispatch via `afterAllTransactions`, origin-guard truth tables, paired-write markers, `applyAgentMarkdownWrite` reference implementation): [`ARCHITECTURE.md`](./ARCHITECTURE.md) + the four spec directories under `specs/2026-04-1[4-6]-*/`. `packages/server/src/server-observers.ts` is the canonical implementation.

The pointer tells the agent **when it matters** (the file list) and **where to go** (the spec). Claude doesn't read the spec on every session — only when the task requires it.

### 2. Prose references over `@path` imports

Anthropic's docs are explicit: `@path` imports expand at load time, so they do NOT reduce context cost. Plain prose references do. All pointers in the slim AGENTS.md use prose, not `@imports`.

### 3. STOP rules survive; rationale moves to code comments

The research recommends keeping 5–8 STOP rules. This codebase's density of load-bearing invariants justified 13. The criterion: does the rule protect against a silent-failure mode (wrong attribution, data loss, convergence race) that a linter can't catch? If yes, inline. If no, move to spec or code comment.

Rationale paragraphs ("this is load-bearing because...") move to inline code comments at the call site — zero context cost at load-time, maximum context proximity at edit-time.

### 4. Keep only behavior drivers

Status banners ("M1 + M2 + M3 + M4 + M5 + M6 shipped...") were cut. The fact that a milestone shipped doesn't change how an agent should work today. Milestones are a git log artifact and a spec artifact.

### 5. Preserve the auto-generated block verbatim

The `<!-- open-knowledge:begin --> ... <!-- open-knowledge:end -->` block (7,599 chars) is regenerated by `open-knowledge init` from `packages/cli/src/content/init.ts`'s `CLAUDE_MD_SECTION` constant. Modifying this content in AGENTS.md would be overwritten on the next `ok init` run. If further compression is needed here, edit `init.ts` and re-run `ok init`. **Out of scope for this PR.**

## Result

| Metric | Before | After | Δ |
|---|---|---|---|
| Total chars | 159,093 | 35,775 | **-123,318 (-77.5%)** |
| Total lines | 1,185 | 295 | -890 |
| Hand-authored chars | 151,494 | 28,176 | -123,318 |
| Auto-generated block chars | 7,599 | 7,599 | 0 |
| Threshold headroom | -297% | +11% | n/a |
| Codex 32KiB cap status | 4.9× over | 1.09× over | near cap |

The slim file still carries **every load-bearing STOP and WARN rule** from the original, with spec/file pointers attached so agents can follow through when the task warrants it.

## Tradeoffs accepted

1. **Lost inline rationale.** "Why `skipStoreHooks: true` exists" now lives in the spec + code comment, not in AGENTS.md. Mitigation: the STOP rule names the file and spec, and the code has comments.
2. **Lost file indexes.** "Key files" blocks are gone. Mitigation: file names are the canonical identifier (grep-able from any spec or code citation).
3. **Lost post-mortem narrative.** "PR #43's multi-client test matrix proved...". Most of these remain in git log / PR descriptions, not agent context.
4. **Lost fully-enumerated tables.** Origin-guard truth table, API endpoints table, Irreducible gaps NG1–NG11. Mitigation: all three live in ARCHITECTURE.md, server/README.md, and spec files respectively.

## Future moves (not in this PR)

1. **Per-package AGENTS.md files.** Per research rec #2, nested AGENTS.md in `packages/{server,app,desktop,core}/` would load only when an agent enters that package. This is a bigger refactor and warrants its own PR. Estimated further savings: bring the root file to ~15k.
2. **Skills for bridge / markdown work.** Rec #4 — convert the bridge architecture content into a skill with `paths: ["packages/server/src/server-observers.ts", ...]`. Skills load only when invoked.
3. **Rationale migration to code comments.** Rec #6 — migrate the "why this is load-bearing" paragraphs to `.ts` file comments at the protected sites. 2–3 line comments at the function definition > 20 lines in AGENTS.md.
