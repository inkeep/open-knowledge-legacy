## Findings

### Finding: `yaml@2` exposes a three-tier API; the Document layer is the canonical comment-preserving round-trip path

**Confidence:** CONFIRMED
**Evidence:** [eemeli/yaml content-nodes docs](https://github.com/eemeli/yaml/blob/main/docs/05_content_nodes.md), [Discussion #473](https://github.com/eemeli/yaml/discussions/473)

API tiers (from least to most metadata):
- **Tier 1:** `parse()` / `stringify()` — plain JS objects, no metadata
- **Tier 2:** `parseDocument()` / `parseAllDocuments()` — Document AST; carries comments/anchors/styling on Scalar/YAMLMap/YAMLSeq nodes
- **Tier 3:** `Lexer` / `Parser` / `Composer` — low-level CST; preserves byte-level whitespace

Document-layer node properties that round-trip:
- `comment`, `commentBefore`, `spaceBefore` (boolean — adds blank line)
- `anchor`, `tag`, `range` (character offsets)
- For scalars: `type` (`'QUOTE_SINGLE'`, `'BLOCK_LITERAL'`, etc.)
- For collections: `flow` (block vs. flow notation)

Mutation API: `doc.setIn(path, value)`, `doc.deleteIn(path)`, `doc.getIn(path)`, plus `add`/`set`/`get`/`has`/`delete` on collections. `setIn` auto-creates intermediate maps/sequences.

Maintainer-confirmed canonical pattern (Discussion #473):

```javascript
const doc = parseDocument(file);
doc.setIn(["devices", "laptop", "name"], "asus");
doc.deleteIn(["items", "mouse"]);
fs.writeFileSync(path, doc.toString());
```

**Critical detail:** Use `doc.toString()` or `String(doc)`, NOT `yaml.stringify()`. The latter strips anchor metadata; only the Document-bound serializer reads the AST nodes' formatting fields.

**Implications:** `yaml@2` is the only mainstream JS library that natively round-trips with comment preservation. Document is the canonical layer for "edit + write back."

### Finding: Document layer preserves comments + anchors + key order + scalar style; does NOT preserve byte-level whitespace

**Confidence:** CONFIRMED
**Evidence:** [Discussion #510](https://github.com/eemeli/yaml/discussions/510)

Maintainer in #510: *"For that you'll need to work with the CST API, as the Document level does not preserve whitespace."*

Document round-trip preserves:
- comments (line, before-key, after-value)
- blank lines (via `spaceBefore`)
- anchors / aliases
- custom tags
- key order
- scalar style (quoted/plain/block)
- block-vs-flow collection style

Document round-trip does NOT preserve:
- exact whitespace and inline formatting (e.g., `[50, 100]` becomes `[ 50, 100 ]`)
- trailing-comment association is documented as not completely stable

**Implications:** For ~99% of "user wrote a config with comments and reasonable formatting" cases, Document layer is sufficient. CST is required only for byte-perfect preservation (rare for config files).

### Finding: js-yaml has no comment preservation by design; multiple long-standing unresolved feature requests confirm this

**Confidence:** CONFIRMED
**Evidence:** [js-yaml #624](https://github.com/nodeca/js-yaml/issues/624), [#689](https://github.com/nodeca/js-yaml/issues/689), [#549](https://github.com/nodeca/js-yaml/issues/549), [#282](https://github.com/nodeca/js-yaml/issues/282), [#196](https://github.com/nodeca/js-yaml/issues/196), [#709](https://github.com/nodeca/js-yaml/issues/709)

`js-yaml.load()` / `dump()` operate on plain JS objects with no AST layer. Feature requests for comment preservation date back to #282 (2016). All remain open; no `--preserve-comments` option exists; no metadata side-channel.

The pages do not surface explicit "won't fix" prose from maintainers; what is observable is the empirical pattern — multi-year-open requests with no roadmap commitment, while third-party `preserve-yaml-comments` and `enhanced-yaml` exist precisely to fill the gap. Both are built on top of `eemeli/yaml`, not js-yaml.

**Implications:** Any architecture requiring round-trip with user-comment preservation must use `yaml@2`. js-yaml remains usable for one-way reads.

### Finding: No prior-art argues for converting YAML → JSON internally before validation; the canonical pattern is `yaml.parse() → schema.parse()`

**Confidence:** CONFIRMED
**Evidence:** [Zod docs](https://zod.dev/), [DEV.to: Validate YAML with Zod](https://dev.to/_d7eb1c1703182e3ce1782/how-to-validate-yaml-in-javascript-and-nodejs-320k); negative search across stackoverflow, dev.to, GitHub for "YAML to JSON before validation"

`yaml.parse(str)` returns plain JS objects with the same shape Zod consumes for any other JS object. No documentation, blog post, or library example was located that argues for an interstitial JSON serialization step. Inserting one would add cost (allocation, copy) without functional benefit — Zod doesn't care whether the input came from `JSON.parse` or `yaml.parse`.

If the Document layer is in use (`parseDocument`), `doc.toJS()` is the bridge — it resolves anchors/aliases (preserving JS object identity for repeated references) and produces a Zod-validatable plain object.

**Implications:** "Convert YAML → JSON internally" is an anti-pattern; the parser already produces JS objects.

### Finding: Two write-back semantics — AST-mutation (preserves comments) vs stringify-validated-object (loses comments). Production prior art uses AST-mutation.

**Confidence:** CONFIRMED
**Evidence:** [firebase-tools PR #6987](https://github.com/firebase/firebase-tools/pull/6987), [Discourse blog: Ruby psych comment preservation](https://blog.discourse.org/2026/02/how-we-fixed-yaml-comment-preservation-in-ruby-and-why-we-sponsored-it/), [Discussion #473](https://github.com/eemeli/yaml/discussions/473)

**Approach (a) — AST mutation:** `parseDocument()` + `setIn`/`deleteIn`/`addIn` + `doc.toString()`. Preserves comments, blank lines, anchors, most styling. firebase-tools PR #6987 explicitly: *"Edit configs at the AST level to preserve formatting and comments."* Discourse Ruby blog (Feb 2026) documents the parallel pattern in Ruby's psych library, sponsored to add the same capability that `yaml@2` provides natively.

**Approach (b) — Stringify validated object:** `validatedObj` → `yaml.stringify(validatedObj)`. Loses comments, anchors; key order normalizes; formatting canonicalizes to library defaults.

For partial UI patches, (a) is the only option that retains the "human-edited config file" property. Library API for (a) is the Document-node accessor methods listed in the first finding.

**Implications:** A multi-CRUD-path system that wants to preserve user comments must keep the Document AST as system-of-record and treat the validated JS object as a derived view.

### Finding: No comparable system in the surveyed set ships a dedicated GUI editor for its config file

**Confidence:** CONFIRMED
**Evidence:** [Renovate Config Validation](https://docs.renovatebot.com/config-validation/), [Renovate JSON Schema](https://docs.renovatebot.com/json-schema/), [Mintlify settings docs](https://www.mintlify.com/docs/organize/settings), [redhat-developer/vscode-yaml](https://github.com/redhat-developer/vscode-yaml), [kustomize.io](https://kustomize.io/)

**Renovate** uses JSON / JSON5, not YAML. Schema is autogenerated from config-option definitions in `tools/generate-schema.ts` and published at `docs.renovatebot.com/renovate-schema.json`. Three schema variants emitted. Validation tier:
1. Editor-level via `$schema` reference (autocomplete, inline validation)
2. CLI: `renovate-config-validator` standalone binary, with `--strict` flag
3. Runtime: server-side validation in Renovate codebase for things JSON Schema cannot express

**No dedicated UI for editing.** Configuration is text-edited; PR-based "reconfigure branch" is the closest thing to a guided workflow.

**GitHub Actions workflows** use YAML + JSON Schema published to schemastore.org (`https://json.schemastore.org/github-workflow.json`). IDE intellisense via redhat-developer/vscode-yaml, JetBrains, or the dedicated VS Code GitHub Actions extension. There is no GitHub-published GUI editor; the third-party "Workflow Editor" VS Code extension exists but is community-built.

**Mintlify** uses JSON (`mint.json`, now `docs.json`), not YAML. `$schema` reference at top of file. Editor-only validation; no UI editor. Mintlify recently consolidated tabs/anchors/versions/groups into a single recursive structure in `docs.json`, citing complexity from the prior multi-field mapping in `mint.json`.

**Kubernetes/Kustomize** is the standard tool for AST-aware YAML edits in the K8s ecosystem. Two patch dialects: strategic-merge patch (Kubernetes-specific semantics) and JSON Patch RFC 6902 (generic, finer control). Helm renders templates first, then Kustomize patches the rendered YAML. Neither Kustomize nor Helm publishes a GUI editor.

**Pattern across all four:** schema-driven editor support is universal; dedicated GUI editors are universally absent or third-party. Validation is layered (editor → CLI → server-runtime), never single-stage.

**Implications:** Shipping a settings-page form-UI on top of a YAML config is not the dev-tool norm — most equivalent tools rely on IDE intellisense + CLI validators. The space is not unexplored, but it's also not crowded with prior art.

### Finding: Anchors / aliases / custom tags survive `yaml@2` round-trip but are flattened by the time Zod validates

**Confidence:** CONFIRMED
**Evidence:** [Discussion #303](https://github.com/eemeli/yaml/discussions/303), [content-nodes docs](https://github.com/eemeli/yaml/blob/main/docs/05_content_nodes.md)

In `yaml@2`:
- `parseDocument()` retains anchors as `node.anchor` properties on Scalar and collection nodes
- `doc.toString()` re-emits them faithfully
- `doc.toJS()` resolves aliases — *"Where possible the JS representation of alias nodes will be the actual source object,"* so multiple aliases pointing to the same anchor become the same JS object reference (`===`)
- During `stringify()` of plain JS structures, the library auto-generates anchors when it detects multiple references to identical objects

Zod sees the resolved JS object — anchors look like ordinary repeated values, aliases are object-identity references. The schema cannot directly observe "this was an alias to anchor X."

**Implications:** If anchor preservation matters for write-back, the Document AST must be the system-of-record. Custom tags follow the same pattern.

## Cross-cutting observations

- The "AST-as-system-of-record" / "JS object as derived view" split is the architectural fork in this dimension. If round-trip integrity matters, the Document AST holds canonical state. If validation can rewrite the file, comments and formatting are sacrificed regardless of which library is used.
- Library choice for comment-preserving round-trip is binary: `yaml@2` (eemeli) is the only mainstream option that natively preserves comments via the Document AST. js-yaml has been observed-disqualifying for ~10 years across multiple unresolved issues; third-party comment-preserving wrappers are themselves built on top of `eemeli/yaml`.
- In the surveyed cohort (Renovate, GitHub Actions, Mintlify, Kustomize), schema-driven editor intellisense + CLI validation is the universal pattern. Dedicated GUI editors are absent or third-party.
