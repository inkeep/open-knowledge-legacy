# Evidence: D6.4 — Giant-Document Scaling and Memory Pressure in JS Markdown Parsers

**Dimension:** Published benchmarks and real-world memory/performance findings for parsing large markdown documents (1MB, 10MB, 100MB) and MDX builds.

---

**No source published a clean "1 MB / 10 MB / 100 MB markdown × parser" table** — that gap is itself a finding (see §5).

The Pro Git observation is the closest thing to a giant-document data point: scaling from 1× to 20× a moderately large book yields ~25–47× slower throughput.

## 2. Per-Parser Memory Complexity Findings

### markdown-it
- Benchmark harness (`benchmark/benchmark.mjs` + `samples/` + `implementations/`) is a wrapper around Benchmark.js. **No memory instrumentation built in.** Source: https://github.com/markdown-it/markdown-it/tree/master/benchmark
- Confirmed quadratic-time pathology in inline rules: CVE-2022-21670 (see D6.3).
- Internal token stream (`Token[]`), not a persistent AST; memory ~ O(tokens) plus the original string. No documented O(n²) memory.

### micromark
- **Streaming is partial, not true.** The maintainer documents that "the full markdown document needs to be parsed before compiling HTML/JS output" because of features like reference-link definitions where the two halves can sit on opposite ends of the document. The Node stream API exists but ultimately buffers.
  - Sources: https://github.com/micromark/micromark, https://github.com/micromark/micromark/blob/main/packages/micromark/dev/stream.js
- Implementation: state machine emitting concrete tokens with positional info, then HTML compilation. Every byte is accounted for in tokens — token memory is O(n) on the input.
- Maintainer's explicit guidance: **"It is wise to cap the accepted size of input (500kb can hold a big book)"** — tacit admission that >500 KB is not a tested/guaranteed regime. Source: https://github.com/micromark/micromark
- Performance: maintainer states micromark is **~50% slower than remark-parse used to be**, and one community benchmark recorded only ~229 ops/sec (vs markdown-it ~5,538). Source: https://github.com/orgs/micromark/discussions/29

### remark / remark-parse
- Built on micromark since remark v13/14 — inherits micromark's whole-document-buffering constraint.
- Official remark README warns: **"an attacker could cause a crash or slow down with big files"** and "crashes can also originate from smaller payloads, often when thousands of things (such as lists or links) are opened." Source: https://github.com/remarkjs/remark
- The remarkjs/discussions/1027 thread acknowledges that **virtualization in react-markdown won't work for the parse step** because lookahead/lookbehind features prevent splitting. Source: https://github.com/orgs/remarkjs/discussions/1027

### marked
- Marketing positions it as "low-level compiler for parsing markdown without caching or blocking for long periods of time." No concrete memory complexity is documented in the README.
- Older snapshot shows marked at 415 → 729 ops/sec across optimization rounds; competitive with markdown-it on small inputs but no published large-document scaling.
- Snyk/CVE history: ReDoS vulnerabilities have been recurrent (see D6.3).

### commonmark.js
- Reference implementation; no streaming claim. Operates on the full input. 709 ops/sec on the spec corpus.

### MDX (`@mdx-js/mdx`)
- Pipeline is materially heavier than CommonMark parsers: micromark (markdown) → MDAST → JSX-aware AST → estree → JS.
- The `mdx-js/mdx#1152` umbrella issue identifies "numerous babel parse and transformation steps throughout the MDX transpilation pipeline" — multiple babel parses for imports/exports, shortcodes, and `mdxType` resolution, plus JSX-string output that callers must re-transpile.
- Concrete observation in that issue: memory use spiked from ~2 GB to over 3 GB for one MDX-heavy build.
- Source: https://github.com/mdx-js/mdx/issues/1152

## 3. Real-World Incident Reports

### Docusaurus + MDX OOM cluster (most consistent OOM vector in JS-land markdown)

- **[Issue #4785 — npm start, build fails with `JavaScript out of memory`](https://github.com/facebook/docusaurus/issues/4785)** — V8 heap allocation failure during webpack compilation at 23–42% complete; reproduction is the user's own large MDX files.
- **[Issue #8329](https://github.com/facebook/docusaurus/issues/8329)** — Docusaurus 2.2.0 multi-locale build, OOM at ~1945–1949 MB during server compilation.
- **Issues #7410, #1782, Discussion #3132, #11259, #10523, #10895** — recurring pattern of OOM in large doc sites. Workarounds documented: `NODE_OPTIONS=--max_old_space_size=4096` (or 16000), pruning git history, splitting into multiple sites, archiving older versions, switching to Docusaurus Faster (Rspack) which "consumes less memory than Webpack."
  - https://github.com/facebook/docusaurus/discussions/10523
- Discussion #3132 attributes blame partially to remark plugins (directives, emojis) and regex-heavy transforms on certain pages.

### Astro + MDX
- **[withastro/astro#4894](https://github.com/withastro/astro/issues/4894)** — server hangs until OOM if an MDX file imports something that throws an error.
- Steve Fenton write-up: Astro + GitHub Actions OOM, fixed via `NODE_OPTIONS=--max_old_space_size=4096`. https://stevefenton.co.uk/blog/2023/07/astro-javascript-heap-out-of-memory/

### Gatsby + MDX
- **[ChristopherBiscardi/gatsby-mdx#411](https://github.com/ChristopherBiscardi/gatsby-mdx/issues/411)** — crash threshold around **>1590 lines** of MDX with PrismJS syntax-highlighting; Babel emits the warning "code generator has deoptimised the styling of undefined as it exceeds the max of 500KB." Increasing Node heap, removing source maps, and removing styled-components all failed; **only removing MDX worked**.

### react-markdown / remark in browser
- **[remarkjs/react-markdown#289](https://github.com/remarkjs/react-markdown/issues/289)** — large markdown file pegs CPU at 100% during tokenization, blocking the event loop and freezing animations. Solution explored: setTimeout yields between tokenizer batches.
- **[remarkjs/react-markdown#459](https://github.com/remarkjs/react-markdown/issues/459)** — "Improving performance" tracking issue.
- **[remarkjs discussion #1027](https://github.com/orgs/remarkjs/discussions/1027)** — virtualization debunked at the parse layer; recommendation to do parsing server-side for very large inputs.

### GitHub render DoS
- No publicly documented production DoS report on GitHub's renderer specifically attributable to a giant-markdown payload was located (negative finding — see §5).

## 4. MDX-Specific Scaling Notes

- MDX compounds two cost drivers on top of plain markdown:
  1. The markdown→MDAST→hast→estree pipeline must hold the entire document tree because MDX import/export and component scope analysis happen across the whole module.
  2. Babel parse passes for JSX/ES module shape happen multiple times in the v1/v2 pipeline (per #1152).

- Practical OOM threshold from the Docusaurus #8329 trace: a multi-locale doc site hits ~1.95 GB heap before V8 aborts on default settings — i.e., the **default 2 GB Node heap is the de facto ceiling**.

- `gatsby-mdx#411`'s 500 KB Babel deopt warning is the most concrete published threshold: Babel's code generator deoptimises styling above 500 KB of generated source, producing a phase shift that frequently combines with PrismJS expansion to push a build over the heap limit.

## 5. Negative Searches Noted

- A direct "I parsed a 100 MB markdown file with X parser" empirical post — **none located** on HN/Algolia, parser READMEs, or community benchmarks.
- The markdown-it `benchmark/` directory **does not publish results** — harness only, no checked-in result table.
- The micromark README **does not document memory complexity or large-document benchmarks**.
- Issue-tracker search for "OOM"/"memory leak"/"out of memory" inside the `micromark/micromark` repo surfaced **no public OOM/leak issues filed against micromark proper** (the OOMs surface downstream — Docusaurus, Astro, Gatsby).
- No published GitHub-side incident attributing a render DoS to a single giant-markdown payload was located.

---

## 6. Synthesis of the Scaling Story

1. **All popular JS markdown parsers buffer the whole document.** Even micromark's "streaming" interface is a duplex wrapper around an internally-buffering tokenizer because reference-style links and other lookaround features forbid true streaming. Parser memory is at least O(n) over input plus O(tokens) over derived state, with no implementation offering chunked parse-and-discard.

2. **Throughput is super-linear in the bad cases.** The 20× Pro Git → ~1–2 ops/sec datapoint and the documented quadratic ReDoS in markdown-it's whitespace handling (CVE-2022-21670) both indicate that large or pathologically structured input leaves the linear regime.

3. **Maintainers explicitly recommend a 500 KB input cap** (micromark) and warn about big-file crashes (remark) — the published parsers consider sub-megabyte the supported regime.

4. **The MDX pipeline is the real-world scaling failure point.** Docusaurus, Astro, and Gatsby OOMs cluster around MDX builds, not raw markdown rendering. Default Node 2 GB heap is routinely exhausted; the standard mitigation is `NODE_OPTIONS=--max_old_space_size=4096+`. The Babel 500 KB deopt is a documented phase shift inside MDX's compile step.

5. **No 1 MB / 10 MB / 100 MB head-to-head benchmark exists in the public record** for these parsers.
