# Evidence: D6.2 — Deep-nesting and stack overflow bugs in JS markdown parsers

**Dimension:** Documented stack-overflow / RangeError / recursive blow-ups when parsing deeply nested markdown structures.

**Sources searched:**
- https://github.com/remarkjs/remark/issues
- https://github.com/micromark/micromark/issues
- https://github.com/markdown-it/markdown-it/issues
- https://github.com/markedjs/marked/issues
- https://github.com/mdx-js/mdx/issues
- https://github.com/syntax-tree/mdast-util-from-markdown/issues

---

## 1. Summary of Bug Class

JS markdown parsers across the major implementations have a long history of stack-overflow / `RangeError: Maximum call stack size exceeded` failures triggered by pathological deeply-nested or repetitive constructs. The class spans three failure modes:

- **Lexer recursion** — the parser reaches into nested constructs (blockquote-in-blockquote, list-in-list, link-in-link) by calling itself; deeply-nested input blows the JS engine's call stack.
- **Token post-processing recursion** — even tokenizer-based parsers (micromark) recurse during a "subtokenize / unravel" phase that linearizes nested token streams, and that recursion can blow the stack on giant inputs even when the main parse is iterative.
- **Engine-API misuse** — `Array#splice(...thousandsOfArgs)` and similar variadic calls in the post-processor can themselves push enough frames to overflow the stack.

**JS-engine call-stack ceilings** (from [marked#1462 reproducer](https://github.com/markedjs/marked/issues/1462)):
- Node: ~3,842 frames
- Chrome: ~3,931 frames
- Firefox: ~49,392 frames

A handful-of-KB pathological input can DoS a server-side renderer.

## 2. Per-Issue Findings

### micromark / unified ecosystem

**[micromark#20 — `unravelLinkedTokens RangeError: Maximum call stack size exceeded`](https://github.com/micromark/micromark/issues/20)** (CONFIRMED, FIXED)
- Parser: micromark 2.6.1 on Node 14
- Payload: `[](` repeated ~35,000 times (file in the 105 KB – 1 MB range); also reported on giant paragraphs where every character becomes an individual token
- Error (verbatim): `RangeError: Maximum call stack size exceeded at unravelLinkedTokens (micromark/dist/util/subtokenize.js:1:16585)`
- Status: Closed / labelled "phase/solved"
- Root cause: post-processing `unravelLinkedTokens` recursion plus `Array#splice` taking all items to be inserted as individual arguments
- Fix: combined a `while`-loop conversion with a chunked splice

**[micromark security note in README](https://github.com/micromark/micromark/blob/main/readme.md)** (verbatim quotes):
- "an attacker could throw a 100mb file at micromark, in which case the JavaScript engine will run out of memory and crash"
- "possible to crash micromark with smaller payloads, notably when thousands of links, images, emphasis, or strong are opened but not closed"
- Mitigation: "It is wise to cap the accepted size of input (500kb can hold a big book) and to process content in a different thread or worker"

This is a tacit acknowledgement that micromark does not by itself protect against deep-nesting / unbalanced-construct DoS.

### markedjs/marked

**[marked#1462 — `marked(">".repeat(5000))` crashes](https://github.com/markedjs/marked/issues/1462)** (CONFIRMED, FIX REVERTED)
- Payload (verbatim): `marked(">".repeat(5000))`
- Error: `RangeError: Maximum call stack size exceeded` at `Lexer.token` (line 307), recursing through `RegExp.exec`
- Engine ceilings reported: Node ~3,842, Chrome ~3,931, Firefox ~49,392 nesting levels
- Status: Closed; classified internally as "L0" security issue
- Fix attempt: [PR #1464 "Nested blockquotes"](https://github.com/markedjs/marked/pull/1464) converted blockquote parsing from recursion to iteration. **Important caveat**: the PR was merged 2019-05-03 but reverted in PR #1497 on 2019-06-12 after regressions surfaced.

**[marked#1471 — Nested lists `Maximum call stack size exceeded`](https://github.com/markedjs/marked/issues/1471)** (CONFIRMED)
- Payload (verbatim):
  ```js
  const marked = require('marked');
  const iterations = 300;
  let spaces = 0;
  let markdown = '';
  for (let i = 0; i < iterations; i++) {
    markdown += ' '.repeat(spaces) + '- a\n';
    spaces += 2;
  }
  marked(markdown);
  ```
- Performance pathology: 300 iterations ≈ 3 s; 600 iterations ≈ 40 s (super-linear before crashing)
- Status: Closed; classified L0 security
- Cross-references sibling issue #1462

**[marked#2220 — RangeError on emoji-bearing markdown](https://github.com/markedjs/marked/issues/2220)** (CONFIRMED)
- Parser: marked 2.0.5
- Payload: a one-line message with emoji `👋` and a markdown link wrapping `📣` — the issue surfaced infinite recursion in `Tokenizer.link` rather than from depth nesting per se.
- Indicates the recursion-DoS class isn't limited to obvious deep nesting; certain Unicode + link interactions also recurse.

**[marked#3379 — v13 custom-renderer stack overflow](https://github.com/markedjs/marked/issues/3379)** (CONFIRMED)
- Parser: marked 13.0.2
- Payload: 10,000 sequential calls to `marked.parse(content)` after `marked.use({ renderer: { heading(...) {...} } })`
- Root cause: each `.use({ renderer })` reassigns `rendererFunc`, so calls accumulate as a wrapper chain — every `parse` adds a stack frame layer.
- Status: Closed; PR #3380.

### markdown-it

**No direct issue tracker hits for "stack overflow" landed in the searches.** The library's defense is structural: it ships with a `maxNesting` knob.

- Default preset: `maxNesting: 100` — [`lib/presets/default.mjs`](https://github.com/markdown-it/markdown-it/blob/master/lib/presets/default.mjs)
- CommonMark preset: `maxNesting: 20` — [`lib/presets/commonmark.mjs`](https://github.com/markdown-it/markdown-it/blob/master/lib/presets/commonmark.mjs)

When tokenizers reach the limit they bail out instead of recursing further, so users do not see `RangeError` from nesting in normal config. (The package's `docs/security.md` URL is currently 404; security guidance is folded into the README.)

### MDX / remark-mdx / micromark-extension-mdx-jsx

No clean "deep JSX nesting → stack overflow" issue surfaced in the public tracker, but multiple "parser crash on nested JSX/Markdown interaction" issues exist:

- **[mdx#2172 — nested lists break the parser](https://github.com/mdx-js/mdx/issues/2172)** — error `Expected a closing tag for <li>(3:3-3:7) before the end of paragraph` from `mdast-util-mdx-jsx`. Parser version: @mdx-js/mdx 2.1.5. Status: closed as "no/question."
- **[mdx#1476 — code blocks inside nested components break stringify](https://github.com/mdx-js/mdx/issues/1476)**
- **[mdx#1318 — nested JSX with hanging `>`](https://github.com/mdx-js/mdx/issues/1318)**
- **[mdx#1202 — JSX wrapping a multiline nested list breaks the parser](https://github.com/mdx-js/mdx/issues/1202)**
- **[remarkjs/remark-jsx#10 — complex nesting of markdown and component can break parsing](https://github.com/remarkjs/remark-jsx/issues/10)**
- **[micromark-extension-mdx-jsx#10 — partial mode / crash on incomplete MDX](https://github.com/micromark/micromark-extension-mdx-jsx/issues/10)** — when MDX is incomplete the parser explicitly calls a `crash` function in `factory-tag.js`. Not a stack overflow per se, but a hard failure on imbalanced-tag input that an attacker can shape.
- **[withastro/astro#4533 — call stack errors when Astro component props are changed repeatedly in MDX](https://github.com/withastro/astro/issues/4533)** — the same `marked#3379`-style "config mutation accumulates frames" bug, manifest in the MDX/Astro pipeline.

### Adjacent ecosystem corroboration

- **[pandoc#621 — markdown table that causes stack overflow on writing](https://github.com/jgm/pandoc/issues/621)** — confirms the failure mode reaches Haskell parsers.
- **[dotnet/docfx#8854 — explicit "Markup failed: Markdown elements in the input are too deeply nested - depth limit exceeded"](https://github.com/dotnet/docfx/issues/8854)** — example of an industrial markdown processor that surfaces a depth limit as a first-class user-visible error.

## 3. Architectural Notes Per Parser

### marked
Classic recursive-descent lexer in JavaScript. `Lexer.token` (and historically the blockquote tokenizer) calls itself on nested content, so each level of `>` or list indent costs one JS stack frame. The fixes shipped have been targeted (specific construct → iterative) rather than a global depth limit, and at least one fix (PR #1464) was reverted for regressions. There is no global `maxNesting` user knob comparable to markdown-it's. Instead, hot constructs are individually de-recursed over time. The recursion ceiling is therefore JS-engine-defined (~3.8k frames in Node).

### markdown-it
Token-stream architecture with explicit per-parser `maxNesting` counter checked at every nested-construct entry. When a new block/inline rule enters, it bumps a depth counter; when it exceeds `options.maxNesting` (default 100, commonmark preset 20), the rule bails and emits the raw text rather than recursing. This is the only mainstream JS markdown parser with a documented depth-limit knob. Operationally, markdown-it does not stack-overflow on `>`.repeat(N); it just stops recognising `>` as a blockquote past depth N.

### micromark
A streaming character-by-character state machine — the four-stage pipeline (preprocess → parse → postprocess → compile) is largely iterative, and the README claims this design beats predecessor parsers on safety. **However**: the `subtokenize` / `unravelLinkedTokens` post-processor that reconstructs nested content from the linear token stream **does recurse**, and that recursion has been the source of at least one documented stack-overflow (#20). Mitigation was a `while`-loop rewrite plus chunked `Array#splice` to avoid variadic-call frame blow-up. Constructs are pluggable state-machine objects that emit `enter`/`exit` events for nested content; the linked-tokens approach replaces direct recursion with token references that a later pass walks. There is **no `maxNesting` knob** — deep nesting is structurally cheaper than in marked but not bounded.

### mdast-util-from-markdown
Walks the micromark token stream and constructs an `mdast` tree. It is a tree builder, so it pushes onto a stack as it sees `enter` events and pops on `exit`. The stack is a JS array, not the call stack, which makes it more robust than recursive descent — but the consuming visitors (`mdast-util-to-hast`, `mdast-util-mdx-jsx`, etc.) do recurse via `unist-util-visit` and friends, so deep mdast trees can still blow the stack downstream of the parser.

### MDX (remark-mdx + micromark-extension-mdx-jsx)
Adds JSX tag tokenization on top of micromark. The `factory-tag.js` state machine has explicit `crash` calls on malformed input rather than recovery, and the JSX bridge to `acorn` for expression evaluation introduces a second recursion budget (acorn's parser). Documented failure modes are mostly "throws a parse error" rather than "stack overflows," but the multiple "nested X breaks parser" issues show the failure surface is wide.

## 4. Mitigation Knob Inventory

| Parser | Depth-limit option | Default | Notes |
|---|---|---|---|
| markdown-it (default preset) | `maxNesting` | **100** | Bails to raw text past limit; no exception. |
| markdown-it (commonmark preset) | `maxNesting` | **20** | Stricter cap. |
| marked | none | n/a | No global depth knob; relies on per-construct iterative rewrites. Recursion ceiling is JS-engine-defined (~3.8k frames in Node). |
| micromark | none | n/a | README explicitly recommends external defenses: cap input size at ~500 KB, run in a worker. |
| mdast-util-from-markdown | none | n/a | Tree builder uses a JS-array stack, not call stack; depth bounded by available heap, not call-stack ceiling. |
| MDX (remark-mdx) | none | n/a | Inherits micromark's lack of depth limit; adds acorn parser limits implicitly. |
| Adjacent: docfx (.NET) | depth limit | exists | Surfaces "depth limit exceeded" as user-visible error. |

## 5. Concrete Reproducer Payloads

```js
// marked#1462: blockquote bomb (~5KB triggers crash on Node)
marked(">".repeat(5000));

// marked#1471: indented list bomb
let s = '';
for (let i = 0, sp = 0; i < 300; i++, sp += 2) s += ' '.repeat(sp) + '- a\n';
marked(s);

// micromark#20: unclosed link bomb
parseMarkdown("[](".repeat(35000));

// marked#3379: config-mutation accumulation
for (let i = 0; i < 10000; i++) marked.use({ renderer: { heading: () => '' } });
marked.parse(content); // crashes

// MDX-flavored speculation (NOT TESTED IN PUBLIC RECORD):
parseMdx("<A>".repeat(N) + "x" + "</A>".repeat(N));
parseMdx("<A>".repeat(N)); // unclosed nested JSX
```

## 6. Negative Searches Noted

- No issues titled "stack overflow" + "remarkjs/remark" matched directly in `site:github.com` searches.

- No CVE / GitHub Security Advisory specifically tagged against `micromark`, `remark`, or `mdast-util-from-markdown` for stack-overflow / depth-DoS.
- `talk.commonmark.org` discourse search returned only general "writing a CommonMark parser is hard" essay material; no specific deep-nesting threads surfaced.
- **No public issue found that reproduces deeply-nested MDX `<A><B><C>...` (~thousands of levels)** — this remains untested in the public record and is a plausible undisclosed reproducer that the parent pipeline could exploit for its own coverage.
