# Evidence: D6.3 — ReDoS and Quadratic-Time Pathological Inputs

**Dimension:** ReDoS vulnerabilities and quadratic-time pathological inputs in JS markdown parsers and their plugin ecosystem.

---

## 1. Per-Package Findings

### `marked` — Heavy regex history, multiple confirmed ReDoS

**CVE-2022-21680 / GHSA-rrrm-qjm4-v8hf — `block.def` cubic backtracking** (CONFIRMED, High, CVSS 7.5)
- Affected: `marked < 4.0.10`
- Vulnerable component: the `block.def` regex (link-reference definition rule)
- PoC payload (verbatim): `marked.parse(\`[x]:${' '.repeat(1500)}x ${' '.repeat(1500)} x\`)` — alternating whitespace + characters drives cubic-time backtracking.
- URL: https://github.com/markedjs/marked/security/advisories/GHSA-rrrm-qjm4-v8hf

**CVE-2022-21681 / GHSA-5v2h-r2cx-5xgj — `inline.reflinkSearch` exponential backtracking** (CONFIRMED, High, CVSS 7.5)
- Affected: `marked < 4.0.10`
- Vulnerable component: `inline.reflinkSearch` regex used during reference-link resolution
- PoC pattern: many repeated unclosed `[](` groups, e.g. `[x]: x\n` followed by 30+ chained `[](` openers.
- Reporter: @makenowjust (also reported the markdown-it newline ReDoS).
- URL: https://github.com/markedjs/marked/security/advisories/GHSA-5v2h-r2cx-5xgj

**CVE-2018-25110 / GHSA-p9wx-2529-fp83** — Older ReDoS in `marked` (Moderate). URL: https://github.com/advisories/GHSA-p9wx-2529-fp83

### `markdown-it` — Some regex; smaller but real ReDoS history

**CVE-2022-21670 / GHSA-6vfc-qv3f-vr6c — newline rule ReDoS** (CONFIRMED, Moderate, CVSS 5.3)
- Affected: `≤ 12.3.1`; patched in `12.3.2`
- Vulnerable regex: `/\s+$/` at line 23 of `lib/rules_inline/newline.js`
- PoC: `x ${' '.repeat(150000)} x  \nx`
- Sources: https://github.com/markdown-it/markdown-it/security/advisories/GHSA-6vfc-qv3f-vr6c, https://security.snyk.io/vuln/SNYK-JS-MARKDOWNIT-2331914

**SNYK-JS-MARKDOWNIT-6483324 — linkify infinite loop** (CONFIRMED, High, CVSS 7.5)
- Affected: `< 13.0.2`. Crash/infinite loop via the linkify inline rule when fed malformed input.
- URL: https://security.snyk.io/vuln/SNYK-JS-MARKDOWNIT-6483324

**CVE-2026-2327 / GHSA-38c4-r59v-3vqw — Linkify ReDoS regression in v13.x** (CONFIRMED)
- Versions `13.0.0 ≤ markdown-it < 14.1.1` are vulnerable to ReDoS via `/*+$/` in the linkify code path. Trigger: long sequence of `*` followed by a non-matching character.
- Fixed in 14.1.1.
- Sources: https://github.com/markdown-it/markdown-it/blob/master/CHANGELOG.md, https://security.snyk.io/package/npm/markdown-it

**Quadratic complexity (non-regex) fixes in markdown-it changelog**:
- `14.1.0` (2024-03-19): "quadratic complexity when parsing references"; "quadratic output size with pathological user input in tables."
- `12.3.0` (2021-12-09): "quadratic complexity in pathological `***<10k stars>***a***<10k stars>***` case" (emphasis delimiter run).
- `10.0.0` (2019-09-11): "quadratic parse time for some combinations of pairs."
- `3.0.6` (2015-02-12): hang on long vertical link lists.
- Source: https://github.com/markdown-it/markdown-it/blob/master/CHANGELOG.md

### `micromark` / `remark` — Character-by-character state machine

- micromark's README and CommonMark talk page describe it as "a state machine that emits concrete tokens, so that every byte is accounted for." It does not rely on backtracking regexes for the parser core.
  - Sources: https://github.com/micromark/micromark, https://talk.commonmark.org/t/js-micromark-a-new-way-to-parse-markdown/3634
- Snyk reports **zero known direct vulnerabilities** for the `micromark` package and no listed entries for `micromark-core-commonmark`. (https://security.snyk.io/package/npm/micromark)
- This is the strongest empirical signal in the ecosystem that the architectural choice (state machine vs regex) eliminates the ReDoS class.
- **Caveat:** does not eliminate algorithmic-complexity bugs — see CommonMark emphasis below.

### `commonmark.js` — Two historical ReDoS fixes

- Issue #157 (Anders Kaseorg): two exponential-backtracking regexes. Root cause: `ESCAPED_CHAR` already matched `\\`, and matching it again in another alternative produced exponential ambiguity. Affected link-reference parsing and angle-bracketed link destinations.
- Issue #281 (Michael Howell): title-related backtracking on empty strings.
- No CVE assigned; fixed in the repo. Source: https://github.com/commonmark/commonmark.js/releases

## 2. Pathological Input Patterns Inventory

| # | Pattern | Targets | Complexity class | Mechanism |
|---|---|---|---|---|
| P1 | `'[x]:' + ' '.repeat(1500) + 'x ' + ' '.repeat(1500) + ' x'` | marked `block.def` | Cubic | Alternating whitespace/text in link-reference definition |
| P2 | `'[x]: x\n' + '[]('.repeat(N)` | marked `inline.reflinkSearch` | Exponential | Unclosed reference-link openers |
| P3 | `' '.repeat(150_000) + '\n'` | markdown-it `/\s+$/` newline rule | Quadratic | Trailing-whitespace regex matches all positions |
| P4 | `'*'.repeat(N) + 'x'` | markdown-it linkify `/*+$/` (v13.x) | Quadratic+ | Greedy `*+` against non-matching tail |
| P5 | `'!['.repeat(100_000)` (a.k.a. MarkdownTime `'![l' * 100000`) | cmark-gfm autolink extension | Polynomial | Each `l` rescans bracket stack |
| P6 | `'***' + 'a'.repeat(10_000) + '***' + 'a' + '***' + ...` | markdown-it emphasis delimiter run (pre-12.3.0) | Quadratic | CommonMark delimiter-stack pairing |
| P7 | `'<a foobar="foo' + '\\!'.repeat(N) + ' />'` | markdown-to-jsx `ATTR_EXTRACTOR_R` | Exponential | Ambiguous escape character class |
| P8 | `'~~' + '\\}'.repeat(N) + '~'` | simple-markdown strikethrough | Exponential | Overlapping char classes inside `~~ ~~` |
| P9 | `' '.repeat(100_000) + '@'`, `'['.repeat(100_000)`, `'('.repeat(100_000) + '\n'` | CodeMirror 5 markdown mode (5 regexes) | Catastrophic | Greedy quantifiers with end anchors |
| P10 | Wide tables: many pipes per row × many rows | markdown-it tables (pre-14.1.0) | Quadratic output | "quadratic output size with pathological user input in tables" |
| P11 | Many reference-style links sharing a label namespace | markdown-it (pre-14.1.0) | Quadratic | "quadratic complexity when parsing references" |

The originally hypothesized patterns from the prompt (`"a".repeat(N)+"*"`, `"[".repeat(N)+"]".repeat(N)`, `"\\".repeat(N)`, `"<".repeat(N)`, wide tables, footnote `^[]` chains) all map to documented bugs, with the exception of footnote-specific ReDoS — Snyk shows **no direct vulnerabilities** for `markdown-it-footnote`.

## 3. Mitigations Per Parser

**`marked`** — No built-in timeout or circuit breaker. Official guidance: *"To prevent ReDoS attacks you can run marked on a worker and terminate it when parsing takes longer than usual."* `marked.parse({ async: true })` only awaits async `walkTokens`; it does not guard parse time. Sanitization was removed in v8.0.0 (delegates to DOMPurify). Source: https://marked.js.org/using_advanced

**`markdown-it`** — No built-in timeout. The project has fixed each ReDoS as it surfaced (newline 12.3.2; linkify regression 14.1.1) and has refactored algorithmic hotspots (emphasis 12.3.0, references/tables 14.1.0). The official `docs/security.md` URL 404s; security guidance is folded into the README. Source: https://github.com/markdown-it/markdown-it/security

**`micromark` / `remark`** — Architectural mitigation: state-machine, character-by-character. No backtracking regex engine in the parse hot path. Snyk shows zero known vulns for `micromark`. This is not a guarantee against algorithmic DoS, but it eliminates the ReDoS *class* by construction. Source: https://github.com/micromark/micromark

**`commonmark.js`** — Mitigates by patching specific regexes (e.g., removing `ESCAPED_CHAR` ambiguity). No timeout/circuit breaker.

**Static analysis tooling that has caught markdown ReDoS**:
- `recheck` (makenowjust): combines fuzzing + static analysis with backreference/look-around support; the same author reported CVE-2022-21670, CVE-2022-21680, and CVE-2022-21681. Source: https://makenowjust-labs.github.io/recheck/
- `safe-regex`, `RegexStaticAnalysis`, `saferegex`: general-purpose ReDoS detectors. Source: https://github.com/NicolaasWeideman/RegexStaticAnalysis

## 4. Quadratic-Time Non-Regex Bugs

These are algorithmic-complexity bugs in parser logic, not regex backtracking:

1. **CommonMark emphasis delimiter run** — The reference algorithm for pairing `*`/`_` runs ("look for link or image" + "process emphasis") is famously tricky. markdown-it explicitly logged "quadratic parse time for some combinations of pairs" (10.0.0) and "quadratic complexity in pathological `***...***a***...***`" (12.3.0).

2. **Reference-link lookup** — Without deduplication / hashing, `[ref][label]` resolution is O(n²) over labels. markdown-it 14.1.0 specifically called out "quadratic complexity when parsing references."

3. **Table output explosion** — markdown-it 14.1.0: "quadratic output size with pathological user input in tables." Wide rows × many rows produces O(n²) HTML.

4. **MarkdownTime / cmark-gfm autolink bracket stack** — Polynomial: each character in the unclosed `![l...` stream re-scans the accumulated bracket stack. Affects every cmark-gfm-derived renderer including GitHub's, Reddit's, GitLab's. Source: https://www.legitsecurity.com/blog/dos-via-software-supply-chain-innumerable-projects-exposed-to-a-markdown-library-vulnerability

5. **Long vertical link lists** — markdown-it 3.0.6 hang.

## 5. Negative Searches

- **No GHSA found** for `remark-*` or `rehype-*` core plugins (remark-gfm, remark-rehype, remark-directive, etc.). The unified pipeline parses through `micromark`, which avoids regex backtracking by design; this likely explains the absence.
- **No GHSA found** for `vfile-message` or `vfile-statistics` (they are utility/diagnostic packages, not parsers).
- **No GHSA found** for `markdown-it-sub`, `markdown-it-sup`, `markdown-it-footnote`, `markdown-it-container`. (Absence of evidence, not evidence of absence.)
- **No CVE found** for `gray-matter` matching the ReDoS/prototype-pollution combination implied by the prompt. `gulp-front-matter` has a documented prototype-pollution issue, but `gray-matter` itself shows no advisory matching this dimension.
- **No public advisory** for `commonmark.js` with a CVE — fixes shipped via PRs (#157, #281) without CVE assignment.
