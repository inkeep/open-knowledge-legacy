# Evidence: D7.3 — GFM-Specific Divergences Across Implementations

**Dimension:** Where GitHub Flavored Markdown implementations diverge — five extensions (tables, task lists, strikethrough, autolinks, disallowed raw HTML).

---

## 1. Overview

GFM adds five extensions over CommonMark. Each is a frequent source of cross-parser divergence because:
1. The GFM spec leaves multiple edge cases unspecified (e.g., list-vs-table precedence)
2. cmark-gfm itself contains documented bugs that GitHub.com's renderer does not exhibit (because GitHub runs additional pre/post-processing)
3. Downstream JS parsers (`remark-gfm`, `marked`, `markdown-it` plugins) have diverged in either direction trying to "match GitHub" vs "match the spec"

Three subtly different "GFM compatibility" targets exist: GFM written spec, cmark-gfm behavior, GitHub.com behavior.

---

## 2. Per-Feature Divergence Notes

### Tables (Section 4.10)

The most under-specified extension:

- **Strict column-count rule:** Header row and delimiter row must have the same cell count or the entire construct is treated as a paragraph (Spec example 203). Data rows are flexible — fewer cells get padded with empty `<td>`, more cells are silently dropped (Example 204).

- **List vs table precedence (UNDEFINED in spec):** Input like `a | b\n- | -\n1 | 2` is rendered by GitHub.com as a list, by cmark-gfm and markdown-it as a table. The GFM spec is silent; cmark-gfm issue #333 is open. Source: https://github.com/github/cmark-gfm/issues/333

- **Escaped backslash in cell:** `a \\| b` (escaped backslash followed by pipe) — by CommonMark rules `\\` should escape itself and the `|` should be a delimiter. cmark-gfm issue #277 documents that cmark-gfm incorrectly treats this as a literal pipe inside the first cell. GitHub.com follows cmark-gfm.

- **Pipe inside code span inside cell:** GFM tables require pipes inside code spans to be backslash-escaped, which directly contradicts CommonMark's rule that backslashes in code spans are literal (cmark-gfm issue #24, marked issue #1237/#1259).

- **Trailing whitespace becoming an extra column:** remark-gfm issue #11 — trailing space after a row's final pipe causes remark-gfm to emit one extra empty `<td>`; GitHub strips it.

- **Tables in blockquote with lazy continuation:** parsers differ on whether `> | a | b |\n| --- | --- |\n| 1 | 2 |` (no `>` on continuation lines) is a table inside a blockquote. flexmark/pegdown handle indentation differently than cmark-gfm.

### Task Lists

- **Marker case:** `[X]` vs `[x]` — both checked. `[ ]` (one ASCII space) — unchecked. cmark-gfm and remark-gfm agree.

- **Non-breaking space inside marker:** cmark-gfm issue #192 — `- [\u00A0]` is rendered as a literal text checkbox, not recognized as a task list item.

- **Task lists inside tables:** GitHub renders them; remark-gfm does not (remark-gfm issue #27).

- **Loose vs tight:** tight task lists produce `<li>` with inline content; loose produce `<li><p>...</p></li>`. Frequent React-Markdown bug surface.

### Strikethrough

The single largest binary divergence in GFM:

- **GFM spec (Section 6.5):** requires `~~text~~` (exactly two tildes).
- **cmark-gfm:** single-tilde `~text~` also produces strikethrough — documented as deliberate non-conformance to the spec (cmark-gfm issue #71). GitHub.com follows cmark-gfm.
- **markdown-it (default):** requires double tildes only (markdown-it issue #446) — matches the spec, diverges from GitHub.
- **marked:** historically allowed single-tilde (marked issue #1561), later restricted.
- **remark-gfm:** defaults to `singleTilde: true` (matches cmark-gfm/GitHub), configurable.
- **commonmark-java strikethrough extension:** requires double tildes, with config flag for single.
- **Pandoc:** GFM reader accepts only one tilde (pandoc issue #4840).

### Autolinks (extended) — Section 6.9

Three new patterns: bare URL (`http(s)://`, `ftp://`), `www.` prefix, and email. The trailing-punctuation algorithm is precise but parsers still diverge:

- **Specified trailing-trim set:** `?`, `!`, `.`, `,`, `:`, `*`, `_`, `~` are stripped from the end. Quotes (`'`, `"`) are NOT in this set per spec. marked PR #2673 fixed marked which was incorrectly also stripping quotes.

- **Parenthesis algorithm:** scan the URL — if closing parens > opening parens, trim trailing `)`. (marked PR #1293 fixed multi-backpedal cases.)

- **Underscore-in-last-two-segments rule:** a domain with `_` in the last two label segments (e.g., `https://foo_bar.example.com`) is rejected by the spec; some parsers (markdown-it autolink plugins, dart-lang/markdown) historically accepted them.

- **Semicolon resembling an entity:** trailing `;` is excluded only when it forms `&alphanumerics;` (looks like an HTML entity).

- **WWW autolinks:** `www.example.com` becomes `http://www.example.com` (note: `http`, not `https`) per spec — most parsers default to `http://`, but some configurations rewrite to `https://`.

- **Position constraint:** GFM requires autolinks to begin at line start, after whitespace, or after `*`, `_`, `~`, `(`. cmark-gfm enforces this strictly; some markdown-it plugins (e.g., `markdown-it-linkify-it`) link bare URLs in more positions.

### Disallowed Raw HTML (tagfilter) — Section 6.11

GFM lists exactly nine tags: `<title>`, `<textarea>`, `<style>`, `<xmp>`, `<iframe>`, `<noembed>`, `<noframes>`, `<script>`, `<plaintext>`. Spec: replace the leading `<` with `&lt;`.

- **Application:** cmark-gfm applies this only when the tagfilter extension is enabled. Many wrapper libs (cmarkgfm Python, mgenware/cmark-gfm-js) enable by default; others do not.

- **Case sensitivity:** per spec, matching is case-insensitive. Some implementations (older marked) only matched lowercase.

- **Closing tags:** spec applies to opening tags; closing `</script>` handling varies.

- **Other dangerous URLs:** separately, cmark-gfm scrubs `javascript:`, `vbscript:`, `data:`, `file:` URLs by default unless `CMARK_OPT_UNSAFE` is set — this is not part of GFM's tagfilter but is bundled with "GFM rendering" in practice. remark-gfm leaves URL scrubbing to the consumer (e.g., react-markdown).

- **pulldown-cmark:** opted not to implement disallowed raw HTML at all (issue #608).

---

## 3. Key Cross-Cutting Findings

1. **cmark-gfm IS the de facto spec for GitHub.com**, including its bugs. Any parser claiming "GFM compatibility" must choose: track the GFM written spec, track cmark-gfm behavior, or track GitHub.com behavior. These are three subtly different targets.

2. **Strikethrough is a forced choice**: `singleTilde` config exists in nearly every implementation precisely because spec and cmark-gfm disagree.

3. **Tables in non-trivial contexts (blockquote, list item, with code spans containing pipes)** are the highest-divergence area in practice.

4. **GitHub.com adds layers** beyond cmark-gfm: HTML sanitization (allowed-tags), URL scrubbing for `javascript:`, autolinking inside issue references, @mentions, emoji shortcodes — none of these are in the GFM spec or cmark-gfm.

5. **Remark/micromark family is generally the most spec-compliant**, but tracks GFM-spec rather than cmark-gfm bugs in some cases (e.g., escaped backslash handling).
