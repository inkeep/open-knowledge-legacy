# Evidence: D7.1 + D7.2 — Babelmark3 + CommonMark Forum Divergences

**Dimension:** Cross-parser divergence snippets mined from Babelmark3, the CommonMark spec, and `talk.commonmark.org` forum threads.

---

## 1. Spec Ambiguity Inventory

Areas where the CommonMark spec is silent, contradictory, or implementation-dependent — these are systematic sources of cross-parser divergence:

1. **Emphasis "openers_bottom" optimization** — Rule 9's mod-3 constraint should only apply when a delimiter is BOTH opener and closer; spec wording is loose enough that cmark, MD4C, commonmark.js, and downstream parsers all over-apply it. Source: https://talk.commonmark.org/t/i-dont-understand-how-emphasis-is-parsed/3866

2. **Asymmetric emphasis nesting** — Patterns like `*b**a***` vs `***a**b*` produce structurally different trees with no spec guidance on the asymmetry. Source: https://talk.commonmark.org/t/emphasis-strong-emphasis-corner-cases/2123

3. **Code span vs link title precedence** — Spec says code spans have priority over emphasis but is silent on link titles, which actually win. Source: https://talk.commonmark.org/t/precedence-of-link-title-over-code-span/8982

4. **Reference link followed by `[ ]`** — Spec example 570 talks about labels but doesn't address whitespace-only labels. Real divergence: commonmark.js / markdown-it / marked / remark say "no link"; pulldown-cmark and goldmark say "link". Source: https://talk.commonmark.org/t/reference-links-followed-by-space-only-pair-of-brackets/4581

5. **HTML block end-conditions inside other HTML blocks** — Type 6/7 blocks terminate at blank lines; if a `<pre>` is inside a `<table>`, blank lines inside `<pre>` wrongly close the outer block. No fix landed. Source: https://talk.commonmark.org/t/end-conditions-within-end-conditions/2388

6. **List tightness propagation across nesting** — Spec says looseness propagates to all list items but is unclear about inner-vs-outer behavior; parsers diverge on whether an inner loose list forces the outer to be loose. Source: https://talk.commonmark.org/t/tightness-and-looseness-of-nested-lists/4622

7. **Spec self-contradiction on multiple blank lines in lists** — Pre-0.27 spec said "two blank lines end all containing lists" while another sentence said "any number of blank lines between items". Resolved in 0.27. Source: https://talk.commonmark.org/t/multiple-blank-lines-inside-a-list/2289

8. **Inline precedence rules scattered** — commonmark-spec issue 438: precedence rules are sprinkled across sections 6.1–6.5 with no central index. Source: https://github.com/commonmark/commonmark-spec/issues/438

9. **Whitespace handling in code spans** — Pre-0.27 was inconsistent across implementations; spec now specifies "strip one space if both leading and trailing", but Markdown.pl, showdown, discount still differ. Source: https://talk.commonmark.org/t/leading-and-trailing-white-spaces-in-code-blocks/628

10. **Backslash escapes in autolinks vs link destinations** — Spec disallows escapes in autolinks but allows in link destinations; rationale not explained. Source: https://talk.commonmark.org/t/backslash-escapes-inside-link-destinations/2312

11. **Setext heading "could be a paragraph" exception** — A setext heading underline cannot be a thematic break or other block — boundary cases (e.g., setext after a list item) are spec-quiet. Source: https://talk.commonmark.org/t/setext-headers-and-empty-list-items/1173

12. **Autolink scheme list** — Spec restricts autolink schemes to a fixed list; jgm acknowledged the rule is somewhat arbitrary. Source: https://talk.commonmark.org/t/what-is-the-point-of-limiting-uri-schemes-in-autolinks/555

13. **Hard break rendering: `\\` (kramdown) vs `\` (CommonMark)** — Both forms exist in the wild; CommonMark canonicalizes to single backslash, but kramdown documents differ.

## 3. Notable Findings

- **Emphasis is the densest divergence cluster.** Patterns like `*b**a***`, `*a**b**c*`, and `*****Hello*world****` produce counterintuitive ASTs in cmark/commonmark.js/markdown-it/remark, and even the spec authors acknowledge the openers_bottom mod-3 rule is over-applied.

- **The biggest "modern" JS parser split** is `[foo][ ]` (whitespace-only label): JS family (commonmark.js, markdown-it, marked, remark) treats it as no-link; Rust (`pulldown-cmark`) and Go (`goldmark`) treat it as a link.

- **HTML blocks are a perennial divergence source** because of the type 1–7 system and the "blank line ends type 6/7" rule that wrongly applies to nested `<pre>` inside `<table>`.

- **GFM previews three big extension-driven divergences** — strikethrough (single vs double tilde), tables (pipe in code, missing pipes), task lists (none in core CommonMark) — all of which depend on whether the JS parser has loaded a GFM-flavored plugin. Defaults vary: marked/cmark-gfm enable by default, markdown-it/remark require opt-in.

- **Babelmark3 itself documents ~25 implementations** including pandoc, commonmark.js, marked, markdown-it, kramdown, Markdig, cebe/markdown variants, blackfriday, and the original Markdown.pl. The tool flags spec-compliant outputs with a badge. Babelmark URLs are dynamic (input-encoded).

- **List tightness was the most spec-self-contradictory area** until the 0.27 cleanup. Modern JS parsers all converge, but older marked/remark forks may still differ.
